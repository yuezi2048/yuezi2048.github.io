十二、异常

12.1 为什么要异常处理（阻断）

  • 异常处理是为了不因为Exception而阻断程序的运行，保证正确的业务逻辑。（阻断运行实际上是JVM机的默认的异常处理机制，后面会说到）

  • 提供一个实际案例，我们被除数是0，虽然程序编译通过，但实际运行时报了ArithmeticException，程序结束，我们现在不想让他结束并且能捕获到出错的位置

  • 提供一种解决方案：使用try-catch对异常捕获

    • 技巧：选中被try包裹的代码块，ctrl + alt + t 按6就是try-catch
  • 提出一个问题，请思考：这里为什么用Exception也可以捕获，而不是ArithmeticException
    • 接下来我将解释异常的继承关系，结合动态绑定机制就说得通了。

12.2 异常快速入门

在java语言中，程序发生不正常的情况就是异常，包括两大类

  • Error: JVM机无法解决的严重问题，如栈溢出StackOverFlowError , OOM(out of memory), 是严重错误，会导致程序崩溃
  • Exception：一般性问题，可以针对性地解决，这里面又包括两大类
    • 运行时异常 RuntimeException 比如空指针NullPointerException
    • 编译时异常 比如文件读取时读取不存在的文件，网络连接中断等，后续的章节会提到这些

*12.3 异常体系图

  • 使用ctrl + h查看类图、ctrl + alt + b查看实现的方法
  • 虚线是实现，实线是继承

可以发现：异常包括运行时异常和编译时异常

  • 运行时异常编译器是不会报错的，java.lang.RuntimeException及其他的子类属于运行时异常
    • 运行时异常可以暂不处理，完全处理可能影响程序可读性和运行效率
  • 编译时异常是必须处理的

12.3.1 运行时异常(RuntimeException)

预告：我们可以用异常来保证 输入的正确性，后面会提到。

  • NullPointerException 空指针异常

  • ArithmeticException数学运算异常

  • ArrayIndexOutOfBoundsException 数组下标越界异常

  • ClassCastException 类型转换异常

  • NumberFormatException 数字格式不正确异常

看几道练习

12.3.2 编译异常

前面提到，这是我们在编译期间必须处理的异常，否则代码无法通过编译

  • SQLException //操作数据库时，查询表可能发生异常
  • IOException //操作文件时，发生的异常
  • FileNotFoundException //当操作一个不存在的文件时，发生异常
  • ClassNotFoundException //加载类，而该类不存在时，异常
  • EOFException // 操作文件，到文件末尾，发生异常
  • IllegalArguementException //参数异常

此时我们发现d:\\aa.jpg可能并不存在，所以必须加一层异常处理机制才不会报编译异常。

12.4 异常处理机制

12.4.1 try-catch-finally

  • catch负责捕获异常

    • 异常发生后， 进入catch块

    • 如果没有发生，当然也就不会进入catch块

  • finally是最终必须执行的操作

    • 不管异常有没有发生，finally都必须执行
    • 通常用于释放资源，比如文件、数据库

  • try代码块如果有多个异常

    • catch可以写多条语句分别处理，要求：子类写前面，父类写后面

  • try-finally结构

    • 表示不管有没有发生异常，必须要执行一段业务逻辑
    • 由于没有catch捕获异常，程序会照常异常终止

细节——catch+finally return的机制

思考：如果一个方法体里面，catch返回了一个值，finally怎么执行呢？如果finally里面也返回值那返回谁的呢？(答：finally)

应用——输入符合要求的类型

我们利用到的本质是就是try出错语句后面语句不执行的规则，那么我们可以借此来实现输入符合要求类型的需求

  • 执行死循环，在死循环内，套一个try-catch，里面包含: 出错语句+break
    • 如果出错了，进入catch-finally，不执行break，继续循环
    • 否则break，退出循环，此时输入的数符合要求

12.4.2 throws

  • throws实际上是默认的异常处理机制（当程序员没有编写异常处理时）
  • throws和try-catch-finally不需要都写
  • throws不断向调用它的类”甩锅”，到JVM机位置，此时显示异常，程序崩溃退出

细节——throws抛出异常

  • 我们抛出异常给调用者的时候，如果这个异常是编译类型，那么调用者必须给他解决，即显式地对编译异常进行异常处理机制。
    • 而运行时异常则隐式地抛出，如果没有写，则默认网上抛出该异常
  • 子类重写父类的方法时，如果父类的方法抛出了一个异常，那么子类重写时抛出的异常类型必须和父类一致或者是其子类

12.5 自定义异常(throw生成异常对象)

步骤

  • 定义类 继承 Exception 或者 RuntimeException(推荐)

    • 继承Exception 表示是 编译异常

    • 继承RuntimeException 表示是运行时异常

  • 重写构造方法，比如Exception(message)

  • 使用时 使用throw关键字 生成异常对象

一般之所以继承运行时异常，还是归因于运行时异常隐式抛出异常的特性

12.6 throws 和 throws

  • throws是处理异常的方式，在方法声明处，后面跟的是异常类
  • thorw是创建一个异常对象，在方法体内，后面跟的是异常对象
