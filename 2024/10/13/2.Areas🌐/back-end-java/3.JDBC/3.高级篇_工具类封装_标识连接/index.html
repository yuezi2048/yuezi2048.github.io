高级篇

一、JDBC优化和工具类封装

1.1 存在的问题

我们发现，虽然连接池解决了用户频繁请求释放连接造成的资源浪费，但有这些问题：

  • 用户连接一次就要写 创建连接池、建立连接、连接回收的代码，存在代码冗余的问题 —> 工具类封装
  • 用户频繁连接，会对应连接池不同的新连接，在使用事务时，用户无法保证事务的正常提交 -> 怎么让用户线程在一段时间内标识唯一的连接？ –> map

1.2 解决1：工具类的封装

使用druid的配置文件

编写工具类，要点如下

  • 维护一个连接池对象
    • 定义dataSource属性
    • 通过类加载器从resource中读取properties，并将输入流载入properties
    • 用DruidDataSourceFactory，传入properties对象创建dataSource
  • 对外提供在连接池获取连接的方法
    • dataSource.getConnection();
  • 对外提供回收连接的方法
    • connection.close();

注：此时仅解决了工具类的封装，但无法保证单个请求连接的线程，多次操作数据库时，连接是同一个，无法保证事务！引出ThreadLocal

1.3 ThreadLocal

  • JDK 1.2的版本中提供java.lang.ThreadLocal，解决多线程的并发问题，多用于多线程管理共享数据库连接、Session等
  • ThreadLocal保存线程的共享变量，每个线程都对应一个ThreadLocalMap<ThreadLocal， Object>，Object就是该线程的共享变量
    • map通过ThreadLocal的set、get、remove方法操作，对于同一个static ThreadLocal，每个线程之间的操作都是相互隔离的（符合事务的隔离性）
  • 作用
    • 对象跨层传递（如Service->Dao）时，ThreadLocal因为可以从线程中得到共享变量，避免了数据的多次传递
    • 线程间的数据隔离
    • 进行事务操作，用于存储线程事务信息
    • 数据库连接(共享Connection变量)，Session会话管理
  • 操作
    • ThreadLocal对象.get: 获取ThreadLocal中当前线程共享变量的值。
    • ThreadLocal对象.set: 设置ThreadLocal中当前线程共享变量的值。
    • ThreadLocal对象.remove: 移除ThreadLocal中当前线程共享变量的值。

1.4 解决2：线程操作同一个连接

在V1.0的版本基础上，我们将连接对象放在每个线程的ThreadLocal中，保证从头到尾当前线程操作的是同一连接对象。

  • 维护一个连接池对象
    • 在dataSource属性基础上，加入ThreadLocal<Connection> threadLocal
    • 通过类加载器从resource中读取properties，并将输入流载入properties
    • 用DruidDataSourceFactory，传入properties对象创建dataSource
  • 对外提供在连接池获取连接的方法，首先需要判断threadLocal.get()
    • 如果为空，则threadLocal.set(dataSource.getConnection());
    • 否则，通过threadLocal.get()获取该线程的共享连接
  • 对外提供回收连接的方法，同样是首先判断threadLocal.get()
    • 如果为空，就不做
    • 否则在原来归还给连接池的基础上
      • （1）使用remove方法将connection对象从threadLocal中移除
      • （2）connection.setAutoCommit(true); 保证最终事务是自动提交的

测试代码，可以发现改进后创建的连接都是一样的地址

二、DAO的封装和BaseDao工具类

2.1 为什么有DAO层

  • 问题：接下来我们要使用这些工具类来帮助我们crud，但即使是在正式的代码中使用这些工具类也还是显得冗余 –> 我要是能像DBMS那样传入SQL语句和参数得到结果就爽了
  • 解决：该工具类仅包含维护连接、建立连接和释放连接的功能，中间还有一段SQL执行的代码，我们专门执行SQL的功能封装到一个工具里就可以 –> 封装对表的操作

引出Dao(Data access object)层，数据访问对象。

  • 在ORM的基础上，我们明确mysql的数据在java是以实体类的方式存在的，一个表对应一个实体类，而我们对表的操作就是一个DAO对象！
  • 通过Java操作数据库时，同一张表的crud通过DAO层的类来统一维护
  • DAO只关心对数据的操作，供Service使用，职责明确

2.2 为什么有BaseDao父类

  • 调用者觉得使用SQL语句复用性还是太高，就想通过一个方法名比如selectById，传入参数就能得到结果 –> 公共的crud抽取出来作为公共代码

–> 引出BaseDao

  • 一个数据表对应一个DAO接口和实现类，而对于所有表的crud很多代码重复度高，因此将公共代码写入BaseDao作为公共父类，供各个实现类复用

2.3 BaseDao搭建

DML

DML操作

  • 参数使用可变长数组
  • 返回值int，受影响的行数
  • 通过for循环对参数赋值
    • 注意1：考虑到通用性，使用Object类型赋值
    • 注意2：preparedStatement预编译语句占位符下标从1开始
  • 如果自动提交事务了，则释放连接，没有自动提交事务，需要交由调用者来控制commit / rollback

*DQL

多行多列的通用查询语句

执行与DML差异不大，难点是：对结果集的通用性处理，涉及到泛型和反射的综合应用

  • 返回的类型由调用者决定，编写者不知道，故使用方法泛型<T>，可以理解为T类型交由调用者决定
  • 返回的结果保存到List<T>集合中，这样也包含了单行单列、单行多列的情况（返回第一个元素list.get(0)即可）
  • 处理结果集的思路
    • 需要用结果集的getMetaData方法获取到元数据对象，进而获取到列的数量metaData.getColumnCount()信息
    • 使用next循环结果集
      • 为了获取到该记录对应的对象，通过调用者传入的Class对象使用反射技术，从而创建一个T类型的对象(clazz.newInstance)
      • 通过先前获取到的列数，for循环遍历对该对象赋值
        • 首先需要获取列名称，使用MetaData.getColumnLabel(i + 1)获取到第i+1个列名（要求调用者使用SQL语句要起别名对应上属性名）
        • 接下来通过Class对象，得到Field字段，clazz.getDeclaredField(fieldName);（私有字段注意要爆破setAccessible）
        • 得到Field字段对象，和字段名，即可赋值，field.set(t, value);
  • 最后别忘了结果集要额外的手动关闭（非公共操作）

三、事务

3.1 事务回顾

  • 给出经典场景：

    • 一个人少钱，一个人多钱，要么成功，要么失败，不允许仅一个人多钱或者仅一个人少钱的情况
    • 批量删除/添加，要求要么都删除/添加，要么都执行失败，一个都不能删除/添加

  • 使用事务实现，事务是SQL语句执行的缓存机制

    • 一个事务内所有语句成功，可以触发commit来永久更新数据
    • 事务内任意一个语句失败，事务失败，触发ROLLBACK回滚结束事务，恢复到事务开始时的状态

  • 事务特性

    • 原子性（Atomicity）：非无即有的方面。理解为Bean，是一个不可分隔的单位，
    • 一致性（Consistency）：状态变换一致的方面。事务必须满足从一个一致性状态 –> 另外一个一致性状态
    • 隔离性（Isolation）：多个事务的方面。事务内部操作对并发的其他事务是隔离的，各个事务之间互不干扰
    • 持久性（Durability）：事务提交结果的方面。事务一旦提交，对数据库的改变是永久性的，接下来的操作/故障不会对该事务的结果有任何影响

  • 事务的提交方式：

    • 自动提交：默认方式。每一条语句成功就提交，失败就会回滚
    • 手动提交: 需手动开启SET autocommit = FALSE。每一段语句自行根据执行结果 手动提交，或手动回滚
