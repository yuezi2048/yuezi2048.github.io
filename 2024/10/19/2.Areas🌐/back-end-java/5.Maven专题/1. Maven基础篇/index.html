一、Maven基础篇

1.1 Maven简介

1.1.1 为什么要用Maven

传统构建java项目的弊端：不可避免用到一些jar包

  • 需要提前准备jar包，繁琐性问题
  • jar包之间jar包之间有依赖关系（版本冲突），版本兼容问题
  • 导入很多jar包后，项目会变得庞大（尤其是多个项目复用同一个jar包的情况），不利于后期的部署和维护

解决方案：Maven工具，传统Web项目构建的crm项目工程占用空间大小如下

1.1.2 什么是Maven

Maven一句话来说就是一个项目管理工具，他所解决的问题：

  • 无需手动收集jar包，版本管理，解决版本冲突
  • 自动执行单元测试，检验代码质量
  • 自动项目打包、发布，便于后续发布到服务器
  • 统一开发结构

可以从下图发现，Maven定义了项目对象模型POM（Project Object Model），包括

  • 项目生命周期(Project Lifecycle)
    • maven从编译测试清理打包安装部署一系列的生命周期
  • 依赖管理系统(Dependency Management System)
    • pom.xml可以定义坐标，通过依赖可以快速定位到项目所需要的依赖，而这些依赖存储在仓库中（本地仓库、私服【可选】、远程中央仓库）
  • 运行定义在生命周期阶段(phase)中插件(plugin)目标 (goal)的逻辑
    • 通过插件的安装可以帮助我们在这些声明周期中实现一些功能

1.2 maven下载与安装

1.2.1 下载与安装

官网：http://maven.apache.org/
下载地址：http://maven.apache.org/download.cgi

兼容性说明

  • IDEA 2022 兼容maven 3.8.1及之前的所用版本
  • IDEA 2021 兼容maven 3.8.1及之前的所用版本
  • IDEA 2020 兼容Maven 3.6.3及之前所有版本
  • IDEA 2018 兼容Maven3.6.1及之前所有版本

maven的安装（版本号要与idea版本适配）

  • maven解压到非中文路径
  • maven安装的前提是存在jdk1.8的环境
  • 环境变量 配置MAVEN_HOME，配置path里面的路径 %MAVEN_HOME%\bin
  • 使用mvn -v检查是否安装成功

简单说明maven软件包内部路径

  • bin:存放了 maven 的命令，比如我们前面用到的 mvn tomcat:run
  • boot:存放了一些 maven 本身的引导程序，如类加载器等
  • conf:存放了 maven 的一些配置文件，如 setting.xml 文件
  • lib:存放了 maven 本身运行所需的一些 jar 包

1.2.2 maven相关概念

1.2.2.1 仓库 repository

仓库存储maven需要引用下载的包，三种形式：本地、私服、中央仓库

  • 本地仓库：maven优先从本地仓库查找 默认本地仓库位置在${user.dir}/.m2/repository，${user.dir}表示 windows 用户目录

    • 建议修改本地仓库位置到D盘，在 MAVE_HOME/conf/settings.xml 文件中配置本地仓库位置

  • 远程仓库：本地仓库没有，默认去远程仓库找，可以在局域网，也可以是互联网

    • 中央仓库地址：http://repo1.maven.org/maven2

1.2.2.2 坐标 dependency

坐标：通过<dependency>标签描述，前三个是必须写的，帮助我们快速定位仓库的资源https://mvnrepository.com/

组成：

  • groupId 定义当前maven项目隶属组织名称(通常是域名反写，例如org.mybatis)
  • artifactId 定义当前maven项目名称(通常是模块名称)
  • version 定义当前项目版本号
  • packaging 定义该项目的打包方式

1.2.2.3 镜像仓库 mirror

镜像仓库：解决远程仓库资源下载慢的问题

在settings.xml配置阿里云地址即可

1.3 构建第一个Maven项目

1.3.1 目录结构和常用命令

目录结构

  • src/main/java —— 存放项目的.java 文件
  • src/main/resources —— 存放项目资源文件，如 spring, hibernate 配置文件
  • src/test/java —— 存放所有单元测试.java 文件，如 JUnit 测试类
  • src/test/resources —— 测试资源文件 target —— 项目输出位置，编译后的 class文件会输出到此目录
  • pom.xml——maven 项目核心配置文件 注意：如果是普通的 java 项目，那么就没有 webapp 目录

maven常用命令

  • mvn compile：编译class字节码到target目录下
  • mvn clean：删除target下字节码文件
  • mvn test：执行测试结果代码，生成结果报告
  • mvn package：编译 + 测试，成功后打包为jar包 / war包
  • mvn install：编译 + 测试 + 打包，成功后，发布到本地仓库

1.4 依赖管理

1.4.1 依赖类别与依赖的传递性

依赖：当前项目所需要的jar包

依赖的传递性：项目内包含了很多依赖（资源）。说明我们的项目依赖于这些资源

  • 而定义在pom.xml里面的依赖又分为两种

    • 第一种：直接依赖 直接在pom.xml里面定义的资源，例如junit

    • 第二种：间接依赖 直接依赖包含的子依赖，比如org.hamcrest:hamcrest-core.1.3

  • 依赖传递性的体现：对于项目来说，项目不仅依赖Junit，还依赖org.hamcrest:hamcrest-core.1.3

1.4.2 依赖冲突问题的解决(自动与手动)

测试依赖

  • maven自动解决冲突

    • 直接依赖覆盖原则：对于直接定义在项目中的直接依赖，是以后面一个为准
    • 间接依赖第一声明优先原则： 同一个间接依赖，以前一个为准
    • 路径近优先原则：对于直接依赖和间接依赖的冲突，以直接依赖为主

  • 手动管理依赖

    • 排除依赖原则，冲突的版本可以手动排除，但要确定不影响项目正常运行，注意排除的依赖 无需加版本号
    • 版本锁定：通过dependencyManagement 固定版本号（常用）

问：dependencyManagement和dependencies标签的区别？

  • dependencyManagement标签：只是声明项目对jar包的依赖关系并对所依赖jar包进行版本管理，并不实现引入。而dependencies标签则是将所依赖的jar直接加到项目中。

  • dependencies标签：因为dependencyManagement里只是声明依赖，并不实现引入，因此子项目需要显示的声明需要用的依赖。如果不在子项目中声明依赖，是不会从父项目中继承下来的；只有在子项目中写了该依赖项，并且没有指定具体版本，才会从父项目中继承该项，并且version和scope都读取自父pom；另外如果子项目中指定了版本号，那么会使用子项目中指定的jar版本。

1.4.3 依赖范围 scope

  • 依赖的jar包在默认情况下可以在任何范围内使用，我们可以通过scope标签来控制其作用范围 （默认作用范围compile）
  • 作用说明
    • 主程序范围有效(main文件夹)
    • 测试程序范围有效(test文件夹)
    • 是否参与打包(package指令范围之内)

测试：定义Junit依赖的scope范围后，主代码main下使用@Test失败

1.5 生命周期与插件

1.5.1 生命周期核心

生命周期的核心

  • clean： 清理工作阶段
    • pre-clean 执行一些需要在clean之前完成的工作
    • clean 移除所有上一次构建生成的文件
    • post-clean 执行一些需要在clean之后立刻完成的工作
  • default：核心工作阶段。比如编译compile、测试test、打包package、安装install 部署deploy等
  • site： 产生报告，发布站点等

1.5.2 插件Plugin

  • 插件和生命周期各个阶段绑定在一起，当执行到某个生命周期时，就可以通过插件执行相关操作
    • maven在各个生命周期阶段上绑定了一些默认的预设功能
  • 我们可以自定义编写插件
