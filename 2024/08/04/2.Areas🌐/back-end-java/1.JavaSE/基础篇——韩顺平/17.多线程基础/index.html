十七、多线程基础

17.1 线程相关概念

程序(program)：完成特定任务，使用特定语言编写的指令集合。

进程：表示运行的程序。例如启动了QQ，就表示启动了一个进程，操作系统为其开辟一块内存空间。

  • 进程是程序的执行过程，是一个动态的过程，有其产生、存在和消亡的生命周期

线程：线程是通过进程创建的（也可以通过线程创建），是进程的一个实体

  • 一个进程可以拥有多个线程，引申出两种线程类型
    • 单线程：同一个时刻只能运行一个线程
    • 多线程：同一个时刻可以执行多个线程（比如迅雷的多任务下载）

理解：

  • 一个线程就可以联想到一个猴毛，让一个小猴干一件事情而不影响本体的运行

并发和并行的概念：

  • 并发：同一个时刻，多个任务交替执行
    • 举例：（1）单核CPU实现多任务；（2）同时开车和打电话

  • 并行：同一个时刻，多个任务同时执行，多核CPU可以实现并行

  • 并发和并行可以同时进行

17.2 基本使用

创建线程两种方式，本质是一样的：

  • 继承Thread类
  • 实现Runnable接口

17.2.1 继承Thread类

  • 继承Thread类后，重写run方法
  • 调用者创建对象以后，调用start方法启动线程
  • 解答：为什么不是调用run方法而是用start方法？
    • （1）run方法仅仅是main线程方法的调用，与多线程无关，调用方法会阻塞main线程；
    • （2）start方法底层调用了native的start0()方法，该方法由Jvm底层调用（内核态） 这个方法实实在在的为我们创建了一个线程，这个操作不会阻塞main线程

  • 技巧：使用JConsole来监视线程的执行情况

17.2.2 实现Runnable接口（静态代理）

给定一个场景：需要对Tiger类实现多线程，而Tiger类继承了Animal类，没有条件继承Thread类，无法使用第一种方法。

解决方案：借助静态代理Runnable接口实现多线程（本质上就是通过接口多态来动态绑定到我们多线程的run方法）

  • 在Thread类中，实现Runnable接口，并提供Runnable类型的属性target
    • 创建thread对象后，在构造器中对该属性赋值
    • thread对象调用start方法后，本质上也是调用了start0()方法创建了一个线程，但是在调用run方法时，会动态绑定到对应实现Runnable的类的run方法

17.2.3 继承Thread和实现Runnable比较

  • 没有本质区别，Thread类本身实现了Runnable接口
    • 无非一个是继承Thread类后重写run方法的多态；一个是实现Runnable接口的方法，然后接口实际运行的时候多态绑定到实现的run方法的原理
  • Runnable接口，比较适合多个线程共享一个资源的情况（可以通过一个对象 创建多个线程），并且规避了单继承的问题
    • 提醒：继承Thread类则需要创建不同的对象，因此每个对象的hashCode是不一样的
    • 因此使用synchronized修饰方法后（即声明this为对象互斥锁），会出现每个线程争用的锁不一样导致同步失败的问题，后面会提到

17.3 线程终止（通知）

线程终止有两种情况

  • run方法结束，线程执行完了任务
  • 通过变量控制run方法，从而提前结束r线程,，采用了通知的方式

17.4 线程常用方法

17.4.1 线程设置

  • 设置线程名称

    • setName ：设置线程名称，使之与参数 name 相同
    • getName ：返回该线程的名称

  • 设置线程的优先级（提供了Thread.MIN_PRIORITY 等常量）

    • setPriority ：更改线程的优先级
    • getPriority ：获取线程的优先级

  • 控制线程生命周期（后面会提供线程生命周期的讲解，思考调用这些方法后进入了什么生命周期）

    • start ：使该线程开始执行;Java 虚拟机底层调用该线程的 start0 方法

    • run ：调用线程对象 run 方法:

    • sleep：在指定的毫秒数内让当前正在执行的线程休眠（暂停执行）

    • interrupt：中断线程（注意，并没有结束线程，一般都是用于中断休眠的线程）

17.4.2 线程控制

  • yeid：线程礼让，让其他线程先执行，但基于内核态，不一定成功
  • join：线程插队，一旦插队成功，会让插队的线程先执行完

17.4.3 用户线程和守护线程

  • 用户线程：也叫工作线程，随着线程执行完或者以通知的方式结束
  • 守护线程：为工作线程服务，所有的用户线程结束，守护线程结束
    • 用setDaemon(true)设置为守护线程（一般用于监控其他线程）
    • 常见的守护线程：JVM垃圾回收机制

17.5 线程生命周期

jdk1.8中枚举了6类状态（实际上Runnable状态可以细分为ready和running，所以也可以说有生命状态有7类）

  • NEW：未启动，调用start()方法后进入RUNNING
  • RUNNING
    • Running：被调度器选中后执行，结束运行后进入TERMINATED
    • Ready：Running状态下调用（1）suspend; （2）Thread.yeild() 变为Ready状态
  • TimeWaiting：休眠（提供特定时间）
    • 睡觉的时候进入该状态，睡觉结束后继续RUNNIG
  • waiting：等待另一个线程完成
    • 调用join，被其他线程插队的时候进入等待，插队的线程结束后进入RUNNING
  • Blocked：阻塞
    • 等待进入同步代码块，拿到锁才进入RUNNING
  • TERNNUBATED：线程退出

可以通过getState观察当前的生命周期

17.6 线程同步

17.6.1 为什么要线程同步

多个线程同时访问一个数据，会造成读取和修改异常，需要通过同步技术来保证数据的完整性

  • 给定一个售票系统，可以发现读写存在异常
    • 读到的是相同的票数
    • 票数为0后，但是还会执行，显示票数为-1
  • 原因就是ticket变量同时被多个线程访问，造成了线程安全的问题

17.6.2 Synchronized实现同步

  • 当希望一个对象不能被多个线程同时访问时，通过Synchronized关键字来保证该数据在同一时刻只能有一个线程访问
    • 理解：当一个线程对一块内存地址操作时，其他线程都不能对该内存地址操作，必须等待该线程完成操作（雪人持有门对象锁上厕所的例子）

  • 实现方法
    • 同步代码块：Synchronized(对象) {}
    • Synchronized 放在方法的声明中： public Synchronized void m() {}

实现原理（注意 互斥锁是非公平锁，有可能一直都是一个线程占用）

  • 多个线程通过争用一个互斥锁
    • 互斥锁是为了保证共享数据操作的完整性，但互斥锁引入会导致运行效率的降低
    • 使用Synchronized来与对象的互斥锁联系，表示该对象在任一时刻只能有一个线程访问
    • 非静态同步方法的锁可以是this，也可以是其他对象（要求每个线程的对象锁是同一个对象）
    • 静态同步方法的锁是类名.class本身（细节：静态方法属于类加载范畴，不能用this这些关键字）
  • 争到该对象锁的线程对tick属性进行操作，并输出打印，其他线程被阻塞
  • 该线程结束后，返还对象锁，返回第一步

引入互斥锁后，解决售票问题

  • 分析需要上锁的代码：对票数操作的方法sell()
  • 选择同步代码块/同步方法
  • 要注意：每个线程的锁对象要同一个

17.6.3 细节1——锁对象

  • 非静态方法——默认锁对象为this
  • 静态方法——默认锁对象为当前类.class
  • 再次注意1：一定要多个线程争夺同一个锁对象才能达到同步的目的
  • 再次注意2：锁对象是让线程有资格进入相应的代码块/方法，里面的内容才是具体执行的可能产生冲突的业务

17.6.4 细节2——释放锁的条件

同步代码块执行结束，线程会释放锁，包含以下情况：

  • 同步方法/代码块 正常执行结束（上完了）
  • 同步方法/代码块 遇到 break/return（上的时候被拉出去改bug了）
  • 同步方法/代码块 出现Exception/Error（上的时候发现没带纸）
  • 同步方法/代码块 执行了线程对象的wait()方法，进入WAIT状态，当前线程暂停，并释放锁（拉不出来，出门酝酿，酝酿好了再来）

以下情况不会释放锁

  • 同步方法/代码块 调用Sleep方法，进入TIMED_WAITING状态，调用yield方法，进入ready状态，当前线程暂停，但是不会释放锁（上的时候睡眠）
  • 同步方法/代码块 调用了suspend()方法 和 resume()控制线程 进入ready/running状态（不推荐使用），当前线程暂停，但是不会释放锁

17.7 死锁

死锁：双方互相占用了对方的锁资源，导致两个人都出不来而且也不能释放锁，造成了死锁的局面
