4.3 spring事务管理API

4.3.1 事务管理API关键概念

上一章节提到，spring实际上已经封装好了事务管理的一系列操作

现在我们正式讨论spring对上述的事务进行封装的API，进而帮助我们简化对于事务管理的开发。

首先明确一些关键概念（通过ctrl+n快捷键可以搜索类的相关定义）

  • 1.PlatformTransactionManager 平台事务管理器

    • 接口。是Spring管理事务的真正对象，其底层又包括：
      • DataSourceTransactionManager 底层通过JDBC管理事务
      • HibernateTransactionManager 底层通过Hibernate管理事务

  • 2.TransactionDefinition：事务定义信息：隔离级别（面试）、超时信息、传播行为、是否可读等。

    • getName() ：获取事务对象名称

    • getIsolationLevel() ：获取事务隔离级别（spring默认隔离级别ISOLATION_DEFAULT与数据库隔离级别一致）

      • 事务隔离级别反应出事务提交并发访问的处理态度，具体类别如下：
      • ISOLATION_READ_UNCOMMITED 可读未提交数据
      • ISOLATION_READ_COMMITED 只能读已提交数据，解决脏读问题（oracle默认）
      • ISOLATION_REPEATALE_READ 可读取其他事务修改后的数据，解决不可重复读问题（MySQL默认）
      • ISOLATION_SERIALIZABLE（读取其他事务提交添加后的数据，解决幻读问题）

    • getPropagationBehavior()：获取事务传播行为（即什么时候要开启事务如增删改，什么时候可以不开启事务如查询）

      • REQUIRED：没有事务时会新建一个事务，如果存在事务，就加入到该事务中（默认）
      • SUPPORTS：支持当前事务，如果没有事务，以非事务方式进行
      • MANDATORY：使用当前事务，如果没有事务，抛出异常
      • REQUERS_NEW：新建事务，如果当前在事务中，则把该事务挂起
      • NOT_SUPPORTED：以非事务方式执行操作，如果存在事务，则把该事务挂起
      • NEVER：以非事务方式运行，如果存在事务，抛出异常
      • NESTED：如果存在事务，则在嵌套事务内执行，如果没有事务，则执行类似REQUIRED操作

    • getTimeout() ：获取事务超时时间（当提交/回滚达到多长时间过期，默认为-1，如果有，以秒为单位）

    • isReadOnly()：获取事务是否只读（建议查询时设置为只读）

      • 读写型事务：增删改时，会开启事务
      • 只读型事务：执行查询时也会开启事务

  • 首先要开启对aop事务注解相关的支持
  • 使用@Transactional来定义事务
    • 具体包含隔离级别和传播行为
    • 可以修饰类也可以修饰方法

4.4 Spring5新特性

Spring4到Spring 5是近四年内的一次重大升级，主要发展之一就是SpringBoot项目的演变。

4.4.1 新功能概述

Spring5的新特性主要是两个部分

  • 支持 响应式编程（Reactive Programming）

    • 微服务通常是事件通信的架构构建的，应用常常是对一个事件（消息）做出反应

      • 响应式编程则提供了一种可选的编程风格，专注于构建对事件响应的应用程序

    • Java8没有内置的响应式编程支持，可通过框架来对响应式编程支持，包括：

      • Reactive Streams：尝试定义与语言无关的响应式API
      • Reactor：基于java实现的响应式编程
      • Spring WebFlux：基于响应式编程的Web应用程序开发，并提供类似于Spring MVC的编程模型

  • 函数式Web框架

  • 基于java8支持java9，兼容了很多Java8新特性

    • 核心Spring接口中static方法
    • 基于反射增强的内部代码改进
    • 框架代码中使用函数式编程——lambda表达式、Stream流

4.4.4 Spring Web Flux（了解）

  • 传统web框架比如SpringMVC，这些基于Servlet容器以及Webflux是一种异步非阻塞的框架
    • 异步非阻塞的框架在Servlet3.1以后才支持，核心是基于Reactor的相关API实现的。

什么是异步非阻塞

  • 异步和同步、非阻塞和阻塞 两者都是针对对象不一样。
    • 异步和同步针对调用者，调用者发送请求，如果等着对方回应之后才去做其他事情就是同步，如果发送请求之后不等着对方回应就去做其他事情就是异步。
    • 阻塞和非阻塞针对被调用者，被调用者受到请求之后，做完请求任务之后才给出反馈就是阻塞，收到请求之后马上给出反馈然后再去做事情就是非阻塞

WebFlux的特点：

  • 非阻塞：在有限资源下，提高系统吞吐量和伸缩性，以Reactor为基础实现响应式编程。
  • 函数式编程：Spring5框架基于java8，Webflux使用java7函数式编程方式实现路由请求

与Spirng MVC比较：

  • 两个框架都可以使用注解方式，都运行在Tomcat等容器中。
  • SpringMVC采用命令式编程，Webflux采用异步响应式编程。

4.4.4.1 基于java实现响应式编程(观察者模式)

响应式编程是什么？

  • 响应式编程是一种面向数据流和变化传播的编程范式。这意味着可以在编程语言中很方便地表达静态或动态的数据流。
  • 而相关的计算模型会自动将变化的值通过数据流进行传播
    • 例如，在命令式编程环境中，a=b+c表示将表达式的结果赋给a，而之后改变b或c的值不会影响a。但在响应式编程中，a的值会随着b或c的更新而更新

4.4.4.2 基于Reactor实现响应式编程

响应式编程操作中，Reactor是满足Reactive规范框架。

Reactor有两个核心类，Mono和Flux，这两个类实现接口Publishe，提供丰富操作符。

  • Flux对象实现发布者，返回N个元素；
  • Mono实现发布者，返回0或者1个元素。

Flux和Mono都是数据流的发布者，使用Flux和Mono都可以发出三种数据信号：元素值、错误信号、完成信号

  • 错误信号和完成信号都代表终止信号
  • 终止信号用于告诉订阅者数据流结束了
  • 错误信号终止数据流同时把错误信息传递给订阅者

三种信号的特点：

  • 错误信号和完成信号都是终止信号，不能共存的。
    • 如果没有发送任何元素值，而是直接发送错误或者完成信号，表示是空数据流。
    • 如果没有错误信号，没有完成信号，表示是无限数据流。
  • 调用just或者其他方法只是声明数据流，数据流并没有发出，只有进行订阅(subscribe)之后才会触发数据流，不订阅什么都不会发生的

实际上，数据流发送之后，一般是要经过加工后，再订阅加工后的数据流。有如下几种方式

  • map元素映射：{1,2,3} -> {4,6,9}
  • flatMap元素映射为流：每个元素转化成流，然后多个流合并为一个大的流，顺序不确定，{hello, abc, xq} -> {xqanchello}

4.4.4.3 WebFlux执行流程与核心APi

SpringWebflux基于Reactor，默认使用容器是Netty，Netty是高性能的NIO框架，即异步非阻塞的框架

BIO（阻塞）

  • 在做Java网络编程时，服务端的accept方法和read是阻塞的，直到有新连接进来或有数据可以读取
  • 这就意味着在阻塞期间线程不能干别的事情，这就是传统的BIO（blocking-io，同步阻塞IO）模型
  • 衍生出的问题：BIO模型在高并发场景下会面临服务端线程数过多浪费系统资源和频繁上下文切换问题

NIO（非阻塞 )

  • SpringWebflux执行过程和SpringMVC类似

  • SpringWebflux核心控制器 DispatchHandle 实际上是实现了接口WebHandler

  • HandlerResultHandler：响应结果处理
