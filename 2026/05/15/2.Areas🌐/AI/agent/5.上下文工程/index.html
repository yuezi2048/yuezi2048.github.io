5.上下文工程

发表于2026-05-15|更新于2026-05-15|2.Areas🌐AIagent
|总字数:5.6k|阅读时长:21分钟|浏览量:

context is all you need

我们主要学习上下文工程和两个相关工具：

  • ContextBuilder：基于GSSC (Gather-Select-Structure-Compress) 流水线，来统一管理上下文窗口
  • NoteTool：对智能体进行持久化的管理
  • TerminalTool：终端工具，对文件系统操作，进行即时上下文检索

请回答：上下文工程和早期提示词工程区别？为什么要上下文工程？上下文工程目标是什么？那怎么从上下文检索内容？长时程的任务你又怎么处理？

  • 区别？
    • 提示工程：关注如何编写与组织 LLM 的指令
    • 上下文工程：在推理阶段，如何策划与维护“最优的信息集合（tokens）”，回答的是什么样的上下文配置，最有可能让模型产出我们期望的行为？
      • 不仅包含提示本身，还包含其他会进入上下文窗口的一切信息。
  • 为什么？
    • 上下文腐蚀问题：LLM有注意力预算，token每增加，对应开销就会变大（注意力两两建模能力被拉薄了）
      • 尽管位置编码插值等方式确实可以适配长序列，但是以牺牲token位置理解为代价的。
    • 智能体不断循环运行下一轮的推理数据，必须周期性地提炼，所以我们有必要在候选集宇宙中，甄别哪些内容进入到有限的上下文窗口
      • 系统指令、工具、MCP（Model Context Protocol）、外部数据、消息历史等

什么算好的上下文：信息充分且紧致

  • 系统提示（System Prompt）：追求能完整勾勒期望行为的“最小必要信息集”，两个极端是过度硬编码、过于空泛
  • 工具（Tools）：最小可行工具集（MVTS），极端是工具臃肿（边界模糊）
  • 示例（Few-shot）：力求体现多样且典型，两个极端是列举所有边界条件、没有示例

怎么检索？我们正在从推理前一次性检索（embedding 检索）”逐步过渡到“及时（Just-in-time, JIT）上下文

  • 我们更多是维护轻量引用（路径 URL等）来便于 运行时动态加载数据，且使用head/tail等方式无须把整块塞入上下文。
    • 引用元数据：目录层级、命名约定、时间戳等也可以本身传达出目的和实效的信息
  • 渐进式披露：每一步交互会产生新上下文，指导下一步的操作。而我们维护文件大小–复杂度，命名–用途，时间戳–时效性这一系列元信息，智能体就可以层次化地理解，在工作记忆中保留我们需要的最小必要子集，像写笔记一样地逐步完善的持久化，而不是被 大而全 的历史信息搞垮。
  • Embedding检索和JIT上下文的权衡：速度 vs 启发式
    • 我们选择加载少量高价值上下文保证速度，然后让智能体自主探索（边界根据任务动态需求和时效性）
    • 工程上，维护一个Readme等文档，提供glob/grep等原语，让智能体实时读取高价值上下文信息，避开过时索引/语法树的沉没成本

长时程任务的上下文工程：压缩整合（Compaction）、结构化笔记（Structured note-taking）与子代理架构（Sub-agent architectures）。

  • 压缩整合（Compaction）：到达上下文上限，压缩后开启新的窗口，保证长程连贯性
    • 调参建议：先优化召回（确保不遗漏关键信息），再优化精确度（剔除冗余内容）；一种安全的“轻触式”压缩是对“深历史中的工具调用与结果”进行清理。
  • 结构化笔记（Structured note-taking）：固定频率将关键信息写入上下文外的持久存储，后续按需使用
    • 低开销维持持久状态与依赖关系：TODO 列表、项目 NOTES.md、关键结论/依赖/阻塞项的索引，跨数十次工具调用与多轮上下文重置后的进度保存
    • 注：在非编码场景中同样有效（如长期策略性任务、游戏/仿真中的目标管理与统计计数）。结合MemoryTool，可轻松实现文件式/向量式的外部记忆并在运行时检索。
  • 子代理架构（Sub-agent architectures）：主代理负责规划整合，派发专家子代理在干净的上下文深挖探索，最后传回凝练摘要 1k-2k tokens
    • 好处：实现关注点分离。庞杂的搜索上下文留在子代理内部，主代理专注于整合与推理；适合需要并行探索的复杂研究/分析任务。
    • 经验：公开的多智能体研究系统显示，该模式在复杂研究任务上相较单代理基线具有显著优势。

方法取舍：

  • 压缩整合：长对话连续性，强调上下文的“接力”。
  • 结构化笔记：适合 里程碑/阶段性成果的迭代式开发与研究。
  • 子代理架构：适合 复杂研究与分析，能从并行探索中获益。

ContextBuilder

目标：简单高效”，去除不必要的复杂性，统一以”相关性+新近性”的分数进行选择，符合 Agent 模块化与可维护性的工程取向。

  • 统一入口：将”获取(Gather)- 选择(Select)- 结构化(Structure)- 压缩(Compress)”这一套抽象为可复用流水线，减少在 Agent 实现中的重复上下文代码逻辑。

  • 稳定形态：输出固定骨架的上下文模板，便于调试、A/B 测试与评估。我们采用了分区组织的模板结构：

    • [Role & Policies]：明确 Agent 的角色定位和行为准则

    • [Task]：当前需要完成的具体任务

    • [State]：Agent 的当前状态和上下文信息

    • [Evidence]：从外部知识库检索的证据信息

    • [Context]：历史对话和相关记忆

    • [Output]：期望的输出格式和要求

  • 预算守护：在 token 预算内尽量保留高价值信息，对超限上下文提供兜底压缩策略。这确保了即使在信息量巨大的场景下，系统也能稳定运行。

  • 最小规则：不引入来源/优先级等分类维度，避免复杂度增长。实践表明，基于相关性和新近性的简单评分机制，在大多数场景下管理上下文已经足够有效。

在定义GSSC流水线之前，首先定义基本的数据类型

  • 基本上下文单元：候选信息包 ContextPacket，包含内容、时间戳、token 数量和相关性分数等核心属性
  • 配置统一管理ContextConfig：reserve_ratio参数是保证系统指令的预留比例

GSSC流水线

（1）Gather_多源信息汇集：关键在于容错性和灵活性。

  • 容错机制：每个外部数据源的调用都被 try-except 包裹，是处理 IO 密集型外部调用 的地道写法，确保单个源的失败不会影响整体流程
  • 优先级处理：系统指令被标记为高优先级relevance_score=1.0，确保始终被保留
  • 历史限制：对话历史只保留最近的几条conversation_history[-5:]，避免上下文窗口被历史信息占据
  • 扩展支持：支持custom_packets，更灵活加入自定义的信息

（2）Select_智能信息选择：根据相关性和新近性对候选信息进行评分和选择，这是核心，输出的信噪比决定了最终上下文质量

  • 评分机制：采用相关性和新近性的加权组合 $CombinedScore = (w_{rel} \times s_{relevance}) + (w_{rec} \times s_{recency})$，权重可配置
  • 贪心算法：按分数从高到低填充，确保在有限预算内选择最有价值的信息
  • 过滤机制：通过 min_relevance 参数过滤低质量信息

（3）Structure_结构化输出：将选中的信息组织成结构化的上下文模板

  • 可读性：清晰的分区让人类和模型都更容易理解上下文结构
  • 可调试性：问题定位更容易，可以快速识别哪个区域的信息有问题
  • 可扩展性：添加新的信息源只需要创建新的分区

（4）Compress_兜底压缩：对超限上下文进行压缩处理

体现了”保持结构完整性”的原则，即使在 token 预算紧张的情况下，也要尽量保留每个分区的关键信息。

  • 分区（Sectioning）： 使用 context.split("\n\n")。这在 Python 中是极其高效的操作。它假设上下文是由双换行符分隔的独立块（如：系统指令、一条笔记、一段对话）。
  • 贪心保留（Greedy Preservation）： 遍历这些 sections。如果当前已容纳的 Token 数加上这块完整的长度不超标，就将其原封不动地装入 compressed_sections。
  • 末尾截断（Truncation）： 当遇到第一个“装不下”的块时，程序不是直接丢弃，而是计算 remaining_tokens（剩余余量）。如果余量大于 50，就调用 _truncate_text 强行切出一部分装进去，并打上 [... 内容已压缩 ...] 的补丁。

使用

（1）基础使用

（2）运行效果展示_结构化的上下文输出：

  • [Role & Policies]：明确了 AI 的角色和回答要求
  • [Task]：清晰地表达了用户的问题
  • [Evidence]：从 RAG 系统检索的相关知识
  • [Context]：对话历史和相关记忆，提供了充分的背景信息
  • [Output]：指导 LLM 如何组织回答

（3）ContextBuilder与 Agent 集成

通过这种方式，ContextBuilder 成为了 Agent 的”上下文管理大脑”，自动处理信息的收集、筛选和组织，让 Agent 始终能够在最优的上下文下进行推理和生成。

最佳实践

  1. 动态调整 token 预算：根据任务复杂度动态调整 max_tokens，简单任务使用较小预算，复杂任务增加预算。
  2. 相关性计算优化：在生产环境中，将简单的关键词重叠替换为向量相似度计算，提升检索质量。
  3. 缓存机制：对于不变的系统指令和知识库内容，可以实现缓存机制，避免重复计算。
  4. 监控与日志：记录每次上下文构建的统计信息(选中信息数量、token 使用率等)，便于后续优化。
  5. A/B 测试：对于关键参数(如相关性权重、新近性权重)，通过 A/B 测试找到最优配置。
