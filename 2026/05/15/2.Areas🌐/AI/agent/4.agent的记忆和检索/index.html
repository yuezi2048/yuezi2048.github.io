记忆能力

为什么要引入记忆能力？

  • 对话无状态：这就导致上下文丢失，进而会出现回答问题时前后矛盾的问题，同时无法个性化、无法从过往经验学习
  • 内置知识局限性：LLM训练数据是静态、有限的，从而会有时效性、专业性、事实准确性（偏见）、可解释性的怀疑

解决思路，参考人类的记忆过程：总结为短期的感知记忆、当前任务的短期记忆、还有长期记忆（技能、语言表达的情景 / 知识）

  1. 感觉记忆（Sensory Memory）：持续时间极短（0.5-3秒），容量巨大，负责暂时保存感官接收到的所有信息
  2. 工作记忆（Working Memory）：持续时间短（15-30秒），容量有限（7±2个项目），负责当前任务的信息处理
  3. 长期记忆（Long-term Memory）：持续时间长（可达终生），容量几乎无限，进一步分为：
    • 程序性记忆：技能和习惯（如骑自行车）
    • 陈述性记忆：可以用语言表达的知识，又分为：
      • 语义记忆：一般知识和概念（如”巴黎是法国首都”）
      • 情景记忆：个人经历和事件（如”昨天的会议内容”）

那么在人类记忆基础上，我们就得到了实现记忆的整体思路：

  • 编码（Encoding）
  • 存储（Storage）：内存 / 事件索引 / 向量数据库 / 知识图谱
    • 检索（Retrieval）：提取信息
    • 整合（Consolidation）：短期记忆整理为长期
    • 遗忘（Forgetting）：剔除不重要的信息

展开后的实现思路，我们细化记忆：

  • 短期记忆
    • 工作记忆 (Working Memory)：短期记忆角色，存储当前对话上下文信息
      • 容量被刻意限制，生命周期绑定会话，结束后清理
    • 感知记忆 (Perceptual Memory)：专门处理图像、音频等多模态信息，并支持跨模态检索。
      • 生命周期会根据信息的重要性和可用存储空间进行动态管理。
  • 长期记忆
    • 语义记忆 (Semantic Memory)，长期记忆抽象知识，是组成知识体系和进行关联的核心
      • 的用户偏好、需要长期遵守的指令或领域知识点，都适合存放在这里
    • 情景记忆 (Episodic Memory)：长期记忆智能体具体事件的学习情景，有完整的上下文信息
      • 能够按照时间或者主题 搜索，是智能体复盘的基础

四种记忆实现

工作记忆：作为最活跃的存储临时信息，重点是如何快速访问和自动清理 ==> 响应速度 和 资源效率

  • 方案：纯内存(上限默认50条) + TTL清理机制，重启后丢失，但也刚好符合工作记忆存储临时信息。
  • 检索方案：混合检索策略（可靠性），评分考虑语义相似度7+时间3+重要性： (相似度 * 0.7 + 时间衰减 * 0.3) × (0.8 + 重要性 × 0.4)
    • 首先使用TF-IDF向量化语义检索
    • 失败了回退关键词匹配

情景记忆：负责具体的事件记录，重点是保留事件完整性，时间序列关系

  • 情景记忆追溯：(Session_id, episode_id) -> (context, content, timestamp)
  • 方案：SQLite（结构化存储+复杂查询支持） + Qdrant（高效向量检索） 混合存储方案
  • 整体的检索，同样考虑相似度8+时间2+重要性0.8~1.2：(向量相似度 × 0.8 + 时间近因性 × 0.2) × (0.8 + 重要性 × 0.4)

语义记忆：这是最复杂的，需要存储抽象的知识，重点是知识结构化和推理能力

  • 引入预训练模型，对文本进行图嵌入表示
  • 解决方案：Neo4j图数据库 + Qdrant向量数据库 混合架构，前者便于关系推理，后者便于语义检索

语义记忆的添加（知识图谱构建）：

  • 在记忆本体embedding基础上，还需要提取实体entity和关系relation，构建结构化的知识表示：

检索：同样是混合搜索策略，结合了向量检索的语义理解能力和图检索的关系推理能力：

语义记忆的评分公式为：(向量相似度 × 0.7 + 图相似度 × 0.3) × (0.8 + 重要性 × 0.4)

  • 向量检索权重（0.7）：更倾向于结果保证语义相似
  • 图检索权重（0.3）：关系推理作为一种补充，挖掘隐含关联
  • 重要性权重范围[0.8, 1.2]：一方面是重要性的考量，另一方面控制重要性的权重范围

感知记忆：存储文本、图像、音频等多种模态，需要考虑的问题是维度不匹配的问题

  • 采用模态分离的策略，每个模态存储于vector_store[modal_type]下，解决维度不匹配，并支持检索功能。
  • 同模态检索：通过专门的编码器精确匹配
  • 跨模态检索：引入更复杂的语义对齐机制
  • 评分考虑：(向量相似度 × 0.8 + 时间近因性 × 0.2) × (0.8 + 重要性 × 0.4)
  • 当进行跨模态检索时，系统会自动调整评分权重，确保检索结果的多样性和准确性。
  • 感知记忆中的时间近因性计算采用了指数衰减模型：模拟了人类记忆中的遗忘曲线，确保了感知记忆系统能够优先检索到时间上更相关的记忆内容。

检索能力

检索增强生成（Retrieval-Augmented Generation，RAG）：

  • 数据准备阶段：数据提取、文本分割和向量化

  • 应用阶段：用户提问：

    • 检索：从知识库中查询相关内容；

      • 早期朴素RAG 2020：依赖关键词匹配TF-IDF / BM25，依赖词频评估相关性，字面匹配好但语义匹配差
      • 高级RAG 2022：引入稠密嵌入（Dense Embedding），考虑文本转换成高维向量，从而匹配语义相关性，引入了查询重写、文档分块的优化
      • 模块化RAG 2023：强调系统的组合性，一系列的查询增强策略，引入混合检索、多查询拓展MQE、假设性文档嵌入HyDE策略

    • 增强：将检索结果注入提示词；

      • 早期朴素RAG：直接拼接提示词，给生成模型
      • 高级RAG：引入重排序的优化

    • 生成：输出兼具准确性与透明度的答案。

高级检索策略

多查询拓展MQE：通过生成语义等价的多样化查询来提高检索召回率的技术。

优势：能够自动理解用户查询的多种可能含义，特别是对于模糊查询或专业术语查询效果显著。

  • 多查询的处理
    • splitlines()：将 LLM 返回的一整块文本按行切割成列表。
    • strip("- \t")：去掉 LLM 习惯性生成的转义符（如 - 或制表符），只保留纯查询词。
  • 容错：切片or 回退 outs[:n] or [query]。如果 LLM 挂了或没返回内容，则至少返回原始查询词，确保流程不中断。

假设文档嵌入（HyDE）：核心思想是”用答案找答案”。因为

优势：传统是通过问题匹配文档，而问题通常是疑问句，文档是陈述句，所以HyDE就先预设一个假设性的段落去匹配，缩小了语义鸿沟，特别是专业知识。

扩展检索框架

  • 通过enable_mqe和enable_hyde参数，自主选择是否启用，MQE在性能敏感或者token敏感场景下禁用，高召回率下可以同时启用
  • 思路
    • MQE多样化查询+HyDE假设文档
    • 对每个查询并行进行检索，值得学习的是动态候选池：通过candidate_pool_multiplier参数（默认为4）动态控制搜索范围（默认4倍）
      • 比如生成了5个扩展查询，每个查询就去库里搜per个结果，最终就能得到巨大的候选池，保证能得到top-k个结果
    • 去重和排序后合并为top-k
      • where 在向量库层面进行“租户隔离”。保证只有当前 rag_namespace 且标记为 is_rag_data 的数据才会被命中，防止多项目并行时的数据串门。
      • 使用 agg 字典进行去重：如果同一个 memory_id 被多个查询搜到了，代码会判断 s > float(agg[mid].get("score", 0.0))，只保留评分最高的记录
      • 最后按照分数 reverse=True 进行全局排序。

  • recall：从记忆系统中检索学习历程
  • get_stats：获取当前会话的统计信息
  • generate_report：生成详细的学习报告并保存为JSON文件

总结

在本章中，我们成功地为HelloAgents框架增加了两个核心能力：记忆系统和RAG系统。
