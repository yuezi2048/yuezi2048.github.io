2.常用agent框架分析

发表于2026-05-15|更新于2026-05-15|2.Areas🌐AIagent
|总字数:9.3k|阅读时长:32分钟|浏览量:

框架开发实践

框架的价值：

  • 代码复用：提供通用的BaseAgent，并封装Agent基本的ReAct/PlanAndSolve等循环
  • 组件的解耦和可扩展性：
    • 模型层：支持替换不同的模型厂商
    • 工具层：标准化定义工具，注册，执行，添加工具不影响其他代码。
    • 记忆层：长期记忆/短期记忆，根据需求可以切换不同记忆策略（滑动窗口/摘要记忆等），更换/升级组件会更加清晰
  • 状态管理标准化：我们之前ReflectionAgent中简单的用列表来表示状态，但真实的长时间运行的智能体需要考虑上下文窗口限制、历史信息持久化、多轮对话状态跟踪等问题，框架可以提供通用的状态管理机制，我们无须过于关注这些问题
  • 可观测性和调试过程：如事件回调机制，在关键节点自动触发日志上报

  • AutoGen：思想是将多智能体的协作看成是群聊，通过定义不同的角色+交互规则（Coder写完代码后Tester接管），如此对话——协作——迭代——完成目标，适合流程化的任务
  • AgentScope：亮点是工程化和易用性，提供了友好的编程接口定义智能体、构建通信网络（内置消息传递机制 分布式部署支持）、管理应用声明周期，大规模、运维复杂的多智能场景非常适用
  • CAMEL：他更强调角色扮演，为两个智能体设定好角色和共同目标后，就能在初始提示的引导下，自主多轮对话，相互启发，共同完成任务，复杂度更低，涌现效果好，适合创意性任务
  • LangGraph：亮点是将交互过程看成是状态图，这突破了传统的信息单向流动，我们每一步Agent的操作（调用LLM，执行工具）看成是图中的一个节点，通过边来定义跳转逻辑，这种结构天然支持循环，易于实现Reflection的迭代、修正，非常简单和直观。

AutoGen

0.7.4版本架构：

  • 分层设计
    • autogen-core：底层核心，封装模型交互，消息传递，保证稳定性/拓展性
    • autogen-agentchat：在 core 上层，提供开发智能体应用接口，符合职责单一性。
  • 异步优先：该架构全面转向异步编程 (async/await)，在等待LLM响应时可以运行其他任务
  • 核心组件（明确区分了Assistant的思考和执行）
    • AssistantAgent (助理智能体)： 核心，内部封装LLM，根据历史对话生成恢复，通过System Message可以赋予不同的角色
    • **UserProxyAgent (用户代理智能体)：**这是AutoGen的特色，这里的用户有两个角色：
      • 充当人类，发起任务
      • 充当执行器，执行代码/调用工具，反馈给其他智能体

工作思路是轮询群聊 (RoundRobinGroupChat)： 确定性、顺序的对话协调方案，非常适合流程固定的任务，如软件开发流程中产品经理——工程师——代码审计

  1. 首先，创建一个 RoundRobinGroupChat 实例，并将所有参与协作的智能体（如产品经理、工程师等）加入其中。
  2. 当一个任务开始时，群聊会按照预设的顺序，依次激活相应的智能体。
  3. 被选中的智能体根据当前的对话上下文进行响应。
  4. 群聊将新的回复加入对话历史，并激活下一个智能体。
  5. 这个过程会持续进行，直到达到最大对话轮次或满足预设的终止条件。

案例：软件开发团队

开发实时显示比特币当前价格，需要完整覆盖软件开发的典型环节：从需求分析、技术选型、编码实现到代码审查和最终测试。

目标：感受自然的对话驱动协作、角色专业化分工、流程自动化管理和完整的开发闭环。

客户端配置，0.7.4版本提供了OpenAIChatCompletionClient可以对接OpenAI API规范的模型（对于非 OpenAI 系列的模型，需要OpenAIChatCompletionClient 的参数中传入模型信息字典）

智能体角色配置：关键是编写高质量的System Message, 职责确认：

  • ProductManager (产品经理): 负责将用户的模糊需求转化为清晰、可执行的开发计划。
  • Engineer (工程师): 依据开发计划，负责编写具体的应用程序代码。
  • CodeReviewer (代码审查员): 负责审查工程师提交的代码，确保其质量、可读性和健壮性。
  • UserProxy (用户代理): 代表最终用户，发起初始任务，并负责执行和验证最终交付的代码。

服务编排

  • 参与者顺序: participants 列表的顺序决定了智能体发言的先后次序。
  • 终止条件: termination_condition 是控制协作流程何时结束的关键。这里我们设定，当任何消息中包含关键词 “TERMINATE” 时，对话便结束。在我们的设计中，这个指令由 UserProxy 在完成最终测试后发出。
  • 最大轮次: max_turns 是一个安全阀，用于防止对话陷入无限循环，避免不必要的资源消耗。

执行

分析

  • 优势
    • 流程透明：将流程自然映射成人类团队协作模式，精力更多可以放在角色定义和每个角色职责上，不需要精细控制流程的走向
    • 便于维护：一个精心设计提示词的智能体可以复用
    • 流程可控（human-in-loop）：这种设计，人类可以作为发起者，也可以作为监督者/验收者，保证自动化系统可控
  • 缺陷：
    • 不确定性：可能会走到意外分支，死循环问题
    • 调试困难：调试时面对的是长串的对话历史，这种“对话式调试”定位问题点是非常棘手的

AgentScope

如果说 AutoGen 的是灵活的”对话工作室”，那么达摩院开发的agentScope 就是一个完整的”智能体操作系统”，在组合式架构和消息驱动模式的工程实践下，提供了分布式部署、容错恢复、可观测性等企业级特性

  • 基础组件层 (Foundational Components)
    • Message定义了统一的消息格式，从简单的文本到复杂的多模态
    • Memory提供了长短期记忆管理
    • Model API抽象了不同大预言模型调用
    • Tool封装了智能体和外部世界交互
    • 更高级的抽象：预构建智能体（浏览器等），经典ReAct范式、智能体钩子、并行工具调用、状态管理等，原生支持异步执行与实时控制
  • 多智能体协作层 (Multi-Agent Cooperation)
    • MsgHub 作为消息中心，负责智能体间的消息路由和状态管理
    • Pipeline 系统则提供了灵活的工作流编排能力，支持顺序、并发等多种执行模式
  • 开发与部署层 (Deployment & Development)
    • AgentScope Runtime 提供了生产级的运行时环境
    • AgentScope Studio 则为开发者提供了完整的可视化开发工具链。

AgentScope的核心创新就是消息驱动架构，智能体交互抽象为消息接收和发送，天然支持了：

  • 异步解耦: 消息的发送方和接收方在时间上解耦，无需相互等待，天然支持高并发场景。
  • 位置透明: 智能体无需关心另一个智能体是在本地进程还是在远程服务器上，消息系统会自动处理路由。
  • 可观测性: 每一条消息都可以被记录、追踪和分析，极大地简化了复杂系统的调试与监控。
  • 可靠性: 消息可以被持久化存储和重试，即使系统出现故障，也能保证交互的最终一致性，提升了系统的容错能力。

每个智能体有明确的生命周期，基于基类AgentBase实现，解耦了内部逻辑和外部通信，开发者通常只需关心核心的reply方法

消息传递：AgentScope中有一个**消息中心 (MsgHub)**作为整个消息驱动的中枢，他除了负责消息的路由和分发，还支持分布式通信和持久化的功能。

  • 灵活的消息路由: 支持点对点、广播、组播等多种通信模式，可以构建灵活复杂的交互网络。
  • 消息持久化: 能够将所有消息自动保存到数据库（如 SQLite, MongoDB），确保了长期运行任务的状态可以被恢复。
  • 原生分布式支持: 这是 AgentScope 的标志性特性。智能体可以被部署在不同的进程或服务器上，MsgHub 会通过 RPC（远程过程调用）自动处理跨节点的通信，对开发者完全透明。

案例：三国狼人杀

目标：体验每个智能体不仅完成狼人杀的任务，还要体现三国人物的性格特点和行为模式，感受多层次角色建模

架构设计，每个层次同时映射了AgentScope对应的组件：

  • 游戏控制层 (Game Control Layer)：由一个 ThreeKingdomsWerewolfGame 类作为游戏的主控制器，负责维护全局状态（如玩家存活列表、当前游戏阶段）、推进游戏流程（调用夜晚阶段、白天阶段）以及裁定胜负。
  • 智能体交互层 (Agent Interaction Layer)：完全由 MsgHub 驱动。所有智能体间的通信，无论是狼人间的秘密协商，还是白天的公开辩论，都通过消息中心进行路由和分发。
  • 角色建模层 (Role Modeling Layer)：每个玩家都是一个基于 DialogAgent 的实例。我们通过精心设计的系统提示词，为每个智能体注入了“游戏角色”和“三国人格”的双重身份。

思路：以消息驱动代替状态机来管理游戏流程。

狼人阶段，通过MsbHub创建临时，仅狼人玩家的私密通信频道

游戏逻辑遵循的范式：在特定上下文中，以何种模式进行消息交换，而不是僵硬的状态转换。

游戏规则通过结构化的输出约束（Field修饰）

通过提示词工程，同时扮演两个角色（游戏+三国人物特征），这样就可以在游戏过程中看到曹操的伪装，张飞的冲动

并发容错机制：得益于异步架构，在例如投票场景下，可以同时收集多个智能体的决策：

fanout_pipeline允许我们并行地向所有智能体发送相同的消息，并异步收集它们的响应。

  • 这不仅提高了游戏的执行效率，更重要的是模拟了真实狼人杀游戏中”同时投票”的场景。

在关键环节加入容错处理

分析

  • 原生并发：AgentScope通过消息驱动架构，将游戏流程优雅的映射成一系列并发、消息传递事件，避免传统状态机的僵硬
  • 异常兜底：结合结构化输出能力，将游戏规则在代码层面约束，并且如果有当个出现异常也有容错兜底
  • 代价：工程化的复杂性成本，异步编程和分布式通信门槛较高，同时对于简单的多智能体对话，这种架构可能存在过度工程化风险。对于快速原型和简单应用，轻量级的框架可能更合适

Camel

Canmel能自主协作的核心：角色扮演 (Role-Playing) 和 引导性提示 (Inception Prompting)。

早期角色有两类：AI User（军师，下达指令） + AI Assistant（干活的，执行），两者相互协同。

而最关键的是引导性提示，在对话开始前注入一段行动纲领作为初始指令，保证不能偏离主题，陷入无效循环，一般提示由这些组成：

  • 明确自身角色：例如，“你是一位资深的股票交易员…”

  • 告知协作者角色：例如，“你正在与一位优秀的 Python 程序员合作…”

  • 定义共同目标：例如，“你们的共同目标是开发一个股票交易策略分析工具。”

  • 设定行为约束和沟通协议：这是最关键的一环。例如，指令会要求 AI 用户“一次只提出一个清晰、具体的步骤”，并要求 AI 助理“在完成上一步之前不要追问更多细节”，同时规定双方需在回复的末尾使用特定标志（如 <SOLUTION>）来标识任务的完成。

案例：心理电子书

目标：创建一个面向普通读者的拖延症科普电子书，要有可读性和科学严谨性

角色设定：

  • 心理学家（Psychologist）：具备深厚的心理学理论基础，熟悉认知行为科学、神经科学等相关领域，能够提供专业的学术见解和实证研究支持
  • 作家（Writer）：拥有优秀的写作技巧和叙述能力，善于将复杂的学术概念转化为生动易懂的文字，注重读者体验和内容的可读性

首先定义共同目标，作为后续的引导性提示

接下来提供user Role和Assistant Role

通过Cammel的RolePlaying这个API，很方便地将负责规划结构的“作家”分配给 user_role_name，将负责提供专业知识的“心理学家”分配给 assistant_role_name。

最终开始协作，init_chat初始化我们之前的对话定义，通过step来驱动一轮

最终就能得到一个完整的讨论结果

一阶段 (约 1-5 轮): 框架搭建与目标对齐 在对话的初期，“作家”智能体首先会扮演起主导者的角色，提出对电子书整体结构和章节安排的初步设想。随后，“心理学家”会从其专业角度对这个框架进行审视和补充，确保核心的学术模块（如理论基础、关键概念等）没有遗漏，从而在协作开始之初就对最终产出物达成共识。

第二阶段 (约 6-20 轮): 核心内容生成与知识转译 这是最高效的内容创作阶段。协作模式会变为一种稳定的“请求-响应”循环：

  • 心理学家：负责提供“硬核”的专业知识，如对“时间折扣理论”、“执行功能缺陷”等核心概念的科学解释，并引用相关的实验研究来支撑观点。
  • 作家：则发挥其“翻译官”的作用，将这些严谨但可能晦涩的学术概念，转化为生动、形象的比喻和贴近生活的案例。例如，它可能会将“大脑中的‘现在偏见’”这个概念，比作“一个只顾眼前糖果、不顾长远健康的任性孩子”。

第三阶段 (约 21-25 轮): 迭代优化与质量保证 当书籍的主体内容完成后，对话的重心会转移到对已有文本的打磨和完善上。此时，两位智能体的角色会发生微妙的变化：

  • 作家：更侧重于审视文章的整体流畅性、逻辑衔接和语言风格，从“读者体验”出发提出修改建议。
  • 心理学家：则再次扮演“事实核查员”，确保在转译和润色的过程中，核心知识的科学准确性没有丢失，并为某些观点补充更有力的实证研究支持。

第四阶段 (收尾): 总结与升华 在最后的几轮对话中，双方会协作完成实用建议的总结和全书的回顾，确保电子书有一个清晰、有力的结尾，为读者留下深刻印象并提供实际价值。

分析

  • 和AgentScope这类重工程的框架相比，最大优势就是重提示、轻架构，特点表现为自然涌现的协作，比硬编码的工作流更高效和灵活
  • 局限：提示词的高度依赖（随之带来的是调试复杂、不同LLM的一致性挑战）、缺乏复杂对话路由机制（分布式状态管理、有效的仲裁机制）
  • 场景分析
    • Cammel适合的是深度协作和创造性的任务
    • 严格流程控制可能LangGraph图结构更靠谱、大规模并发可能AgentScope的消息驱动更合适、复杂的决策树通过AutoGen的群聊模式更灵活

LangGraph

LangGraph将智能体的流程建模为状态机，表现为有向图（节点是计算步骤，边是一个节点到另一个节点的跳转逻辑），从而天然支持循环，使得进行迭代、反思、自我修正，更直观、简单。

三个基本要素：全局状态State、节点Node、边Edge

全局状态：整个图围绕的共享对象，里面是我们需要跟踪的状态信息（历史信息、当前任务、最终答案、迭代次数等），一般用TypedDict列举

节点：定义输入（接收当前状态），输出函数（返回更新后的状态），是图具体工作的单元。

边：来对节点工作流进行操作，指定了一个节点的输出流向另一个固定的节点。

  • 他有一个强大的功能是条件边，根据一个函数判断当前状态，根据不同的条件分支动态决定下一步要跳转的节点

最终拼接这些组件，组成图

案例：三步问答助手

我们需要一个遵循清晰、固定的三步回答用户问题的场景：

  1. 理解 (Understand)：首先，分析用户的查询意图。
  2. 搜索 (Search)：然后，模拟搜索与意图相关的信息。
  3. 回答 (Answer)：最后，基于意图和搜索到的信息，生成最终答案。

首先定义全局状态贯穿于整个工作流，这是一个共享的数据结构，它在图的每个节点之间传递，作为工作流的持久化上下文

  • 全局状态：SearchState: TypedDict
  • 核心字段：user_query: str，search_query: str 表示用户输入提炼后的关键词

接下来定义节点，具体就是定义执行具体任务的 Python 函数。这些函数接收当前的状态对象作为输入，并返回一个包含更新后字段的字典。

首先初始化函数需要的Tavily搜索服务环境变量

接下来创建三个节点：

理解与查询节点：理解用户意图，并为其生成一个最优化的搜索查询。

LLM完成了“意图理解”和“关键词生成”的任务，结果填入search_query 字段

搜索节点：调用 Tavily API 进行真实的互联网搜索，并具备基础的错误处理

  • 通过tavily_client.search 发起 API 调用。它被包裹在 try...except 块中，用于捕获可能的异常。
    • 如果搜索失败，它会更新 step 状态为 "search_failed"，这个状态将被下一个节点用来触发备用方案。

回答节点：能够根据上一步的搜索是否成功，来选择不同的回答策略

检查 state["step"] 的值来执行条件逻辑

构建图，边则通过节点的名称直接相连即可，最终编译就可以得到最终的图

分析

  • 优势：

    • 高度的可控和可预测性：最大的优势就是对循环的原生支持，这意味着可以很容易构建反思-修正的循环，比如搜索失败后可以回退到备用方案
      • 每个节点都是一个函数，这样子的模块化设计，是非常可控的，对于人机协作”（Human-in-the-loop）提供了坚实的基础

  • 缺陷

    • 需要定义状态、节点、边一系列的构建，前期代码工作比较大，开发者需要在做什么（What）基础上，还得思考如何构建流程（how）。
    • 可控意味着缺乏对话式的动态、涌现式的交互，所以对应的场景就是确定、可靠的流程，而不是开放式不可预测的社会性协作
    • 调试比较困难，节点内部、节点的传递、边传递的条件等，需要对整个图有全局的理解

总结

我们看到，每一个框架都有自己实现智能体构建的思路：

  • AutoGen 将复杂的协作抽象为一场由多角色参与的、可自动进行的“群聊”，其核心在于“以对话驱动协作”。
  • AgentScope 则着眼于工业级应用的健壮性与可扩展性，为构建高并发、分布式的多智能体系统提供了坚实的工程基础。
  • CAMEL 以其轻量级的“角色扮演”和“引导性提示”范式，展示了如何用最少的代码激发两个专家智能体之间深度、自主的协作。
  • LangGraph 则回归到更底层的“状态机”模型，通过显式的图结构赋予开发者对工作流的精确控制，尤其是其循环能力，为构建可反思、可修正的智能体铺平了道路。

通过对这些框架的深入分析，我们可以提炼出一个设计的权衡：“涌现式协作”与“显式控制”之间的选择。

  • AutoGen 和 CAMEL 更多地依赖于定义智能体的“角色”和“目标”，让复杂的协作行为从简单的对话规则中“涌现”出来，这种方式更贴近人类的交互模式，但有时难以预测和调试。
  • 而 LangGraph 要求开发者明确地定义每一个步骤和跳转条件，牺牲了一部分“涌现”的惊喜，换来了高度的可靠性、可控性和可观测性。
  • 同时，AgentScope 则揭示了第二个同样重要的维度：工程化。无论我们选择哪种协作范式，要将其从实验原型推向生产应用，都必须面对并发、容错、分布式部署等工程挑战。AgentScope 正是为解决这些问题而生，它代表了从“能运行”到“能稳定服务”的关键跨越。
