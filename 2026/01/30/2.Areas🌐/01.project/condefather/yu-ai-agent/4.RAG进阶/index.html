RAG进阶学习

文档收集与切割（ETL）

ETL

SpringAI-ETL

ETL中的文档实际上不局限于文本，还有一系列的元信息MetaData、多媒体附件MediaContent

而针对于ETL，遵循一定的流程，对应着三大组件：

  • 读取文档：DocumentReader负责从数据源加载文档
  • 转换文档：根据需求将文档处理为指定的格式（如去除冗余信息、分词、词性标注等），可通过DocumentTransformer实现
  • 写入文档：DocumentWriter将文档以特定格式保存到存储中，如以嵌入向量的形式写入向量数据库，或者通过k-v键值对保存到Redis等数据库中

Extract_DocumentReader

实现了Supplier函数式接口，从数据源中得到Document集合

其内置实现类：

  1. JsonReader: 读取 JSON 文档
  2. TextReader: 读取纯文本文件
  3. MarkdownReader: 读取 Markdown 文件
  4. PDFReader: 读取 PDF 文档, 基于 Apache PdfBox 库实现
    • PagePdfDocumentReader: 按照分页读取 PDF
    • ParagraphPdfDocumentReader: 按照段落读取 PDF
  5. HtmlReader: 读取 HTML 文档, 基于 jsoup 库实现
  6. TikaDocumentReader: 基于 Apache Tika 库处理多种格式的文档, 更灵活

关于ETL的文档：ETL

如果想要自定义文档读取器并不难，核心就是解析文档，转换成document列表

Transform_DocumentTransformer

源码中，实现了 Function<List<Document>, List<Document>>接口，将一组文档转换成另一组文档，这是保证ETL效果的核心步骤，如何将大文档合理拆分为便于检索的知识碎片

SpringAI官方提供了三类：

  • TextSplitter文本分割器

  • MetaDataEnricher元数据增强器

  • ContentFormatter内容格式化工具

TextSplitter

其实现类有TokenTextSplitter，基于token划分边界（考虑句子末尾等来创建有意义的文本段落）

构造函数：

  1. TokenTextSplitter(): 使用默认设置创建分割器。
  2. TokenTextSplitter(int defaultChunkSize, int minChunkSizeChars, int minChunkLengthToEmbed, int maxNumChunks, boolean keepSeparator) : 使用自定义参数创建分割器, 通过调整参数, 可以控制分割的粒度和方式, 适应不同的应用场景。

参数说明 (无需记忆):

  • defaultChunkSize: 每个文本块的目标大小 (以 token 为单位, 默认值: 800)。
  • minChunkSizeChars: 每个文本块的最小大小 (以字符为单位, 默认值: 350)。
  • minChunkLengthToEmbed: 要被包含的块的最小长度 (默认值: 5)。
  • maxNumChunks: 从文本中生成的最大块数 (默认值: 10000)。
  • keepSeparator: 是否在块中保留分隔符 (如换行符) (默认值: true)。

官方文档有对 Token 分词器工作原理的详细解释，简单了解下:

  1. 使用 CL100K_BASE 编码将输入文本编码为 token。
  2. 根据 defaultChunkSize 将编码后的文本分割成块。
  3. 对于每个块:
    • 将块解码回文本。
    • 尝试在 minChunkSizeChars 之后找到合适的断点 (句号、问号、感叹号或换行符)。
    • 如果找到断点, 则在该点截断块。
    • 修剪块并根据 keepSeparator 设置选择性地删除换行符。
    • 如果生成的块长度大于 minChunkLengthToEmbed, 则将其添加到输出中。
  4. 这个过程会一直持续到所有 token 都被处理完或达到 maxNumChunks 为止。
  5. 如果剩余文本长度大于 minChunkLengthToEmbed, 则会作为最后一个块添加。

MetaDataEnricher

核心就是给文档补充元信息，便于后续检索，不影响文档本身的切分规则，有两种实现类：

  • KeywordMetadataEnricher：使用AI提取关键词添加元数据
  • SummaryMetadataEnricher：AI生成文档摘要添加到元数据，不仅可以对当前文档生成，还可以关联前一个和后一个文档，让摘要更完整

ContentFormatter

一般是用于统一文档格式，具体功能：

  • 文档格式化：支持将文档内容和元数据合并为特定格式字符串
  • 元数据过滤：根据不同的元数据模式（MetadataMode）筛选需要保留的元数据项
    • ALL：保留所有、NONE：移除所有
    • INFERENCE：推理场景，排除指定的推理元数据
    • EMBED：嵌入场景，排除指定的嵌入元数据
  • 自定义模板
    • 元数据模板：控制元数据项展示方式
    • 元数据分隔符：控制多个元数据的分隔方式
    • 文本模板：控制元数据与内容如何结合

Load_DocmentWriter

实现了Consumer<List<Document>>接口，负责将转换后的文件写入到目标存储。

有两种内置的DocumentWriter实现：

  • FileDocumentWriter：写入文件
  • VectorStoreWriter：写入向量数据库

也支持写入多个存储系统，定义多个Writer即可。

向量转换与存储

VectorStore接口

官方提供向量数据库接口VectorStore方便接入第三方向量存储，Milvus、Redis、PGVector、Elasticsearch等，提供基本的crud以及计算相似度 获取原生客户端的方法

SpringAI提供了SearchRequest类，构建相似度搜索请求

  • query：搜索查询文本
  • topK：最大结果数，默认4
  • similarityThreshold：过滤掉低于阈值的相似度
  • filterExpression：基于文档元数据的过滤，语法类似SQL，在官方文档中查阅语法即可

向量存储工作机制

  • 嵌入转换：通过嵌入模型将文本转换为嵌入向量
  • 相似度计算：查询文本转化为向量，计算该向量和向量数据库中的相似度
  • 相似度度量：余弦相似度、欧式距离、点积等
  • 过滤和排序：通过相似度过滤结果，返回topK最相关的文档

批处理优化

有时候我们会处理大量文档，而一般嵌入模型会有一个上下文的最大标记限制，即上下文窗口大小（context window size），从而限制了单个嵌入请求可以处理的文本量，如果一次性调用太多可能会出现报错，可能出现数据不完整的问题。

解决思路是使用批处理的策略，将文档分解成若干小批次，适合嵌入模型的上下文最大窗口大小，可以提高性能，并且更有效利于API速率限制。

Spring AI定义了BatchingStrategy接口 提供该功能，是基于文档的标记计数，来实现分批处理文档的

  • bastch方法，接收一个文档列表，分解成一个文档分批后的列表

默认实现：customTokenCountBatchingStrategy是通过估算文档的token数，划分到不超过token数的批次中，如果超出限制就抛出异常，从而保证每个批次不超过最大输入token数

我们根据业务需求可以自定义实现，如数据库每秒只能插入1w条数据，通过这个可以控制速率，也可以进行额外的日志记录和异常处理

文档过滤与检索

文档过滤可以拆分为：检索前、检索时、检索后，分别针对每个阶段提供了可自定义的组件。

  • 在预检索阶段，系统接收用户的原始查询，通过查询转换和查询扩展等方法对其进行优化，输出增强的用户查询。
  • 在检索阶段，系统使用增强的查询从知识库中搜索相关文档，可能涉及多个检索源的合并，最终输出一组相关文档。
  • 在检索后阶段，系统对检索到的文档进行进一步处理，包括排序、选择最相关的子集以及压缩文档内容，输 出经过优化的相关文档集。

预检索：增强用户输入

  • 查询重写转换器RewriteQueryTransformer，对用户原始输入进行改写，使得更加清晰和详细，在用户输入一些含糊或者模糊信息时非常有用
    • 原理就是内部提供了改写提示词，或者通过构造方法的promptTemplate参数自定义组件模板
  • 翻译转换器TranslationQueryTransformer，翻译为模型支持的目标语言，针对特定语言训练的模型而提示词不是目标语言，非常有用
    • 原理也是通过一段prompt实现，但建议实际用第三方翻译脚本，成本相对AI调用会更低一些
  • 查询压缩器CompressionQueryTransformer，将大语言模型的对话历史和后续查询压缩为一个独立查询，适合较长对话，且后续上下文相关的场景，原理同上
  • 多查询拓展，将大语言模型扩展为多个语义的不同变体，有助于检索额外上下文信息，并找到相关结果的机会
    • 理解：搜索内容搜不到的时候，尝试换一种表达就可以找到想要的内容
    • 下面的案例：就会拓展为 给出百度这家公司的公司简介，以及核心业务
    • 默认是包含原始查询的，通过includeOriginal(false)可以不包含
    • 源码中通过AI获得拓展，根据换行符分割

检索：提高查询相关性

文档检索器 DocumentRetrever

SpringAI官方有一个文档检索器DocumentRetrever，不同的存储方案也有自身的检索器实现类，如VectorStoreDocumentRetriever，可实现基于元数据的过滤、相似度阈值、返回结果数的搜索参数

  • 通过filterExpression可以灵活指定过滤条件，另外也提供了Query对象的FILTER_EXPRESSION过滤表达式参数

    • Query query = Query.builder()
    .text("百度创始人是谁？")
    .context(Map.of(VectorStoreDocumentRetriever.FILTER_EXPRESSION, "type == 'boy'"))
    .build();
List<Document> retrievedDocuments = documentRetriever.retrieve(query);

文档合并 documentJoiner

SpringAI内置documentJoiner合并器，通过连接操作，将多个查询和多个数据源合并为单个文档集合（重复文档保留首次出现的）

  • 在源码中，连接的思路就是将Map展开成二维列表，再将二维列表展开文档列表，最后去重，用了优雅的Stream流方法实现

检索后：优化文档处理

处理检索的文档，从而获得最佳的结果，解决的问题：“丢失在中间“的问题（由于模型上下文限制、检索信息中的噪音或冗余等）

具体的方式：

  • 根据查询相关性对文档排序
  • 删除不相关/冗余文档
  • 压缩每个文档内容减少噪音/冗余

这一部分不是实际的重点，且迭代速度非常快。如在M8版本引入了DocumentPostProcessor

查询增强与关联

这是RAG最终缓解，将文档和用户查询结合，为AI提供上下文，从而生成最准确的回答。我们知道SpringAI提供了两种拦截器实现增强

QuestionAnswerAdvisor

Advisor查询向量数据来获取与用户问题相关的文档，将这些文档作为上下文添加到用户查询中

进阶用法，建造者模式来配置更精细的参数

进阶用法2，动态过滤表达式

进阶用法3，自定义提示词模板，自主控制将检索的文档与用户查询结合

RetrievalAugmentationAdvisor

在原先基础上提供了更灵活和定制选项，在该案例中：

  • 通过构造者模式在检索增强器中，指定文档检索器VectorStoreDocumentRetriever，来从向量数据库检索文档
  • 将该Advvisor添加到ChatClient请求中，即可生效RAG

进阶用法，可以在RAG中配合查询转换器

  • 构造者模式的内部通过构造者又构建了一个RewriteQueryTransformer，实现在检索前重写原始查询，提高检索质量（解决用户问题含糊的问题）
空上下文的处理

RetrievalAugmentationAdvisor默认不允许上下文为空，如果没有找到相关文档，会采取保守策略， 让模型不要回答用户的问题。

但是有些场景下，希望及时没有知识库支持，也能回答一些通用问题，所以可以通过同样的构造者方式配置ContextualQueryAugmenter上下文增强器

ContextualQueryAugmenter允许配置自定义提示模板，可以指导模型在没有找到模型时如何回应用户，如解释无法回答的原因，引导用户提问其他问题或提供更多信息。

RAG进阶实战和最佳实践

理论完毕，按照步骤实践。

文档收集与切割-ETL

首先完成ETL工作，这是RAG的基础也是最重要环节，文档质量直接决定AI能力的上限，其他策略只是为了逼近上限。

原始文档优化

文档质量的首要条件是保证知识完备性，如果缺失内容，就无法回答问题，需要通过收集用户反馈/统计知识库检索命中率，不断优化知识库内容。

  • “AI原生文档“：需要专门为AI知识库建立文档，可以按照以下的规则让大模型为我们的文档进行优化。

在知识完整的前提下，我们要注意 3 个方面：

  1. 内容结构化:
  • 原始文档应保持排版清晰、结构合理，如案例编号、项目概述、设计要点等
  • 文档的各级标题层次分明，各标题下的内容表达清晰
  • 列表中间的某一条之下尽量不要再分级，减少层级嵌套
  1. 内容规范化:
  • 语言统一：确保文档语言与用户提示词一致（比如英语场景采用英文文档），专业术语可进行多语言标注
  • 表述统一：同一概念应使用统一表达方式（比如 ML、Machine Learning 规范为“机器学习”），可通过大模型
    分段处理长文档辅助完成
  • 减少噪音：尽量避免水印、表格和图片等可能影响解析的元素
  1. 格式标准化:
  • 优先使用 Markdown、DOC/DOCX 等文本格式（PDF 解析效果可能不佳），可以通过百炼 DashScopeParse
    工具将 PDF 转为Markdown，再借助大模型整理格式
  • 如果文档包含图片，需链接化处理，确保回答中能正常展示文档中的插图，可以通过在文档中插入可公网访
    问的 URL 链接实现

文档切片

切片的尺寸根据实际要调整，原则：避免过短导致语义丢失，避免过长导致引入无关信息，尽量减少语义截断的问题。

具体可以根据以下要素：

  • 文档类型：专业类文献可以增加长度保留更多上下文信息，而社交类帖子，缩短长度可以更精确捕捉语义
  • 提示词复杂度：如果复杂且具体，可能需要增加切片长度，否则，较短的长度更合适

最佳切片策略：智能分块算法 （根据分句标识符划分段落–>根据语义动态选择切片点）+ 人工二次校验

实现：通过SpringAI的ETL pipeline的TokenTextSplitter（documentTransformer实现类）

使用切片器

问题：手动使用切分参数很难把握合适值，容易 出现语义截断的问题。

解决思路：云服务，如阿里云百炼实现好的智能切分 + 人工二次校验

元数据标注

为文档添加更丰富的元信息，形成多维索引，便于向量化处理和精确检索

实现方式：

  • 手动对单个文档添加，通过Document构造函数的Map传入即可
  • 批量添加，通过DocumentReader，可以对每篇文章添加特定标签，比如恋爱状态等
  • 自动添加，SpringAI的transformer组件提供AI自动解析关键词并添加元信息的功能
    • 阿里云百炼可以通过手动设置标签或API也可以实现快速过滤

向量转换与存储

这是影响准确性和效率的核心步骤

首先从成本、性能、数据规模等考虑存储方案，内存/MongoDB/Redis等，具体实现通过builder即可指定vectorStore

接下来考虑合适的嵌入模型embeddingModel，好的嵌入模型直接影响相似度计算和检索准确性

文档过滤和检索

这里就有很多优化的地方

多查询扩展

提示词不完整/歧义的情况下，通过多查询拓展器可以拓展检索范围，提高文档的召回率

注意：

  • 合适的查询数量（3-5个），过多成本会过高
  • 要保留原始查询的核心语义

扩展后，即可直接用于检索文档、或者提取文本改写提示词

多查询扩展的完整使用流程可以包括三个步骤：

  1. 使用扩展后的查询召回文档：遍历扩展后的查询列表，对每个查询使用 DocumentRetriever 来召回相关文档。

  2. 整合召回的文档：将每个查询召回的文档进行整合，形成一个包含所有相关信息的文档集合。(也可以使用文档合并器去重)

  3. 使用召回的文档改写 Prompt：将整合后的文档内容添加到原始 Prompt 中，为大语言模型提供更丰富的上下文信息。

💡 需要注意，多查询扩展会增加查询次数和计算成本，效果也不易量化评估，所以个人建议慎用这种优化方式。

查询重写/翻译

参照文档实现查询重写

  • RewriteQueryTransformer 重写优化查询结构
  • transformedQuery 多语言支持

应用，通过修改用户提示词为重写/翻译后的提示词即可，云服务中，可以通过多轮会话改写获得更专业的提示词

检索器配置

关键配置：相似度阈值（噪音和信息完整性的权衡）、返回文档数量（噪音和信息完整性的权衡）、过滤规则（FilterExpressionBuilder）

高级知识补充

混合检索

RAG 系统的检索质量很大程度决定最终回答

但不同检索方法各有优缺点: 向量检索虽然能理解语义, 捕捉文本间的概念关联, 但对关键词敏感度不够。比如, 当你搜索“2025 年怎么学编程”时, 向量检索可能会返回与编程相关的术语解释, 而不是准确锁定 2025 年编程学习路线。

相反, 基于倒排索引的全文检索在精确匹配关键词方面表现出色, 但它不理解语义, 难以处理同义词或概念性查询。就像你问“百度的创始人是谁”, 全文检索可能不会返回只提到“百度创始人创办了很多网站”而没有明确提到“百度”的文档。

结构化检索支持精确过滤和复杂条件组合, 但依赖良好的元数据。而知识图谱检索能发现实体间隐含关系, 适合回答复杂问题, 但构建成本高。

主要检索方法比较表:

检索方法 原理 优势 劣势
向量检索 基于嵌入向量相似度搜索 理解语义关联, 适合概念性查询 对关键词不敏感, 召回可能不准确
全文检索 基于倒排索引, 匹配关键词 精确匹配关键词, 高召回率 不理解语义, 同义词难以匹配
结构化检索 基于元数据或结构化字段查询 精确过滤, 支持复杂条件组合 依赖良好的元数据, 灵活性有限
知识图谱检索 利用实体间关系进行图遍历 发现隐含关系, 回答复杂问题 构建成本高, 需要专业知识

其中, 全文检索是后端开发要掌握的技能, 对应的主流技术实现是 Elasticsearch,

策略：

  • 并行检索，派出多个专家，整合他们的结果
  • 级联混合检索：先广泛召回，再精细过滤，如先向量获取相似文档，再关键词过滤，最后元数据进一步筛选，漏斗式逐步过滤
  • 动态混合检索，理解为cursor的auto模式，分析查询的类型动态路由到对应的检索方式，如问怎么写java项目，就会匹配到向量检索配合全文检索，最终合并多个模型的结果
    • dify就提供了基于全文+基于向量的语义检索的混合检索策略，可以设置不同权重

大模型幻觉问题

大型模型幻觉是指模型生成看似合理但实际上不准确或完全虚构的内容。就像一个信心十足的学生回答了一个自己并不真正了解的问题。这些幻觉主要有三种表现形式:

  1. 事实性幻觉:生成与事实不符的内容(如错误的日期、人物关系等)。比如“百度创始人发明了计算器”

  2. 逻辑性幻觉:推理过程存在逻辑错误，得出不合理的结论。比如“1+1=3”

  3. 自洽性幻觉:生成内容自身存在矛盾。比如“我很年轻，才80岁”

为什么会出现幻觉呢?原因其实很复杂。

  • 一方面，模型的训练数据中可能包含错误或过时的信息;
  • 另一方面，大语言模型本质上是预测下一个词的概率模型，它们倾向于生成流畅而未必准确的内容。更重要的是，模型并不真正“知道”什么，它只是学会了文本的统计模式。

想象一下，当你问一个从来没去过月球的人关于月球表面的情况，他可能会基于看过的电影或书籍给出看似合理但不准确的描述。大型模型的幻觉本质上与此类似。

那么，如何减少这种幻觉呢?

  • 首先就是我们重点学习的RAG,通过引入外部知识源，我们可以让模型不再完全依赖其参数中存储的信息，而是基于检索到的最新、准确的信息来回答问题。
    • 有效的RAG实现通常会引入“引用标注”机制，让模型明确指出信息来源于哪个文档的哪个部分。当模型不确定时，我们也应该鼓励它诚实地表达不确定性，而不是猜测答案。这就像一个好的学者会明确引用来源，并在不确定时坦诚承认知识的局限性。
  • 此外，还有其他减轻幻觉的方法，比如提示工程优化，可以采用“思维链”提高推理透明度，通过引导模型一步步思考，我们能够更好地观察其推理过程，及时发现可能的错误。很多 Agent 超级智能体都会采用这种模式（如cursor中的推理过程）

RAG应用评估

在小应用产品中，可以直接用云平台的命中测试能力评估RAG效果。但是大公司就会有一套专门的评估体系：

RAG 应用评估本质上回答了 3 个关键问题：

  • 系统检索的信息是否相关？
  • 生成的回答是否准确？
  • 整体用户体验如何？评估的目的是确保回答质量、识别性能瓶颈，从而给出持续优化的思路。

我们可以简单了解下 RAG 应用的评估指标：

  1. 检索质量评估指标
  • 召回率：能否检索到所有相关文档
  • 精确率：检索结果中相关文档的比例
  • 平均精度均值 (MAP)：考虑排序质量的综合指标
  • 规范化折扣累积增益 (NDCG)：考虑到文档的相关性和它们在排名中的位置，是一个衡量排名质量的指标
  1. 生成回答质量评估指标
  • 事实准确性：回答中事实性陈述的准确程度
  • 答案完整性：回答是否涵盖问题的所有方面
  • 上下文相关性：回答与问题的相关程度
  • 引用准确性：引用内容是否确实来自检索上下文

当然，我们还可以根据具体应用场景，定制专门的评估标准。比如系统性能评估、领域适应性评估、多语言评估、时效性评估和用户满意度评估。其中，用户满意度评估在我们开发 AI 产品时尤为常见，经常需要引导用户针对 AI 大型模型的回复进行打分。

RAG 评估流程通常包括 4 个步骤：

  1. 生成评估数据集：创建覆盖不同问题类型的测试集，为每个问题准备标准答案和相关文档。这些测试问题应包括事实性问题、观点性问题、多步骤推理问题等各种类型。

  2. 运行评估检索过程的程序：对每个测试问题执行检索，与人工标注的相关文档比较，计算检索性能指标。

  3. 评估回答质量：实际操作中，评估通常分为自动评估和人工评估两种方式。自动评估使用像 ROUGE (召回率取向摘要评估) 或 BLEU (双语评估替补) 这样的指标来衡量生成内容与参考答案的相似度，或者使用更强大的模型来判断回答质量。但自动评估有其局限性，某些方面如创造性、实用性等仍然需要人工评估。这就是为什么很多 AI 公司会招人来人工标注。

  4. 综合分析与优化：识别失败模式和常见错误，比如区分检索失败和生成失败，针对性改进系统组件。

如果面试时，面试官问到“你是如何评估和调优 RAG 系统的？”，就可以采用下面这样的回答：

我曾参与过一个编程咨询 RAG 系统的评估和优化。系统在回答具体编程技术问题时表现出色，但处理“根据个人编程情况给出学习建议”的复杂案例时表现不好。通过错误分析，我们发现问题出在检索阶段——系统无法同时检索到相关技术知识和类似的学习建议。针对这一问题，我们调整了检索策略，专门为学习建议类问题设计了基于案例的检索方法，从而提升了模型回复的准确度。

高级RAG架构

注：实际应用会结合使用多个，如回答公司财务信息会用C-RAG，复杂的投资策略分析使用RAPTOR多维度分析

最终的挑战：精准检索知识，无缝融入生成过程，给用户生成准确又自然的结果

未来方向：多模态、适应性（根据用户动态调整）、高效率

C-RAG 自纠错RAG

在回答过程中，采用“检索-生成-验证-纠正”的闭环流程，发现错误就立即重新生成，保证回答的准确性，适合医疗、法律等事实要求准确性高场景。

self-RAG 自省式RAG

解决的问题是：并非所有问题都要检索，让回答更自然提高检索效率。

如1+1等于几可以直接回答，不需要额外检索，让他会思考这个问题是否需要引入外部知识

RAPTOR 检索树RAG

场景：可拆分的复杂问题，如数学题，大问题拆分多个小问题

如介绍小红书的点赞模块和评论模块多个问题，RAPTOR会分别检索这几个模块的信息，综合多个信息整合给出最终答案，提高长篇叙述的连贯性和准确性，并可以克服单次检索的上下文长度的限制。

多智能体RAG系统

多个大模型各自处理自身擅长的领域，各司其职，实现复杂任务协同处理
