Prompt工程

Prompt工程：通过精心设计和优化提示词，引导AI模型的高质量输出

Prompt分类

基于角色

  • User Prompt：实际的指令
  • System Prompt：模型行为规则，角色定位隐藏指令
  • Assisant Prompt：响应内容（多轮对话中之前回复的内容）

基于功能

  • Instructional Prompts：指令型
  • Conversational Prompts：对话型
  • Creative Prompts：创意型，故事诗歌等
  • Role-playing Prompts：角色扮演
  • Few-Shot Prompts：少样本学习（提供一些示例）

基于复杂度

  • Simple Prompts：简单提示词
  • Compound Prompts：复杂提示词（多个指令）
  • Chain Prompts：链式提示词（第一步，第二步…）
  • Template Prompts：模板提示词（可替换变量）

提示词优化

Token： 100个单词大概75150token，100个中文大概100200token，成本=输入token * 输入单价 + 输出token * 输出单价 ，实际需要对比各家的价格

  • openAI官方token计算
  • 非官方token计算

系统提示词、用户提示词、输出内容都算做token，需要从这些角度优化token

  • 提示词精简
    • 向量检索代替直接输入
    • 结构化代替自然语言（如表格、列表等结构化代替长段的文本描述）
  • 定期清理上下文

提示词优化思路：

  • 迭代式提示词优化：多轮对话改进
  • 边界测试：给出极限的情况来优化
  • 提示词模板化：统一输出形式便于调试

执行原理

Advisors执行原理（Prompt->AdvisedRequest->【Prompt->ChatResponse】->AdvisedResponse->ChatResponse，AdvisorContext可传递信息）：

  1. Spring AI 框架从用户的 Prompt 创建一个 AdvisedRequest，同时创建一个空的 AdvisorContext 对象，用于传递信息。
  2. 链中的每个 advisor 处理这个请求，可能会对其进行修改。或者，它也可以选择不调用下一个实体来阻止请求继续传递，这时该 advisor 负责填充响应内容。
  3. 由框架提供的最终 advisor 将请求发送给聊天模型 ChatModel。
  4. 聊天模型的响应随后通过 advisor 链传回，并被转换为 AdvisedResponse。后者包含了共享的 AdvisorContext 实例。
  5. 每个 advisor 都可以处理或修改这个响应。
  6. 最终的 AdvisedResponse 通过提取 ChatCompletion 返回给客户端。

拦截器链是有顺序的（责任链设计模式），通过getOrder可得到顺序，值越低越优先

  • 如上述案例先MessageMemory添加历史记录到提示词中，再将所有的内容进行知识库检索。

拦截器支持流式和非流式返回值，如果要自主实现拦截逻辑，逻辑一样但返回值不同，最好都实现一下

ChatMemoryAdvisor

ChatMemoryAdvisor（三种存储方式，提示词/系统提示词/向量数据库）

  • MessageChatMemoryAdvisor: 从记忆中检索历史对话, 并将其作为消息集合添加到提示词中（建议使用，保证上下文连贯性）
  • PromptChatMemoryAdvisor: 从记忆中检索历史对话, 并将其添加到提示词的系统文本中（没有边界信息）
  • VectorStoreChatMemoryAdvisor: 可以用向量数据库来存储检索历史对话

ChatMemoryAdvisor依赖于ChatMemory构建历史对话，提供增删查的方式，内置存储方式：

• InMemoryChatMemory：内存存储
• CassandraChatMemory：在 Cassandra 中带有过期时间的持久化存储
• Neo4jChatMemory：在 Neo4j 中没有过期时间限制的持久化存储
• JdbcChatMemory：在 JDBC 中没有过期时间限制的持久化存储

结构化输出

Structured Output Converter可以将大模型输出内容转换成JSON等结构化格式

  • 调用前：转换器在提示词后面添加格式指令，引导模型生成特定格式响应
  • 调用后：转换器将模型输出转换成结构化类型实例（JSON XML等数据结构）

注：结构化输出转换器仅能尽最大努力输出，AI不一定保证可以按要求输出，建议需要有验证机制或异常处理保证模型输出符合预期

最佳实践

  1. 尽量为模型提供清晰的格式指导

  2. 实现输出验证机制和异常处理逻辑，确保结构化数据符合预期

  3. 选择支持结构化输出的合适模型

  4. 对于复杂数据结构，考虑使用 ParameterizedTypeReference

对话记忆持久化

我们有时需要将对话保存到文件、数据库、redis中

在对话记忆的官方文档中，可以发现有JdbcChatMemoryRepository可以持久化到数据库中，但官方对这个介绍比较少

我们还是建议通过自定义ChatMemory实现（参考已实现的InMemoryChatMemory设计），也可以更灵活地设计

ChatMemory中只有增删查的方法，而在InMemoryChatMemory中，通过ConcurrentHashMap（key是会话id，value是id的消息列表）维护对话信息，我们参考该思路实现。

难点：消息和文本之间如何转换。消息持久化时，要将消息从Message对象转换成文本，而从持久化层读取消息时，需要将文本转换成Message对象。

思路是通过JSON反序列化，但问题是：

  • 持久化的Message是一个接口，对应多个不同的子类实现（UserMessage SystemMessage等）
    • 而每个子类的结构不统一，并且子类无法保证序列化，无参构造函数等

一种解决思路是通过Kyro序列化库来实现转换，通过我们写好的转换后的FileBasedChatMemory对增删相应的方法增强后就可以了。

PromptTemplate模板

官方文档

这是构建提示词的核心组件之一，通过动态替换占位符来生成一些固定但局部有变动的提示词（数据库预编译、记录日志的变量占位符、模板引擎等场景）

PromptTemplate 在以下场景特别有用:

  1. 动态个性化交互: 根据用户信息、上下文或业务规则定制提示词
  2. 多语言支持: 使用相同的变量但不同的模板文件支持多种语言
  3. A/B测试: 轻松切换不同版本的提示词进行效果对比
  4. 提示词版本管理: 将提示词外部化, 便于版本控制和迭代优化

内部原理说明：

SpringAI中，实现了PromptTemplateActions、PromptTemplateMessageActions接口，这些接口提供了一些文本生成的方法

SpringAI提供了几种专用的模板类，对应不同角色的消息:

  1. SystemPromptTemplate: 用于系统消息, 设置 AI 的行为和背景
  2. AssistantPromptTemplate: 用于助手消息, 用于设置 AI 回复的结构
  3. FunctionPromptTemplate: 目前没用

比如系统消息模板可以快速构造一个Prompt，或者设置回复结构

另外，可以从文件中加载模板便于管理（多场景支持，在修改提示词模板时可以不修改代码，开闭原则，推荐）

17.Mybatis缺点
Mybtis 的缺点有以下几点： 1）半ORM：MyBatis 不会像全自动 ORM 框架（如 Hibernate）那样自动生成 SQL 语句， 对于一些简单的查询修改操作，也需要自己写 SQL，增加了工作量。 2）大量XML维护：对大型项目来说，可能需要大量的 xml 文件，会导致配置的复杂性和维护的困难。 3）SQL无法自动生成：因为 MyBatis 缺乏 SQL 自动能力，因此维护成本会高一些，例如修改了表结构或字段名或迁移了数据库，都需要手动修改相应的 SQL，还容易漏改。
