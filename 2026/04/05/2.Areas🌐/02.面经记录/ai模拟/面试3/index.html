分模块复习梳理

模块1：Java集合

当前掌握进度

能清晰区分ArrayList/LinkedList的核心差异、HashMap的底层结构、ConcurrentHashMap的版本区别，基础方向完全正确，无原则性错误，是5个模块中掌握最扎实的模块。

必须重点记忆的核心考点（面试必背）

  1. JDK1.8 HashMap put方法完整执行流程（按顺序）：数组空校验&初始化 → 哈希计算&桶位寻址 → 空桶CAS插入 → 哈希冲突处理（链表尾插/红黑树）→ key重复覆盖 → 树化条件判断 → 扩容触发判断
  2. ArrayList扩容机制：无参构造初始容量为0，首次add元素初始化为10；元素个数超过数组长度时触发扩容，扩容后容量为原容量的1.5倍
  3. LinkedList插入O(1)的前提：仅持有目标节点的引用（如头尾节点）时成立；指定下标插入需先遍历链表定位节点，时间复杂度为O(n)
  4. ConcurrentHashMap JDK1.7 vs 1.8核心区别：1.7采用分段锁（Segment继承ReentrantLock），并发度固定；1.8放弃分段锁，采用桶级synchronized+CAS，锁粒度更细，新增红黑树解决长链表性能问题

待补全短板

  • HashMap扩容的完整流程、红黑树树化&退化的完整条件
  • ConcurrentHashMap多线程协助扩容的底层原理
  • 高频集合类的线程安全问题与适用场景

模块2：MySQL

当前掌握进度

能说出MVCC的核心依赖、最左前缀原则、慢查询的基础排查方法，核心方向正确，但存在MVCC可见性规则完全说反、索引命中判断错误的致命细节问题。

必须重点记忆的核心考点（面试必背）

  1. MVCC核心定义：多版本并发控制，仅在READ COMMITTED(RC)和REPEATABLE READ(RR，MySQL默认)两个隔离级别下生效，核心作用是实现读写不加锁、读写不冲突，提升数据库并发性能
  2. MVCC两大核心依赖：undo log版本链（存储行记录的历史版本）、Read View可见性判断规则
  3. Read View可见性判断规则（按优先级排序）：
    • 行记录的DB_TRX_ID == 当前事务ID → 完全可见（当前事务自己修改的记录）
    • 行记录的DB_TRX_ID < min_trx_id → 完全可见（修改记录的事务已提交）
    • 行记录的DB_TRX_ID >= max_trx_id → 完全不可见（修改记录的事务在当前Read View生成后才开启）
    • 中间值判断：DB_TRX_ID在min_trx_id和max_trx_id之间，在活跃事务列表m_ids中则不可见，不在则可见
  4. RC与RR隔离级别MVCC的核心区别：RC每次执行SELECT都会生成全新的Read View，RR仅第一次执行SELECT时生成Read View，后续复用
  5. 最左前缀原则核心规则：匹配联合索引的最左N个字段，遇到范围查询（>、<、between）会停止匹配，in属于等值查询，不会中断索引匹配
  6. 慢查询完整排查流程：定位慢SQL（慢查询日志/show processlist）→ explain分析执行计划 → 针对性优化 → 上线验证优化效果

待补全短板

  • InnoDB聚簇索引&二级索引的回表完整流程、覆盖索引的适用场景
  • 事务隔离级别与脏读/不可重复读/幻读的对应关系
  • explain执行计划关键字段的完整解读
  • 索引失效的高频场景与根本原因

模块3：Redis

当前掌握进度

能说出分布式锁的核心特性、缓存一致性的基础方案、RDB&AOF的核心区别，核心方向正确，但存在缓存一致性流程说反、分布式锁基础实现错误的问题。

必须重点记忆的核心考点（面试必背）

  1. 分布式锁正确基础实现：必须用SET 锁名 线程唯一标识 NX EX 锁超时时间这条原子命令，禁止单独用setnx+expire两条命令（非原子，会导致死锁）
  2. 分布式锁四大核心特性的实现：
    • 防误释放：用Lua脚本实现「判断锁的标识与当前线程一致再删除」的原子操作
    • 可重入：用Hash结构存储锁的持有者&重入次数，通过Lua脚本实现原子加解锁
    • 锁超时续期：Redisson看门狗机制（默认锁超时30秒，每隔10秒续期到30秒，仅不手动指定超时时间时生效）
    • 主从切换锁失效：高安全场景采用RedLock红锁方案
  3. 旁路缓存（Cache Aside）标准流程（线上最常用）：
    • 读请求：先查缓存 → 命中直接返回 → 未命中查数据库 → 查到数据回填缓存 → 返回结果
    • 写请求：先更新数据库 → 更新成功后，删除缓存（禁止更新缓存）
  4. 延迟双删标准流程：写请求 → 先删除缓存 → 更新数据库 → 延迟几百毫秒（大于读请求平均耗时）→ 再次删除缓存
  5. RDB&AOF核心区别：RDB是某一时刻的全量二进制快照，恢复速度快、数据安全性低；AOF是持续追加的写命令日志，恢复速度慢、数据安全性高；线上环境默认开启RDB-AOF混合持久化

待补全短板

  • Redis单线程高性能的完整核心原理
  • 缓存穿透/击穿/雪崩的完整触发场景与解决方案
  • Redis主从同步&哨兵集群的核心原理
  • Redis核心数据结构的底层实现与适用场景

模块4：Java并发编程

当前掌握进度

能说出JMM三大特性、volatile的核心作用、synchronized锁升级流程、AQS的核心组件，核心方向正确，是本次面试进步最大的模块，但存在内存屏障规则、锁升级触发条件、AQS流程的细节错误与遗漏。

必须重点记忆的核心考点（面试必背）

  1. JMM核心定义：Java虚拟机规范定义的多线程环境下的内存访问规则，抽象了「主内存+线程工作内存」的模型，解决CPU多级缓存、指令重排序导致的共享变量数据不一致问题
  2. JMM三大核心特性：原子性、可见性、有序性
  3. volatile关键字的核心作用：保证可见性+有序性，不保证原子性
  4. volatile内存屏障规则：
    • 写操作前：加入StoreStore屏障，禁止前面的普通写与当前volatile写重排序
    • 写操作后：加入StoreLoad屏障（核心），禁止当前volatile写与后面的volatile读/写重排序，同时强制刷新回主内存保证可见性
    • 读操作前：加入LoadLoad屏障，禁止后面的普通读与当前volatile读重排序
    • 读操作后：加入LoadStore屏障，禁止当前volatile读与后面的普通写重排序，同时强制从主内存加载最新值
  5. synchronized锁升级完整流程（单向不可逆）：无锁 → 偏向锁（单线程无竞争场景）→ 轻量级锁（多线程交替竞争，自适应自旋）→ 重量级锁（多线程同时竞争，内核态mutex互斥量）
  6. AQS两大核心组件：volatile修饰的state同步状态变量、基于CLH改进的双向同步队列
  7. AQS两种核心模式：独占模式（同一时间仅一个线程持有锁，如ReentrantLock）、共享模式（同一时间多个线程持有锁，如Semaphore、CountDownLatch）
  8. 基于AQS实现的JUC工具类：ReentrantLock、ReentrantReadWriteLock、Semaphore、CountDownLatch、CyclicBarrier

待补全短板

  • synchronized底层ObjectMonitor的队列抢锁完整流程
  • AQS独占&共享模式的完整执行流程
  • ThreadLocal的核心原理与内存泄漏问题
  • JUC原子类的CAS实现原理与ABA问题

必须重点记忆的核心考点（面试必背）

  1. Bean完整生命周期11步标准流程（按执行顺序）
  2. BeanFactoryPostProcessor与BeanPostProcessor的核心区别：前者在Bean实例化前执行，修改Bean的定义元数据BeanDefinition；后者在Bean实例化后执行，修改已经生成的Bean实例对象
  3. 三级缓存的标准名称与核心作用：
    • 一级缓存singletonObjects：存储完全初始化完成、可直接使用的单例Bean
    • 二级缓存earlySingletonObjects：存储已实例化、但未完成属性注入和初始化的早期Bean对象
    • 三级缓存singletonFactories：存储Bean对应的ObjectFactory对象工厂，用于延迟生成早期暴露对象
  4. 必须用三级缓存的核心原因：延迟生成AOP代理对象，保证代理对象在Bean初始化后生成的设计原则，避免提前生成代理对象，同时保证代理对象的单例性

【Spring核心模块 面试题完整标准答案（直接背诵）】

第一题：Spring Bean完整生命周期，BeanFactoryPostProcessor与BeanPostProcessor的生效时机与作用

完整标准答案

  1. Spring Bean从容器启动到销毁的完整生命周期（11步标准流程，按执行顺序）

    1. 容器启动，加载BeanDefinition：Spring扫描配置类/XML文件，解析Bean的定义信息（类名、scope、依赖关系等），封装成BeanDefinition，注册到BeanDefinitionRegistry中；
    2. 执行BeanFactoryPostProcessor：在Bean实例化之前，执行所有注册的BeanFactoryPostProcessor实现类，修改BeanDefinition的元数据，此时还未生成任何Bean实例；
    3. 实例化Bean：通过反射调用Bean的构造方法，创建Bean的原始实例对象；
    4. 属性填充（依赖注入）：Spring按照@Autowired、@Resource等注解，完成Bean的依赖注入；
    5. 执行Aware接口回调：如果Bean实现了Aware系列接口，按顺序执行BeanNameAware、BeanFactoryAware、ApplicationContextAware的回调方法；
    6. 执行BeanPostProcessor前置处理：调用postProcessBeforeInitialization方法，在初始化方法执行前对Bean实例进行修改；
    7. 执行初始化方法：按顺序执行@PostConstruct注解修饰的方法 → InitializingBean接口的afterPropertiesSet方法 → 自定义init-method方法；
    8. 执行BeanPostProcessor后置处理：调用postProcessAfterInitialization方法，在初始化方法执行后对Bean实例进行最终修改，Spring AOP的动态代理对象就是在这里生成的；
    9. Bean正常使用：Bean完全初始化完成，放入一级缓存单例池，可正常被容器调用；
    10. 容器关闭，执行销毁前回调：执行@PreDestroy注解修饰的自定义销毁方法；
    11. 执行最终销毁方法：执行DisposableBean接口的destroy方法 → 自定义destroy-method方法，完成Bean销毁。

  2. 两个处理器的核心区别

    处理器 生效时机 核心作用
    BeanFactoryPostProcessor Bean实例化之前，BeanDefinition加载完成后 修改Bean的定义元数据BeanDefinition
    BeanPostProcessor Bean实例化之后，初始化方法执行的前后 修改已经生成的Bean实例对象

第二题：Spring循环依赖为什么必须用三级缓存？只用二级缓存行不行？

完整标准答案

  1. 核心前提：Spring的三级缓存仅能解决单例Bean的setter注入循环依赖，无法解决构造器注入、原型Bean的循环依赖。

  2. 三级缓存的标准名称与作用

    缓存层级 标准名称 存储内容 核心作用
    一级缓存 singletonObjects 完全初始化完成、可直接使用的单例Bean 最终的单例池，所有正常使用的Bean都在这里
    二级缓存 earlySingletonObjects 已实例化、但未完成属性注入和初始化的早期Bean对象 避免同一个Bean在循环依赖中多次生成代理对象，保证单例性
    三级缓存 singletonFactories Bean对应的ObjectFactory对象工厂 延迟生成Bean的早期暴露对象，只有出现循环依赖时，才会调用getObject()方法生成早期对象（有AOP则生成代理对象，无AOP则返回原始对象）

  3. 为什么必须用三级缓存？只用二级缓存行不行？
    结论：只用二级缓存不行，违背Spring的设计原则，且会破坏单例Bean的代理对象单例性，核心原因：

    1. 违背Spring AOP的生命周期设计原则：Spring的设计规范是，AOP代理对象必须在Bean生命周期的初始化后才生成，正常无循环依赖的Bean，在初始化完成前绝对不会提前生成代理对象。如果只用二级缓存，必须在Bean实例化后就提前生成代理对象放入缓存，无论有没有循环依赖，完全违背了Spring的设计原则。
    2. 无法保证代理对象的单例性：如果不用三级缓存，只用二级缓存，当一个Bean被多个循环依赖的Bean依赖时，会多次调用AOP代理生成逻辑，生成多个不同的代理对象，破坏了单例Bean的单例性。而三级缓存+二级缓存的组合，保证同一个Bean只会生成一次早期代理对象，放入二级缓存，后续所有依赖都从二级缓存获取，保证单例性。

  4. 循环依赖完整处理流程（A依赖B，B依赖A）

    1. A完成实例化后，将自己的ObjectFactory对象工厂放入三级缓存；
    2. A填充属性时发现依赖B，转而实例化B；
    3. B完成实例化，将自己的ObjectFactory放入三级缓存；
    4. B填充属性时发现依赖A，按「一级缓存→二级缓存→三级缓存」的顺序查找A；
    5. 一级、二级缓存无A，B从三级缓存拿到A的ObjectFactory，调用getObject()生成A的早期对象，放入二级缓存，删除三级缓存中A的工厂；
    6. B拿到A的早期引用，完成自身的属性填充、初始化，最终放入一级缓存；
    7. A拿到完成初始化的B，继续完成自身的属性填充、初始化，最终放入一级缓存，删除二级缓存中的A。# 第三次Java后端暑期实习模拟面试 完整专业总结

1. Java集合模块

问题类型 具体问题 需补充的标准内容
高频考点遗漏 完全遗漏HashMap树化阈值8的底层依据、树化的第二个必要条件 1. 阈值8的底层依据：理想哈希算法下，链表节点数符合泊松分布，节点数达到8的概率不足千万分之一，仅在哈希严重失效时触发树化；
2. 树化的两个必要条件：链表长度≥8 且 数组容量≥64，数组容量不足时仅扩容，不树化。
术语口误/逻辑颠倒 位运算与取模运算的表述完全说反，核心逻辑表述模糊 1. 核心前提：只有数组长度是2的整数次幂，(n-1)&hash位运算才与hash%n取模运算结果完全等价；
2. 性能差异：位运算是CPU原生指令，执行周期远短于取模运算的除法/余数计算，因此性能更高；
3. 2倍扩容的位置判断：仅需判断hash值的新增高位是0/1，即可O(1)确定新位置，无需重新计算哈希。

2. 安全性：synchronized由JVM自动释放锁，不会出现锁泄漏；ReentrantLock需手动释放，易因代码疏漏导致死锁。

2. MySQL模块

标准术语错误 用“范围锁”替代InnoDB标准术语“临键锁（Next-Key Lock）” 临键锁是「记录锁（Record Lock）+ 间隙锁（Gap Lock）」的组合，锁定左开右闭的区间，是InnoDB RR级别解决幻读的核心机制。

1. 将被调用方法抽到独立Bean中（最佳实践）；
2. 开启代理暴露，通过AopContext.currentProxy()获取代理对象调用；
3. 自注入（自己注入自己的代理对象）；
4. 编程式事务手动管理。
根本原因错误 事务失效的根本原因补充错误 ThreadLocal是事务的底层实现，不是事务失效的根本原因，根本原因是：同类内部调用使用的是this原始对象，而非Spring生成的代理对象，事务切面无法被触发。

第三题：4种场景@Transactional是否失效？说明根本原因

完整标准答案

  1. ① 方法被private修饰，方法上加了@Transactional

    • 结论：事务完全失效
    • 根本原因：Spring事务的底层是AOP动态代理，无论是JDK动态代理还是CGLIB动态代理，都无法重写private修饰的方法，事务切面无法拦截到private方法的执行，注解完全不生效。补充：final、static修饰的方法，事务同样会失效。

  2. ② 方法内用try-catch捕获了异常，仅打印了日志，没有向上抛出异常

    • 结论：事务完全失效，不会触发回滚
    • 根本原因：@Transactional默认的回滚规则是：只有当方法抛出未被捕获的RuntimeException或Error时，才会触发事务回滚。用try-catch把异常捕获后，仅打印日志未重新抛出，事务切面完全感知不到异常，自然不会触发回滚。

  3. ③ 同类中，没有加事务注解的方法A，内部调用了加了@Transactional的方法B

    • 结论：方法B的事务完全失效
    • 根本原因：Spring事务基于AOP动态代理实现，只有通过Spring生成的代理对象调用被@Transactional修饰的方法，事务切面才会被触发。同类内部调用时，用的是this.方法B()，this是当前Bean的原始对象，不是Spring生成的代理对象，事务切面无法拦截方法B的执行，注解完全失效。

  4. ④ 方法的事务传播行为设置为Propagation.NOT_SUPPORTED

    • 结论：事务完全失效
    • 根本原因：Spring事务传播行为Propagation.NOT_SUPPORTED的定义是「以非事务方式运行，如果当前存在事务，就把当前事务挂起」。无论外部有没有事务，这个方法都不会运行在事务中，@Transactional注解完全失效。
