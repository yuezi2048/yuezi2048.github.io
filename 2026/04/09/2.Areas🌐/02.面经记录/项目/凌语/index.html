对话记忆

这个会话记忆是基于chatmemory和Advisor的特性来实现的。

  • 实际上，我是通过API的chatClient集成，Advisors，你可以把它理解为是可插拔的一串拦截器，在调用AI前后我们可以进行额外操作
  • 那么对话记忆就是在调用后的多轮对话上下文，使用MessageChatMemoryAdvisor来实现多轮对话，对话历史能作为消息集合添加到之前的集合中，能记住之前的交流
    • MessageChatMemoryAdvisor 依赖于 ChatMemory 接口的实现来存取对话历史。ChatMemory 接口中定义了保存消息、查询消息和清空历史的方法。
    • Spring AI 提供了多种持久化方案，例如 JdbcChatMemory 可以将对话保存在关系型数据库中。但是在本项目中，考虑到 spring-ai-starter-model-chat-memory-jdbc 依赖版本较少且缺乏相关介绍，我选择了自定义实现 ChatMemory 接口的方式：
      • . 开发一个 FileBasedChatMemory 类，它实现了 ChatMemory 接口。
      • 使用高性能的 Kryo 序列化库将对话消息（Message 对象及其子类）序列化后保存到本地文件中，读取时再进行反序列化。选择 Kryo 是因为 Message 接口有多种实现，结构不一，且没有无参构造和 Serializable 接口，普通的 JSON 序列化难以处理。

具体来说，一个会话ID对应序列化到一个文件中，通过Kyro序列化 Message对象存取。而考虑到本身效率慢，所以我参考了Redis的全量备份+增量备份的思路，每天跑一个定时任务，将Redis中的序列化内容再写入到磁盘，并将Redis作为兜底服务宕机后，数据丢失的问题，启动后则自动进行全量+增量的恢复
- 首先构造时，确定基础路径
- 但是由于Message有很多子类(ChatMemory 接口就是add Message/List<Message> get clear)，像UserMessage SystemMessage，并且是否有序列化，构造函数等，所以就不考虑使用JSON的序列化了
- add时，根据对话id创建新的消息列表，而如果是list add，则是将更新后的信息反序列化到文件
- get则是读取会话文件，反序列化列表，然后读取MAX(Last_N，当前总数)，clear就是删除对应的会话

提问：结构化输出和 Function Calling 有什么区别？

回答：结构化输出是让模型返回指定格式的数据，本质还是”回答问题”。Function Calling 是让模型决定”要不要调用某个函数、传什么参数”，是一种决策行为。结构化输出的结果是最终答案，Function Calling 的结果是中间步骤，拿到参数后还得真去调用函数、把结果再喂给模型。

提问：结构化输出的 Token 消耗会不会比普通输出高很多？

回答：会高一些，因为 Prompt 里多了一段 JSON Schema 描述，输出也是 JSON 格式比自然语言更啰嗦。但通常不会高太多，几十到几百 Token 的增量。如果 Schema 特别复杂、字段特别多，可以考虑只描述关键字段，其他的让模型自由发挥。

知识库构建

两个方案，云端保证+本地兜底
1）方案 1：Spring AI + 本地知识库

这种方式更侧重于 通过编程 完整地实现 RAG 的核心流程。

  • 文档准备与读取：准备好知识文档，放到类路径下，我通过MarkdownDocumentReader读取，并通过MarkdownDocumentReaderConfig配置读取方式，然后添加一些元数据，便于检索
  • 向量转换和存储：读取到的文档列表会通过一个 EmbeddingModel（本项目中使用了 dashscopeEmbeddingModel）转换为向量。为了简化开发和学习，我在初期使用了 Spring AI 内置的、基于内存的 SimpleVectorStore 来存储这些文档向量。SimpleVectorStore 实现了 VectorStore 接口，该接口继承自 DocumentWriter，可以直接接收并处理文档列表的写入请求，内部完成向量化和存储过程。
  • 查询增强和关联：当用户提问时，我通过 Spring AI 的 QuestionAnswerAdvisor 实现知识库的 “无感知” 调用。这个 Advisor 会查询 SimpleVectorStore，找出与用户问题相关的文档片段，并将这些检索到的内容作为上下文附加到用户的原始提问中，形成一个增强的 Prompt，最后发送给大语言模型生成回答。

2）方案 2：Spring AI + 云知识库服务 (以阿里云百炼为例)

这种方式则更多地依赖云平台提供的能力来简化 RAG 的构建。

  • 云端知识库准备：我首先在阿里云百炼平台上传原始的知识文档。平台负责文档的解析、智能切分（将长文档分割成合适的片段）以及向量化，并将处理后的数据存储在向量数据库中。我们可以在百炼控制台手动管理这些知识库和文档切片。
  • 程序对接与查询增强：Spring AI Alibaba 生态提供了与百炼平台集成的 DashScopeDocumentRetriever。我通过配置这个DocumentRetriever（主要提供 API Key 和百炼上知识库的名称），实现从云端的知识库中检索信息。为了将云端检索能力集成到 AI 对话流程中，我使用了 RetrievalAugmentationAdvisor，这是一个更灵活的Advisor，我可以通过它指定使用自定义的 DocumentRetriever。当用户提问时，这个 Advisor 会通过 DashScopeDocumentRetriever 从百炼知识库中获取相关内容，增强用户提问，然后交由大模型处理。

RAG流程：

  • ETL（抽取转换 加载）
    • 文档收集与切割：文档-预处理（去除干扰）-切片，10页的pdf大概50-100块，大概一块有200-500token
      • 使用 DocumentReader 从数据源加载原始文档。我准备了 Markdown 格式的恋爱知识文档，然后引入了 spring-ai-markdown-document-reader 依赖，并创建了 LoveAppDocumentLoader 类加载读取文档
      • 使用 DocumentTransformer 对加载的文档进行处理，我使用了两种方式进行了文档的拆分，包括 MarkdownDocumentReader自动拆分 + TokenTextSplitter 拆分，还使用了 KeywordMetadataEnricher 自动为文档补充关键词元信息。
    • 向量转换与存储：Embedding模型转换成高维向量，像我用的国产的 BGE-M3 支持最多 8192 维。将向量联通原始文本和元数据塞入到向量数据库中，Milvus Pipecone
      • 使用 DocumentWriter 将转换后的文档写入存储库。在项目初期，我使用了 Spring AI 内置的、基于内存的 SimpleVectorStore 作为向量数据库。SimpleVectorStore 自身实现了 VectorStore 接口，而 VectorStore 接口又继承了 DocumentWriter，所以它可以直接接收 Document 列表进行存储。后来我还实践了把文档加载到基于 PostgreSQL 的 PGVector 向量数据库中。
        • 在配置类中，通过 PgVectorStore.builder() 方法创建 VectorStore 实例。这个Builder 需要传入一个 JdbcTemplate 和一个明确指定的 EmbeddingModel。
  1. 配置完成后，这个自定义的 pgVectorVectorStore Bean 就可以像其他 VectorStore 实现一样被注入到服务中，用来添加文档和执行相似性搜索。
  • 文档过滤与检索：请求来，转成Embeddind后在向量数据库相似度检索Top-k，然后也可以根据元数据、时间段，或指定来源的数据条件限制
  • 查询增强与关联：将检索的文档和原始问题拼接后，增强的Prompt给大模型，这样就不依赖外部数据，不容易有幻觉，并且最后也能标注来源，知道是从哪里出来的

Embeding模型用什么？
看成本和效果的权衡。选型要考虑几个维度：维度数、上下文长度、多语言支持、检索精度。OpenAI 的 text-embedding-3-large 精度高但贵，3-small 性价比不错。开源的 BGE 系列在中文场景表现好，Sentence-Transformers 的 all-MiniLM-L6-v2 轻量级，适合本地部署。
还有个关键点，索引阶段和查询阶段必须用同一个模型。用 A 模型生成文档向量，用 B 模型生成查询向量，两边的向量空间对不上，检索效果直接崩盘。

向量数据库用什么？
1）文档量小于 100 万条，Chroma 够用，内嵌 SQLite，部署简单，适合原型验证
2）百万到千万级，Milvus 或 Qdrant，分布式架构，支持水平扩展
3）追求托管省心，Pinecone 按量付费，不用自己运维
4）已经有 PostgreSQL 基础设施，pgvector 插件能直接用，不用额外引入新组件

为什么要设计批处理优化策略？它解决了什么问题？
因为当需要处理和嵌入大量文档时，如果一次性将所有文档或非常大的文档集合提交给嵌入模型进行处理，可能会引发以下几个主要问题：

  1. 嵌入模型的上下文窗口限制：大多数嵌入模型都有一个最大输入Token数量的限制，即 “上下文窗口大小”。如果单次请求中包含的文本量超出了这个限制，嵌入模型的 API 调用通常会直接失败或报错。而批处理策略通过将大量文档分割成较小的、Token 数量在模型限制内的批次，保证每次提交给嵌入模型的文本量都在处理能力范围内。
  2. 性能和效率：处理非常大的单体请求有可能会给嵌入服务或网络传输带来压力，导致处理时间过长或不稳定。将任务分解为多个小批次，至少可以让单个请求的处理时间更可控，提高整体的处理吞吐量和稳定性。
  3. 错误处理和资源管理：如果一个包含数千个文档的大请求在处理过程中失败，可能需要重试整个大请求，浪费时间和资源。将文档分成小批次处理，如果某个批次失败，只需要重试该小批次就好了。而且，小批次也更易于管理内存等客户端资源。
    我之前实验室的发票管理系统中也开发过数据导入导出的功能，也是将数据分批进行处理来优化在大数据量下的性能和稳定性，所以我觉得 Spring AI 提供自定义批处理优化策略这点还是挺妙的。

多查询扩展知道吗？
是RAG预检索的优化，对用户查询本身会有多个版本变体，在用户输入模糊或者知识库样式多样的情况下，可以捕捉到潜在意图，增加匹配文档的机会，提供更好的上下文信息，Spring AI MultiQueryExpander指定期望生成查询数量 会返回一个包含原始查询及多个扩展查询的列表。之后遍历这个列表去topK就行，将所有结果作为上下文

查询重写知道吗？
是RAG预检索的优化，QueryTransformer 接口及其实现类来支持查询重写。比如 RewriteQueryTransformer 可以将简单查询改写得更具体、更专业；而 CompressionQueryTransformer 则专注于处理多轮对话场景，将对话历史和当前问题压缩成一个独立的、包含完整上下文的新查询。使用时，需要为这些转换器配置一个 ChatClient 或 ChatModel，然后调用其 transform 方法传入原始查询即可得到重写后的查询。

上下文查询增强？
把用户的原始查询与从知识库中检索到的相关文档进行结合，形成一个信息更丰富的增强提示，然后将这个增强提示提供给 AI，让模型能基于这些特定知识生成回答。主要作用是为大模型提供必要的、实时的外部知识，这样 AI 的回答就不仅仅依赖于其预训练的通用知识，提高答案的准确性、相关性和时效性。
Spring AI 的 RetrievalAugmentationAdvisor 内部使用 ContextualQueryAugmenter 来实现上下文查询增强。当处理用户提出的无关问题时，ContextualQueryAugmenter 提供了空上下文处理机制。检索不到相关文档时，系统会使用这个自定义模板来指示大模型如何回应

在本项目中，我使用动态 RAG Advisor 工厂（LoveAppRagCustomAdvisorFactory）主要是为了在运行时根据特定条件（比如用户的恋爱状态）动态地创建和配置 RAG 流程中的 Advisor，将条件作为文档检索器的过滤条件，实现更精确的文档查询。
如果针对每一种可能的过滤规则（比如按 “单身”、“恋爱中”、“已婚” 等状态过滤知识库文档）都预先定义一个Advisor Bean，会导致配置繁琐且缺乏灵活性。 通过工厂模式，我们可以将创建和配置Advisor的逻辑集中起来，根据传入的参数（如status="已婚"）动态构建 Filter.Expression，并用此表达式来初始化 VectorStoreDocumentRetriever。 保证在处理不同用户情境的提问时，RAG 系统能精确地从知识库中筛选出最相关的文档子集，提高 AI 回答的针对性和准确性。

提问：检索出来的文档块顺序重要吗？放在 prompt 里的位置会影响模型回答吗？

回答：顺序确实有影响，大模型对 prompt 中间部分的注意力会弱一些，这个现象叫”Lost in the Middle”。相关性最高的文档块最好放在 prompt 开头或结尾，中间塞次要的。有研究测过，把关键信息从开头挪到中间，模型回答准确率能掉 10-20 个百分点。

提问：向量检索的相似度阈值怎么设？设太高会怎样，设太低又会怎样？

回答：阈值设太高，检索不到东西，模型只能干瞪眼，回答”我不知道”或者开始瞎编。设太低，捞回来一堆不相关的内容，噪音太多，模型被带偏。一般做法是先不设阈值，观察 Top-K 结果的分数分布，找一个明显的分数断层点作为阈值。还有一种是动态阈值，根据最高分的比例来定，Top-1 分数的 70% 作为过滤线。

提问：RAG 系统上线后怎么评估效果好不好？

回答：离线评估看检索准确率，标注一批问题和正确文档的对应关系，算 Recall@K 和 MRR。线上评估看两个指标，一个是用户反馈率，回答下面加个”有用/没用”按钮。另一个是追问率，用户连续追问说明第一次没答到点子上。还可以用 LLM 做自动评估，让 GPT 当裁判打分，虽然不完美但比没有强。

提问：文档更新了怎么办？要全量重建索引吗？

回答：看更新量级。零星改几个文档，增量更新就行，删掉旧向量，插入新向量。但如果换了 Embedding 模型，或者切割策略大改，那必须全量重建，新旧向量空间不兼容。大规模文档库一般会搞双索引热切换，新索引在后台慢慢建，建完之后一次性切过去，不影响线上服务。

工具调用

工具调用 (Tool Calling)，也常被称为 Function Calling，是一种允许 AI 大模型在对话过程中，根据需要请求执行外部工具来完成特定任务的机制。

AI 模型本身可能不具备实时查询天气、操作数据库或访问特定 API 的能力，通过工具调用，它可以识别出什么时候需要这些外部能力，并生成一个包含工具名称和所需参数的请求。 应用程序接收到这个请求后，实际执行相应的工具，并将执行结果返回给 AI 模型，模型再基于这个结果继续对话或生成最终答案。

注意，真正执行工具的是我们的应用程序，而非 AI 服务器本身。

1）定义工具：通常使用注解方式，在一个 Java 类中，将希望作为工具的方法标记上 @Tool 注解。方法的参数可以使用 @ToolParam 注解来提供描述和指定是否必需。Spring AI 支持多种 Java 类型作为参数和返回值，但返回值需要可序列化。

2）注册与使用工具：

  • 按需使用：在构建 ChatClient 请求时，通过 .tools() 方法直接传入工具类的实例。
  • 全局使用：在构建 ChatClient 时，通过 .defaultTools() 注册默认工具，这些工具对该 ChatClient 发起的所有对话都可用。
  • 集中注册：可以创建一个配置类，使用 @Bean 方法将所有工具实例化并通过 ToolCallbacks.from() 统一注册为一个 ToolCallback[] 数组，方便管理和注入。

3）调用流程：当使用配置了工具的 ChatClient 进行对话时，如果 AI 模型判断需要使用某个工具，Spring AI 框架会自动执行以下步骤，无需开发者关心：

  • 解析 AI 模型返回的工具调用请求（包含工具名和参数）。
  • 根据工具名找到对应的 Java 方法并执行。
  • 将工具执行的结果转换并返回给 AI 模型。
  • AI 模型根据工具结果生成最终回复。

对比skill

Tool 扩展的是 Agent 的”能力”（What it can do）——让 Agent 能做事； Skill 扩展的是 Agent 的”专长”（What it knows）——让 Agent 会做事。
Tool 是 Agent 的”手”，负责执行具体动作（查数据库、调 API、写文件）；Skill 是 Agent 的”经验”，封装了领域知识、流程编排和行为模式，指导 Agent 如何高效地完成特定场景的任务。

tool中模型理解工具定义（函数签名 + 描述），生成结构化的调用请求，框架负责实际执行并将结果返回模型 流程可靠性不足
skill模型仅需感知”这个技能存在、能做什么、怎么触发”，内部实现对模型完全黑盒。Skill 不直接暴露函数签名给模型，而是通过一个声明文件（如 SKILL.md） 流程稳定可控
这种设计契合 Anthropic 提出的**“渐进式披露”（Progressive Disclosure）** 原则：不一次性将所有细节灌给模型，而是按需加载，让模型专注于决策而非理解实现。

  • 底层用 Tool 实现原子能力（数据查询、消息发送、文件操作等）
  • 上层用 Skill 实现场景化封装（审批流程、报表生成、客户回访等）
  • Skill 内部编排 Tool 的调用顺序，兼顾灵活性与稳定性

工具怎么管理的

主要是参考适配器模式集中注册。在本项目中，我开发了多种工具来扩展 AI 的能力。为了有效管理这些工具并提高系统的可维护性，我采用了集中注册的方式。

具体做法是创建了一个名为 ToolRegistration 的配置类。在这个类中，我使用 Spring 的 @Configuration 注解将其标记为一个配置类，并使用 @Bean 注解定义了一个名为 allTools() 的方法。这个方法负责实例化项目中的所有工具类，例如 FileOperationTool、WebSearchTool、WebScrapingTool、ResourceDownloadTool、TerminalOperationTool 和 PDFGenerationTool。然后，通过调用 ToolCallbacks.from() 方法，将这些工具实例统一转换并封装成一个 ToolCallback[] 数组返回。ToolCallbacks.from 可以看作是一种适配器，它将不同工具类适配成 Spring AI 框架能够统一处理的 ToolCallback 格式。

这种集中注册的方式好处很多：

  1. 统一管理：所有工具的创建和配置都集中在一个地方，使得查看、添加、修改或移除工具变得非常方便。
  2. 清晰的依赖注入：如果某些工具需要配置（如API密钥），可以在这个配置类中统一处理依赖注入（比如使用 @Value）。
  3. 简化使用：其他需要使用这些工具的组件（如 LoveApp ）可以直接注入这个 ToolCallback[] 类型的 Bean，而无需关心每个具体工具的实例化细节。
  4. 易于扩展：当需要引入新的工具时，只需要在 ToolRegistration 类中实例化新工具并将其添加到 ToolCallbacks.from() 的参数列表中即可。

这种设计其实暗含了多种设计模式的应用，比如工厂模式、依赖注入模式、注册模式以及适配器模式（ToolCallbacks.from），有助于提升代码的组织性和可维护性。

工具的安全考虑过吗？
ToolContext上下文传递机制，我们有时候不希望暴露给大模型，仅在应用程序内部使用的额外上下文信息。 这些信息可以包括用户认证信息（如登录用户的 ID 或 Token）、会话 ID、请求追踪 ID、特定的环境配置参数等。在实现用户自助退款功能时，如果用户向 AI 表达 “我要退款”，AI 无需反问 “你是谁”，而是系统可以从已登录会话中获取用户 ID，通过 ToolContext 传递给退款工具，工具直接根据该 ID 执行退款操作，既安全又便捷

提问：如果模型选错了工具，或者生成的参数不对怎么办？

回答：几个处理思路。1）工具描述写详细点，让模型更容易理解每个工具的适用场景，减少选错概率。2）参数校验要严格，不合法直接返回错误信息，模型看到错误后通常会修正参数重试。3）可以在工具返回结果里加引导，比如”查询失败，请确认城市名称是否正确”，模型会据此调整。4）实在不行设置 fallback 逻辑，捕获异常后返回友好提示而不是让整个对话崩掉。

提问：工具调用和 RAG 有什么区别？什么时候用工具调用，什么时候用 RAG？

回答：RAG 是”查知识”，从静态知识库里检索相关文档，适合回答”什么是 XXX”这类知识性问题。工具调用是”做动作”，执行实时查询或操作，适合”帮我查一下今天股价”、”帮我发一封邮件”这类需要外部交互的任务。很多场景两者会组合使用：先用 RAG 找到相关背景知识，再用工具调用获取实时数据，最后综合生成回答。

提问：Spring AI 底层是怎么把工具定义传给模型的？

回答：Spring AI 会把带 @Tool 注解的方法解析成 JSON Schema 格式的工具描述，包含工具名称、功能描述、参数列表、参数类型、是否必填等信息。调用模型 API 时，这些工具描述会作为 tools 或 functions 参数一起发送。模型推理时会参考这些描述来决定是否调用工具以及怎么构造参数。不同模型厂商的 API 格式略有不同，Spring AI 做了适配层统一处理。

MCP

MCP（Model Context Protocol）模型上下文协议是一种开放标准，目的是标准化 AI 大模型与外部工具、资源和服务之间的交互方式。它把 AI 应用想象成一个主机，可以通过 MCP 这个 “USB 接口” 连接到各种外部 “设备”（即 MCP 服务），让 AI 能够访问实时数据、执行代码、调用 API 等，极大地扩展其能力边界。

是C-S架构，一个客户端可以连接多个 MCP 服务器。

利用 Spring AI 可以轻松实现 MCP 客户端和服务端：

1）MCP 客户端实现

  1. 引入依赖：添加 spring-ai-mcp-client-spring-boot-starter 依赖。
  2. 配置连接：在 application.yml 中配置 MCP 客户端属性。
  3. 使用服务：Spring AI 会自动注入一个 ToolCallbackProvider Bean。通过这个 Provider 可以获取到所有已连接 MCP 服务提供的工具（ToolCallback对象），然后将这些工具绑定给 ChatClient，AI 就可以像调用普通工具一样使用 MCP 服务提供的功能了。

2）MCP 服务端实现

  1. 引入依赖：根据传输方式选择依赖，如 spring-ai-mcp-server-webmvc-spring-boot-starter。
  2. 配置服务：在 application.yml 中配置 MCP 服务端的基本信息（名称、版本）和传输方式（启用 stdio 或配置 SSE 端点路径）。
  3. 开发工具：在 Spring @Service 类中，使用 @Tool 和 @ToolParam 注解标记Java方法，将其定义为 MCP 服务对外提供的工具。
  4. 注册工具：在 Spring Boot 主应用类中，通过定义一个 ToolCallbackProvider 类型的 @Bean

Agent

Reasoning + Acting（推理与行动）模仿人类解决问题时“思考 - 行动 - 观察”的循环
将原始问题拆分为多步骤任务，明确当前要执行的步骤。然后，它会调用外部工具执行行动（Act），比如调用搜索引擎或访问网页。最后，它会观察（Observe）工具返回的结果，并将这些结果反馈给智能体，用于下一步的决策。这个过程会不断循环迭代，直到任务完成或达到预设的终止条件。

基于 ReAct 模式构建具备自主规划能力的 AI 智能体，核心在于实现这个 “思考-行动-观察” 的循环。这意味着智能体要实现：

  1. 接收用户指令后，能分析并拆解成可执行的子任务。
  2. 根据子任务的需要，从可用的工具集中选择合适的工具并执行。
  3. 分析工具执行的结果，判断任务进展，决定下一步是继续调用工具、向用户澄清还是结束任务。
    在本项目中，我实现的拥有自主规划能力的超级智能体就是基于 ReAct 模式，通过 ReActAgent 定义先 think 思考再 act 行动的流程，然后通过 ToolCallAgent 具体实现这两个方法。在 think() 方法中决定使用哪个工具（推理），在 act() 方法中执行工具（行动），并将执行结果作为下一步思考的输入（观察）。

OpenManus？
OpenManus 是一个开源的自主规划智能体项目。它的核心亮点在于其 “自主执行” 能力，能够自主规划任务，并且在虚拟机中调用各种工具（如编写代码、爬取数据）来完成用户指定的复杂任务。它的实现原理主要基于以下几点：

  1. 分层代理架构：OpenManus 采用了分层的代理架构，不同层次的代理负责不同的功能，方便系统的扩展和维护。主要代理层次包括 BaseAgent（基础代理，管理状态和执行循环）、ReActAgent（实现 ReAct 模式）、ToolCallAgent（扩展了工具调用能力）以及具体的智能体实例（如 Manus）。
  2. ReAct 模式：OpenManus 的工作流程遵循“思考-行动-观察”的循环，ToolCallAgent 的 think 方法负责与大模型交互、选择工具，act 方法负责执行工具，并将结果反馈给大模型进行下一步决策。
  3. 工具系统：OpenManus 拥有一个灵活的工具系统。所有工具都继承自 BaseTool 抽象基类，提供统一的接口和规范化的参数描述，方便大模型理解和调用。ToolCollection 类可以管理多个工具实例，实现了工具系统的可插拔性。
  4. 核心组件：包括记忆系统（Memory 类存储对话历史和中间状态）、大模型（LLM 类提供思考和决策能力）以及流程控制（AgentState 和执行循环管理状态转换和任务流程）。
  5. 特殊工具设计：比如，Terminate 工具是让智能体自主决定何时结束任务，AskHuman 工具则是让智能体在遇到困难时向人类寻求帮助。
  6. MCP 协议：最新版本的 OpenManus 已经支持，通过 MCPClients 类将 MCP 服务集成到其工具系统中，让远程工具能够像本地工具一样被调用。

你是怎么实现分层架构的？
在本项目中，我参考了 OpenManus 的思想，实现了一个简化的分层智能体架构，主要包含以下几个层次：

  1. BaseAgent：这是智能体的基类，定义了智能体的基本属性，比如名称、系统提示词、最大执行步骤、当前步骤、状态、ChatClient以及一个 messageList 用于维护对话上下文。还包含了一个核心的 run 方法，负责整个多步骤执行流程的控制，以及一个抽象的 step 方法，具体的单步执行逻辑交由子类实现。这种设计也是模板方法设计模式的体现。
  2. ReActAgent：这个类继承自 BaseAgent，进一步将 step 方法的实现分解为两个核心的抽象方法：think()（思考）和 act()（行动）。它实现了 ReAct 模式的核心逻辑：在每一步中先思考（think），然后根据思考结果决定是否行动（act）。
  3. ToolCallAgent：继承自 ReActAgent，作用是实现工具调用的能力。在 think() 方法中，它会根据当前的对话上下文和可用的工具列表，调用大模型决定需要使用哪些工具以及相应的参数，也就是“推理”阶段。在 act() 方法中，实际执行大模型在 think 阶段选定的工具。这里我利用了 Spring AI 提供的 ToolCallingManager 来执行工具调用、获取工具的执行结果。这个过程是“行动”和部分“观察”阶段。工具的执行结果会更新到 messageList 中，作为下一轮 think 的输入。而且，我在 ToolCallAgent 中通过设置 DashScopeChatOptions 的 withProxyToolCalls(true) 禁止 Spring AI 自动托管完整的工具调用流程，方便自行控制和观察 ReAct 的每一步。
  4. YuManus：这是我最终实现的、可以直接使用的 AI 超级智能体实例，它继承自 ToolCallAgent。在这个类中，我为智能体设置了具体的名称、系统提示词、下一步提示词、最大执行步数，并注入了所有可用的工具和 ChatModel。
    基于这个分层架构，我可以很方便地扩展出各种各样不同领域的智能体，除了通用智能体 YuManus 之外，我只需修改 Prompt，就能定义出软件工程师智能体、数据分析师智能体等等。

CoT 思维链？
CoT（Chain of Thought）思维链 就是让大模型别急着给答案，先把推理过程一步步写出来。这招对数学题、逻辑推理、多步骤问题特别有效，准确率能提升 20% 以上。
核心原理很直白：人做复杂题也得打草稿，让 LLM 把”草稿”写出来，中间步骤错了更容易发现，最终答案也更靠谱。

死循环解决
在本项目中，我主要通过循环检测和处理机制来解决。具体方法是在智能体的执行逻辑中：

  1. 检测重复响应：记录智能体最近的几次交互或产生的消息。
  2. 设置重复阈值：定义一个重复次数的上限，比如连续两次或三次出现相同的响应内容，就可能判断为陷入了循环。
  3. 主动干预：一旦检测到这种卡顿或循环状态，系统会向智能体的下一步提示词中注入特定的指令，比如：“观察到重复响应。考虑新策略，避免重复已尝试过的无效路径”。这样做的目的是引导 AI 打破当前的执行逻辑，尝试新的解决方案，跳出死循环。当然，也可以在遇到死循环时强制中断当前任务的执行。

怎么让任务退出
在本项目中，我参考了OpenManus的做法，通过引入一个特殊的工具 TerminateTool 实现任务合理退出。有了这个工具，AI 智能体能根据自身的判断决定什么时候结束任务，而不是仅仅依赖于达到预设的最大执行步骤数或发生错误。具体实现方式如下：

  1. 定义一个 TerminateTool，写好明确的描述，告诉 AI 在任务完成或无法继续时应调用此工具。
  2. 在智能体的思考（Think）阶段，大模型会评估当前任务的进展。如果判断任务已成功完成，或者认为无法取得进一步的进展，AI 会自主选择调用 TerminateTool。
  3. 当 ToolCallAgent 在行动（Act）阶段检测到 TerminateTool 被调用时，直接将智能体的内部状态更新为 FINISHED。随后，BaseAgent 中的主执行循环 run() 方法会监测到这个状态变化，随后终止循环，使整个任务有序退出。

这种机制赋予了 AI 一定的自主性，能更智能地结束任务，避免不必要的资源消耗。

提问：如果模型在 CoT 的中间步骤就推错了，后面的步骤会全跟着错，怎么缓解这个问题？

回答：几个方向可以尝试。第一是 Self-Consistency，采样多条路径然后投票，单条路径错了只要多数对就行。第二是 Self-Verification，生成答案后让模型用答案反推验证，不对就重来。第三是 Step-by-Step Verification，每一步都让模型打分评估置信度，置信度低的步骤触发重新生成。OpenAI 的 GPT-4 训练里就用了 Process Reward Model，对每一步推理打分而不是只看最终答案。实际工程里最简单有效的还是 Self-Consistency，多花几倍成本但稳定性好很多。

提问：Tree of Thoughts 和普通 CoT 有什么区别？什么时候该用 ToT？

详细解答可以看：什么是思维树 Tree of Thoughts？它相比 CoT 有什么优势？

回答：普通 CoT 是一条路走到黑，生成完整推理链后给答案。Tree of Thoughts 是每一步都生成多个候选分支，形成一棵树，然后用搜索算法找最优路径。ToT 适合那种”中间步骤选错了就完蛋”的问题，比如 24 点游戏、创意写作大纲规划。代价是计算量指数级上升，4 个分支点每个 3 个候选就是 81 条路径。实际用的时候通常加剪枝策略，只扩展评分高的分支。一般推理任务用 Self-Consistency 够了，ToT 属于杀鸡用牛刀。

提问：在实际项目里，怎么判断一个任务适不适合用 CoT？

回答：看两个指标。第一个是任务是否需要多步推理，如果问题本身一步就能答上来比如”Python 列表怎么排序”，CoT 就是浪费 token。第二个是直接问答的错误率，先不用 CoT 测一批样本，如果准确率已经 90% 以上，CoT 提升空间有限；如果只有 50%-60%，大概率能通过 CoT 拉高 20 个点以上。还有个经验法则是看问题里有没有数字运算或者条件分支，有的话 CoT 收益通常比较明显。

异步响应

为什么选择SSE交互？WebSocket呢？
SSE 服务器发送事件（Server-Sent Events）是一种用于从服务器到客户端的 单向 数据传输技术，基于 HTTP 协议实现。连接一旦建立，服务器就可以持续地将新的信息流式传输给客户端，不需要客户端重复发起请求，非常适合需要服务器主动更新数据的场景，比如我们 AI 应用中逐步显示大模型生成的回复，实现类似打字机的效果。
不选择 WebSocket 的原因：

  1. 实现成本：SSE 基于标准的 HTTP 协议，实现相对简单，前端使用 JavaScript 的 EventSource API 即可轻松接收。WebSocket 则是一个独立的、更复杂的双向通信协议，需要专门的连接管理和握手过程。
  2. 通信模式：AI 应用的流式响应通常是服务器向客户端的单向数据推送，跟 SSE 的单向通信模型完美契合。WebSocket 提供的是全双工双向通信，对于 AI 流式输出这类场景，双向能力并不必要，反而增加了复杂性。
  3. 内置机制：SSE 规范包含自动重连和事件 ID（lastEventId）机制。如果网络连接意外中断，浏览器可以自动尝试重连，还可以通过上次接收到的事件 ID 告诉服务器从什么地方继续，对于保证数据流的连续性很有用。这些在 WebSocket 需要开发者自行实现。
  4. 资源消耗：一般来说，SSE 比 WebSocket 更轻量，WebSocket 需要建立和维护一个持久的 TCP 连接。

WebSocket 在需要真正实时双向交互（比如在线聊天室、协同编辑工具）的场景下是更优的选择，但对于 AI 应用的场景，SSE 更加简单、高效，比 WebSocket 更合适。

之前 智能协同云图库项目 中，就带大家实战过使用 WebSocket 实现图片的多人协同编辑功能。

SseEmitter 是 Spring MVC 框架提供的一个类，主要用于在服务器端实现 SSE，让后端应用在一个长时间的 HTTP 连接上，能够异步、持续地向客户端发送事件流数据。
特别是在大模型逐步生成响应的 AI 应用中，SseEmitter 可以用来把 AI 服务生成的内容块实时推送给前端，实现打字机效果，优化用户体验，避免用户长时间等待完整结果。
使用 SseEmitter 实现 AI 流式响应的步骤大致如下：
1）Controller 方法返回 SseEmitter：在 Spring MVC Controller 中，处理 SSE 请求的方法需要返回一个 SseEmitter 对象。
2）异步处理 AI 调用：由于 AI 模型的调用可能是耗时的，不能阻塞处理 HTTP 请求的线程。因此，一般需要在一个单独的线程或使用异步机制调用 AI 服务。
3）通过 emitter.send() 推送数据：在异步任务中，每当 AI 模型生成一部分响应，就通过 emitter.send() 方法将这部分数据发送给客户端。

你怎么保证输出的稳定性和性能
为了保证 AI 应用的性能和稳定性，需要综合考虑很多方面，从应用设计、外部依赖管理到部署策略都需要考虑。一般来说有以下措施：

1）异步处理：对于耗时的 AI 操作，要采用异步处理，避免阻塞服务器主工作线程，提高并发处理能力和系统响应速度。还可以利用 SSE (Server-Sent Events) 和 SseEmitter 实现流式输出，提高用户体验。
2）合理的模型选择：选择跟场景、需求相匹配的 AI 模型。不是所有场景都需要最大、最先进的模型；有时候针对特定任务优化的轻量级模型在成本和延迟上更有优势。比如计算向量 Embedding 时，就不必选择深度推理能力强的模型。
3）RAG 系统优化：选择高质量的原始文档和合理的切片策略。以及选择合适的向量数据库，比如使用 PGVector 方案，可以对索引和查询参数调优。
4）外部依赖管理：对工具调用、MCP 服务及其他外部 API 的调用，实现错误处理和重试机制（比如 Guava Retrying 库）。并且针对所有的外部调用设置合理的超时时间。
5）对于流量不确定或需要快速迭代的服务，可以考虑 Serverless 部署，按需分配资源、自动伸缩。
6）API 设计：避免在每个数据块中包含过多冗余信息，只传输核心内容，减少网络负载。
7）可观测性与监控：集成更全面的监控方案，比如使用 Prometheus + Grafana 追踪关键性能指标、延迟、错误率、资源消耗等。

大问题

大模型 Agent 是以大语言模型为大脑，配合规划、记忆、工具调用等模块，能够自主完成复杂任务的系统。它和传统 AI 最大的区别在于：传统 AI 是你问一句它答一句，Agent 是你给它一个目标，它自己拆任务、调工具、一步步干完。

具体差异可以从这几个维度来看：
1）目标导向 vs 被动响应。传统大模型就是个对话机器，用户输入什么它输出什么，没有主动性。Agent 不一样，你给它一个最终目标，比如”帮我调研竞品并写一份分析报告”，它会自己规划要做哪些步骤、先做什么后做什么，然后一步步执行。
2）有记忆 vs 无状态。传统模型只能看到当前对话的上下文窗口，聊完就忘。Agent 有专门的记忆模块，短期记忆存当前任务的中间状态，长期记忆存历史经验，下次碰到类似问题可以复用。
3）能调用工具 vs 只能说话。传统模型只能输出文本，想搜索、算数、操作数据库都得人来做。Agent 能调用搜索引擎、执行代码、读写文件、调 API，真正把事情做出来。
4）多步推理 vs 单轮问答。Agent 能在执行过程中不断评估结果、发现问题、调整策略，形成一个闭环。传统 AI 系统要么靠写死的规则引擎，要么就是一锤子买卖。

挑战：
1）错误累积。每一步都可能出错，10 步下来准确率就成了 0.9 的 10 次方，大约 35%。这是 Agent 目前最大的痛点。
2）成本高。一个复杂任务可能要调几十次 LLM，Token 消耗很快就上去了。OpenAI 的 GPT-4 每百万 Token 输入 30 美元、输出 60 美元，跑一个 AutoGPT 任务花几美元很正常。
3）安全风险。Agent 能调用工具、执行代码，一旦被提示词注入攻击，后果比普通聊天机器人严重得多。
4）可控性差。自主性越强，行为就越难预测。生产环境一般会加各种护栏，限制 Agent 能调用的工具和能执行的操作。

提问：ReAct 和 Plan-and-Execute 两种 Agent 范式有什么区别？实际项目中怎么选？

回答：ReAct 是边想边做，每一步先推理要做什么、为什么，然后执行，看到结果再决定下一步。优点是灵活，能根据中间结果随时调整；缺点是容易跑偏，任务一复杂就会丢失方向。Plan-and-Execute 是先整体规划出完整的步骤列表，再逐个执行。优点是稳定，不容易偏离目标；缺点是不够灵活，如果中间某步失败需要重新规划。实际选择看任务特点：探索性强、结果不确定的用 ReAct；目标明确、步骤可预期的用 Plan-and-Execute；复杂任务可以混用，外层 Plan-and-Execute 保证方向，内层 ReAct 处理细节。

提问：Agent 的记忆模块怎么设计？长期记忆和短期记忆分别用什么技术实现？

回答：短期记忆最简单的做法就是把对话历史和中间结果拼到 Prompt 里，受限于模型的上下文窗口。更好的做法是用 Scratchpad，把关键信息结构化存储，只在需要时取出来。长期记忆一般用向量数据库，把历史经验、知识片段 Embedding 之后存进去，检索时算相似度取 Top-K。常用的向量库有 Pinecone、Milvus、Chroma、Weaviate。设计上要注意记忆的增删改查策略，不能无限堆积，需要定期压缩或淘汰不重要的内容。

提问：怎么解决 Agent 多步执行时的错误累积问题？

回答：几个常见的思路：一是加强单步准确率，用更强的模型、更好的 Prompt、更精准的工具；二是加检查点，每步执行完校验结果是否合理，不合理就重试或回退；三是引入自我反思机制，让 Agent 定期回顾执行历史，发现偏离目标就及时纠正；四是限制自主性，关键操作要人工确认，不让 Agent 完全自动跑。实际生产中这几种方法一般组合使用。

RAG

提问：如果检索出来的文档片段太多，超过了大模型的上下文窗口怎么办？

回答：常见做法有三种。一是做 Rerank，用一个排序模型对检索结果重新打分，只保留最相关的 Top-K 条；二是做摘要压缩，对检索出来的长文档先跑一遍摘要，把信息密度提上去；三是分块处理，把问题拆成多个子问题分别检索回答，最后再做汇总。实际项目里 Rerank + Top-K 截断是最常用的组合。

提问：RAG 和模型微调有什么区别？什么场景下该选哪个？

详细解答可以看：RAG 和模型微调 Fine-tuning 有什么区别？如何选择？

回答：微调是把知识”刻”进模型参数里，适合需要改变模型行为风格或者学习特定任务模式的场景，比如让模型用特定语气说话。RAG 是把知识放在外部，适合知识会频繁更新、需要引用来源、或者私有数据量大的场景。微调一次成本高、知识更新要重新训练；RAG 只需要更新知识库，模型不用动。如果你的场景是”让模型知道更多事实”，选 RAG；如果是”让模型换一种方式说话”，选微调。

提问：向量检索找到的内容语义相近，但可能不是真正相关的，怎么解决？

回答：这是语义检索的经典问题，向量相似不等于真正相关。常见解决方案：一是混合检索，把向量检索和关键词检索结合起来，用 BM25 这类传统检索补一刀；二是加 Rerank 层，用交叉编码器对候选文档做精排，过滤掉语义漂移的结果；三是优化分块策略，文档切得太碎容易丢上下文，切得太大又不够精准，需要根据业务调整 chunk size 和 overlap。

RAG 检索效果差，问题通常出在文档切片、查询理解、检索策略三个环节。优化要从源头抓起。

1）文档预处理：原始文档质量不行，后面再怎么调参都白搭。首先清洗掉水印、页眉页脚、乱码，保证内容干净。切片策略很关键，固定 512 tokens 切一刀这种做法太粗暴，容易把一句话从中间截断。推荐用语义边界切分，比如按段落、按小节，或者用 LangChain 的 RecursiveCharacterTextSplitter 先按段落切，段落太长再按句子切。另外给每个切片打上元数据标签，来源文档、章节、日期这些，检索时能做精确过滤。

2）查询增强：用户问的”鱼皮是谁”跟知识库里写的”程序员鱼皮，原名某某，技术博主”对不上，向量相似度不高。解决办法是用大模型做查询重写，把口语化的问题转成规范表达，或者做查询扩展，一个问题拆成 3-5 个不同角度的子查询，合并召回结果。

3）混合检索：单纯的向量检索有个致命问题，专有名词、代码片段、数字这些精确匹配的东西召回率很低。解决办法是向量检索 + BM25 关键词检索组合，用 RRF 算法融合排序。Elasticsearch 8.x 原生支持这种混合检索。
除此之外，还有重排序：召回 top 50 之后，用 Cross-Encoder 重排序模型精排一遍，把真正相关的排到前 5。Cross-Encoder 比 Bi-Encoder 准确率高 10-15%，但速度慢，所以只用来精排。

提问：如果知识库里有多种格式的文档，比如 PDF、Word、网页，切片策略要怎么处理？

回答：不同格式的文档结构差异很大，不能一刀切。PDF 要先用 PyMuPDF 或 Unstructured 提取文本，注意保留段落结构，表格单独处理。Word 文档可以按大纲级别切分，一级标题下的内容作为一个 chunk。网页要先清洗 HTML 标签，按 DOM 结构或者 <article> 标签提取正文。最好是建一个预处理 Pipeline，根据文件类型走不同的处理分支，最后统一输出成标准化的切片格式。

提问：检索结果里有重复内容怎么办？

回答：这个问题在用滑动窗口切片时很常见，相邻切片内容重叠导致召回的好几条结果其实是同一段话。解决办法是在召回之后做去重，可以用 MinHash 或者 SimHash 算相似度，相似度超过 0.9 的只保留一条。另一种办法是在检索时加上 document_id 去重，同一个文档最多召回 2-3 条。LangChain 的 MultiQueryRetriever 有内置的去重逻辑可以直接用。

提问：怎么评估 RAG 的检索效果？有什么指标？

回答：常用指标是 Recall@K 和 MRR。Recall@K 是看前 K 条结果里有没有包含正确答案，K 一般取 5 或 10。MRR 是正确答案排名的倒数的平均值，排名越靠前分数越高。评估需要一个标注好的测试集，包含问题和对应的正确文档 ID。实际操作中可以先人工标注 100-200 条，跑一遍看基线效果，然后每次调参之后对比变化。如果没人力标注，可以用大模型自动判断检索结果是否包含问题的答案，不完全准确但能看趋势。

CoT 思维链 (Chain of Thought) 和 ReAct 模式 (Reasoning + Acting) 都是增强大型语言模型推理和解决复杂问题能力的技术。

  • CoT 思维链：生成最终答案之前，先引导 AI 模型输出一系列中间的、连贯的推理步骤。引导模型“思考过程化”，模拟人类解决问题时逐步分析和推导的过程。它通过让模型显式地写出思考步骤，可以帮助模型分解复杂问题，减少在复杂逻辑链条中出错的概率。这些中间步骤也为我们理解模型的“思路”提供了便利，调试和优化都会更简单方便。
  • ReAct 模式：一种将LLM的推理 (Reasoning) 能力与行动 (Acting) 能力相结合的框架。它让模型不仅能思考，还能决定采取何种行动（通常是调用外部工具，例如文件操作、联网搜索），然后根据行动的观察结果执行下一轮的思考和行动，形成一个“思考-行动-观察”的循环。

CoT 主要关注提升模型内部的逻辑推理连贯性和深度，通过显化思考过程实现。ReAct 更侧重于让模型与外部世界互动，通过“推理驱动行动，行动反馈观察，观察指导推理”的闭环来解决那些需要外部信息或操作才能完成的复杂任务，更考验模型动态调整推理路径的能力。
