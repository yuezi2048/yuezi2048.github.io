JVM核心

JVM内存：

  • 类加载子系统ClassLoader（动态加载 将编译后的.class文件转换成class对象，让JVM识别）
    • 【加载 连接（验证 准备 初始化） 解析 】
    • 双亲委派机制 类加载器优先委托给父类加载器，父类加载器无法加载时，才由当前类加载器加载，保证java核心类库的安全性，避免核心类库篡改/替换
  • 运行时数据区Runtime Data Area：类加载的内存区域
    • 方法区（永久代/元空间）：使用本地内存而不是堆内存避免OOM，存储类信息、常量、静态变量
      • 1.7及之后，字符串常量池从方法区单独划分到了堆中，通过String类的intern方法获取常量池的引用
      • 而方法区存放的是运行时常量池：类/接口的class编译生成的常量信息（基本数据类型字面量、类/接口引用、方法/字段符号引用）
    • 堆；垃圾回收目标，Eden、Survivor（s0 s1区分）、老年代（Major GC）
    • 虚拟机栈（JVM Stack)：局部变量、操作数栈、动态连接、方法出口信息
      • 局部变量表存储基本数据类型和对象引用
    • 本地方法栈
    • 程序计数器PC：线程字节码行号，很小
  • 执行引擎Execuction Engine：将字节码翻译成指定操作系统指令交给CPU执行
  • 本地方法接口JNI：执行引擎的依赖
  • 直接内存：NIO 不受JVM控制，虚引用cleanner释放（虚引用用来跟踪回收状态，回收的时候，他会进入ReferenceQueue，通过引用队列进行清理）

JIT：运行时将热点代码字节码转换成字节码，方法内联（Inlining）、逃逸分析（Escape Analysis）、循环展开（Loop Unrolling），存储在代码缓冲区中,AOT是运行前就转换成了字节码

逃逸分析有方法逃逸和线程逃逸。优化是三种：栈上分配（没有逃出方法，就不再堆中创建）、标量替换（对象没有逃逸出且可以拆解，直接替换为标量）、同步消除（如果只在内存中使用，移除不必要同步锁）

垃圾回收技术（目标是为了最短暂停时间和高吞吐量，尽可能让对象在年轻代回收，减少进入老年代。）

  • STW阻塞的思路
  • ParNew是Serial的多线程版本（标记 复制），Serial Old是Serial的老年代版本（标记 整理），直线表示的是组合的兼容关系
  • Parallel Scavenge是兼顾吞吐量的控制（代码时间 / 代码时间 + GC时间）

标记-复制的思路（内存换时间，dfs/bfs，CPU缓存亲和性好，没有链表维护，没有碎片问题）

CMS优化：标记-清除（碎片问题+链表遍历+COW不兼容），思路是初始标记GC Roots直接关联 + 增量更新的重新标记 + 并发清理的方式，优化STW的开销

并发保证，三色标记法中说明：当黑色对象A引用白色对象C时，加入写屏障将白色C变成灰色C，从而能保证跟踪到C，避免在GC过程中灰色B->白色C删除引用，进而导致C无法通过可达性检索/引用计数法找到，造成漏标的严重事故。下图说明的是漏标事故没有增量更新/SATB的过程（灰色是当前访问到的，黑色是引用处理完毕，白色是未访问到的认为是垃圾）

而G1的思路是在GC过程中， 灰色删除堆白色对象引用的时候，将白色对象设置为灰，这是对快照的处理STAB（Snapshot-at-the-begining）

CMS垃圾回收时发现老年代Old Generation空间不足以分配新对象时，被迫使用Full GC进行垃圾回收，这是concurrent_mode_failure

  • CMS使用卡表记录老年代引用年轻代的对象，卡表维护简单，老年代指向新生代对象时会触发写屏障标记相应的卡片，在相应区域用写屏障标记为脏卡，后续GC扫描只需扫描这些区域
  • G1的记忆集的粒度可以细化到堆的各个区域（Region），用于跟踪Region中的对象引用了其他Region的对象
    • 基于多层次记忆集维护机制，将老年代对新生代引用、其他Region之间的引用关系都记录在记忆集中，每个Region都有自己的记忆集，维护成本高但有助于G1精准式地增量式回收

G1：全周期的标记-整理思路，所有对象在Region中，在CMS基础上：

  • 初始标记会记录TAMS指针，让下一个线程运行时可以在正确Region中分配新对象
  • 并发标记的过程中：会处理SATB开始阶段快照变动对象
  • 最终标记，处理SATB的变动，同CMS思路
  • 回收：Region的回收价值和成排序，在期望停顿时间下，制定回收计划

PLAB（Promotion Local Allocation Buffer） 是对垃圾回收器的优化，用于G1垃圾回收器，提高对象晋升（Promotion）到老年代的效率

每个线程可在本地的PLAB直接进行对象晋升操作，而不需要全局竞争全局老年代内存空间，减少锁竞争

young GC触发

Young GC（Minor GC） 是针对新生代（Young Generation）对象的垃圾回收，三种情况触发。

  • Eden区分配空间不足：大部分新创建的对象在Eden区，如果Eden区填满则触发，回收新生代不需要的对象
  • Eden区+Survivor区被填满：Eden+Survivor区不足以存放新分配对象时触发young GC清理空间，同时会将幸存对象转移到survivor / 老年代
  • 部分垃圾回收器在full gc之前：Parallel Scavenge等实现会在full GC前先执行young GC，通过参数可调整该策略

补充：有小友提到了TLAB缓冲区，TLAB内存不够后会向Eden区申请

TLAB(Thread-local allocation buffer)是JVM为每个线程分配的一小块堆内存，每个线程通过自己的TLAB可以大大加快内存分配，并且避免竞争共享内存的同步开销

分配内存的具体过程：

  • 每个线程优先从自己的TLAB中分配内存
  • TLAB内存耗尽时，向Eden区申请新的TLAB，或者直接在Eden区分配内存
  • 对象超过一定大小（大对象），就不再TLAB中分配了，而是直接在Eden区分配内存

full GC触发

  • 永久代/元空间不足：PermGen（固定）/MetaSpace（设置阈值）内存如果不足，可能触发Full GC
    • 老年代触发：老年代空间不足且无法通过垃圾回收器释放足够空间后，会触发Full GC
    • 新生代触发：
      • 新生代晋升老年代失败：长期存活对象到老年代后，老年代空间不足，触发Full GC
      • 空间分配担保(Promotion Failed)：To放不下eden区和from拷贝过来的对象后，直接晋升到老年代后，如果老年代空间不足，触发Full GC
      • 平均晋升大小计算：根据之前统计数据发现年轻平均晋升大小 > 老年代剩余空间，越过老年代直接触发Full GC
    • 命令：System.gc()、jmap-dump等指令，建议JVM执行一次Full GC，不保证立刻执行，可能触发。

排查使用：

  1. 内存溢出：jmap + MAT
  2. 内存泄露：jstat + jmap + MAT
  3. GC频繁：GC日志 + jstat
  4. 线程死锁：jstack
  5. CPU占用高：top + jstack
  6. 长期监控：prometheus + Grafana

SpringBoot核心

SpringBoot启动流程：

  • new Application().
    • 推断应用类型（servlet/reative）、设置初始化器、设置监听器、确定主应用类
  • start
    • 准备环境ConfigurableEnvironment：application.properties 系统变量 命令行等配置文件，完成后有ApplicationEnvironmentPreparedEvent
      • @Value @ConfigurationProperties(prefix=”?”) Environment接口注入 env.getProperty(“”)
    • 创建-刷新应用上下文ApplicationContext
      • Web服务器启动时机是刷新应用上下文后自动嵌入
    • 启动完毕后，发送应用启动事件ApplicationStartEvent
    • 回调函数：调用实现了CommandLineRunner / ApplicationRunner接口的bean执行初始化
    • 发布ready事件ApplicationReadyEnvent，此时可以接收业务请求和任务

ApplicationContext创建：自动装配Bean（条件注解）、加载配置类（WebMvcAutoConfiguration 自定义Configuration）、注册BeanPostProcessor（处理Bean生命周期）

ApplicationContext刷新：创建/初始化单例Bean、BeanPostProcessor加工（AOP代理对象等）、事件发布ContextRefreshedEvent

自动装配原理：@EnableAutoConfiguration + spring.factories来节省大量XML配置

细节补充：@EnableAutoConfiguration注解内部还有一个@Import({AutoConfigurationImportSelector.class})，导入的类负责扫描classPath下所有的META_INF/spring.factories的文件，根据文件指定的配置类加载对应的bean自动配置

而对于bean的加载时机，则通过 @ConditionalOnClass、@ConditionalOnMissingBean、@ConditionalOnProperty 等条件注解来控制自动配置的加载条件，比如仅在类路径中存在某个类时，才加载某些配置。

  • 场景1：这个原理在我阅读我组长写的starter源码的时候就用到了这个方法来进行自动配置
  • 场景2：@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})，云图库项目中开发环境下不需要分表相关的模块增加难度，我就去掉了提高测试效率。

操作系统和计网

输入网页过程：

封装过程

  • 解析URL：浏览器根据请求信息解析HTTP请求报文
  • 应用层DNS解析：首先明确对方的ip地址，检查本地缓存（操作系统缓存 甚至路由器缓存），如果没有则向对应的递归DNS服务器发送查询，服务器递归查询最终返回IP地址
  • 传输层TCP/UDP：浏览器调用Socket库委托协议栈工作，选用TCP/UDP协议，如果是TCP则进行三次握手建立连接后，进行数据通信。对应的报文是在HTTP报文基础上封装了一层TCP数据包
  • 网络层IP：TCP包基础上封装源IP 目的IP，得到网络包，IP就可以在多个网络节点确定传输路径，找到目标服务器
  • 网络接口层MAC：在IP头部加上MAC头，封装发送方的MAC和目标MAC，保证子网内设备的通信寻址（区别于IP的多个网络节点寻址）
  • 物理层网卡：网络包此时是二进制数据，通过网卡将二进制数据转换成电信号传输

传递过程

  • 网络接口层交换机：网线连接交换机，是二层网络设备，根据数据包的MAC头找到另一个设备连接在交换机哪个端口，进行传输（找不到就广播）
  • 网络层路由器：数据通过路由器在不同网络节点进行转发，最终到达服务器
  • 传输层匹配进程：ip mac匹配的基础上，匹配TCP序列号，如果匹配就分配到对应的监听进程，此时服务器就能在对应的应用上收到数据
  • 应用层服务器处理：服务器收到请求后，处理业务逻辑（读取数据库 访问文件等），生成HTTP响应（继续经过TCP IP MAC头的封装） 从网卡到交换机再到路由器，最终服务器收到服务器的响应，进行前端页面渲染展示。

四次挥手

  • 第一次挥手（FIN）：客户端主动关闭连接，发送FIN包，进入FIN_WAIT_1阶段，服务器收到FIN后表示不再接收数据，但仍然发送数据（半连接状态）
  • 第二次挥手（ACK）：服务器发送ACK，确认收到FIN，服务器进入CLOSE_WAIT状态，客户端收到后进入FIN_WAIT_2状态
  • 第三次挥手（FIN）：服务器完成数据传输后，发送FIN包，进入LAST_ACK状态，客户端收到后则准备关闭连接
  • 第四次挥手（ACK）：客户端发送最后ACK，进入TIME_WAIT状态，等待可能迟来的FIN包，服务器收到ACK后则进入CLOUS状态，客户端TIME_WAIT进入2MSL计时结束后，才正式关闭连接

三次握手

  • 客户端发送SYN消息给服务器
  • 服务器回复SYN-ACK（同步序列编号-确认）信息
  • 客户端发送ACK信息确认收到SYN-ACK信息，完成三次握手

什么是虚拟内存

虚拟内存是逻辑上的内存地址空间，让程序以为独占一大片连续内存，实际不直接对应物理内存。

操作系统通过页表映射 + 缺页中断，把虚拟地址转成物理地址，不够时用硬盘空间当临时内存。

进程是具有独立功能的程序在一个数据集合上的一次执行过程，是资源分配的基本单位。

线程是进程内的执行路径，是CPU 调度的基本单位，共享进程资源，开销小。

  • 进程：独立资源、独立地址空间、切换开销大、健壮性高。
  • 线程：共享资源、同一地址空间、切换快、通信简单。

进程状态（5 种）

新建、就绪、运行、阻塞、终止

就绪→运行（调度）；运行→阻塞（I/O）；阻塞→就绪（I/O 完成）。

进程通信（IPC）高频 5 种

  1. 管道：半双工、父子进程、字节流。
  2. 命名管道：不相关进程也可通信。
  3. 消息队列：消息链表、独立于进程、可随机读取。
  4. 共享内存：最快 IPC，直接共享物理内存，需同步。
  5. 信号量：用于同步与互斥，P 操作申请，V 操作释放。

死锁（必考）

  1. 死锁定义

多个进程互相等待对方资源，都不释放，导致永久阻塞。

  1. 死锁四个必要条件（必背）

  2. 互斥：资源独占

  3. 请求并保持：占有资源再申请

  4. 不可剥夺：资源不能强行抢占

  5. 循环等待：形成资源等待环

  6. 死锁处理策略

  • 预防：破坏四个条件之一
  • 避免：银行家算法
  • 检测 + 解除：撤销进程、剥夺资源

调度算法（高频）

  1. FCFS：先来先服务，利于长作业，不利于短作业。
  2. SJF：短作业优先，平均等待时间最短，可能饥饿。
  3. 优先级调度：高优先级先运行，可能饥饿。
  4. 时间片轮转 RR：分时系统，公平，适合交互。
  5. 多级反馈队列：综合最优，不需要事先知道作业长度。

临界区

访问临界资源的代码段，同一时间只允许一个进程进入。

  1. 互斥 vs 同步
  • 互斥：同一时间只有一个执行（竞争关系）。
  • 同步：按预定次序执行（协作关系）。
  1. P、V 操作（信号量）
  • P 操作：申请资源，信号量 - 1，<0 则阻塞。
  • V 操作：释放资源，信号量 + 1，≤0 则唤醒。

内存管理（你刚才要的虚拟内存）

  1. 覆盖与交换
  • 覆盖：同一进程内分段覆盖，节省内存。
  • 交换：进程整体换入换出到磁盘。
  1. 分页
  • 把进程分成固定大小页，内存分成物理块。
  • 通过页表映射，地址不连续，无外部碎片。
  1. 分段
  • 按逻辑意义分段，段长不固定，方便共享保护，有外部碎片。
  1. 段页式

先分段，再分页，兼具分段逻辑清晰、分页内存利用率高。

  1. 虚拟内存（最终背诵版）

虚拟内存是操作系统为进程提供的逻辑地址空间，使进程拥有连续、大于物理内存的地址空间。

通过请求调页、页表、缺页中断、页面置换实现，扩充内存、地址隔离、按需调页。

  1. 页面置换算法
  • FIFO：先进先出，可能Belady 异常。
  • LRU：最近最少使用，性能最好，实际常用。
  • LFU：最少使用次数。

文件管理

  1. 文件物理结构
  • 连续分配：顺序快，不易扩展。
  • 链接分配：离散，不能随机访问。
  • 索引分配：支持随机访问，现代系统主流。
  1. 磁盘调度算法
  • FCFS：简单，不公平。
  • SSTF：最短寻道，可能饥饿。
  • SCAN（电梯）：双向扫，最公平常用。
  • C-SCAN：单向循环，平均寻道更稳定。

I/O 管理

  1. I/O 控制方式

  2. 程序查询：CPU 轮询，效率低。
