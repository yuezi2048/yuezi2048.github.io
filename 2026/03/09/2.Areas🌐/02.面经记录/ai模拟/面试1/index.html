第一阶段：一面（基础 + 核心原理，偏知识完整性）

模块 1：Java 集合

说下 ArrayList 和 LinkedList 的核心区别，以及各自使用场景？

  • ArrayList是基于数组结构的，LinkedList是基于链表结构的，他们的核心差异在于存储的方式，数组是基于连续的内存块，而链表是基于结点的，通过next指针相连，内存块之间是不连续的。
  • 这样会导致数组在随机访问时会更有效率，是O1，而链表则需要遍历On访问，但是链表的优势的插入删除，给定指定节点增删情况下只需要时间复杂度O1修改指针
  • 扩容差异：ArrayList在无参构造下初始化为0，第一次扩容到10，每次扩容1.5倍，通过Arrays.copy复制，但是LinkedList没有扩容机制
    • 初始容量的设定
  • 场景
    • 数组适合频繁读取的场景，链表适合频繁增删的场景
    • 另外由于CPU的缓存亲和性，在连续内存块读取时会预读后面的内容，所以在一些遍历访问下最好还是使用数组的方式更加有效。

追问：

JDK1.7 和 1.8 中 ArrayList 扩容逻辑有什么细微差别？为什么 LinkedList 不适合做队列首选，反而 ArrayDeque 更优？

  • 1.8中在无参构造下初始化为0，第一次扩容到10，而在1.7中是直接初始化容量为10的
    • 并且在源码中1.8是添加元素后检查容量ensureCapacityInternal，1.7是先检查再添加，主要是优化了空数组的内存占用
  • LinkedList是双向链表，要额外维护双向指针，更适合需要增删的特殊场景，而ArrayDeque是基于循环数组，内存紧凑，利于CPU亲和性，适合纯队列场景

HashMap JDK1.7 和 1.8 底层结构有什么变化？为什么要引入红黑树？

  • 核心变化（红黑树触发条件、哈希定位方式，扩容优化、尾插法）

    • 在1.8时，如果“链表长度> 8 && 数组长度≥64” 满足时，会升级成红黑树来增加查询和修改效率

    • 并且在计算哈希函数和计算哈希桶的时候也有一些差别（4次异或–>简化成高16位异或+低16位异或的两个扰动函数），哈希函数更加均匀，减少了哈希冲突

    • 并且哈希桶也由原先的indexFor变成了(n-1)&hash【n=table.length()】

      • 扩容时，每次扩容两倍，这样的好处是扩容后的hash定位一次就能定位，具体是判断key在hash的最高位是否为1判断是否在旧数组基础上偏移即可
        • 说明：如16（12）扩容到32，n-1的二进制变化是01111->11111，迁移后的结果只需要判断（32-1）最高为是否为1，来决定是否在原来基础上偏移（32>>1）即可

    • 另外在链表结构中由头插改成了尾插避免了在多线程扩容等条件下出现链表成环的问题。

    • 之所以引入红黑树就是为了解决链表长度过大时，查找/修改效率的问题，是一种内存换时间的策略。

      • 红黑树作为一个平衡二叉树，查询 插入删除性能是Ologn，比普通的二叉树更稳定

追问：

HashMap 红黑树转链表的阈值是 6 而非 8，为什么？JDK1.8 扩容时，红黑树节点是怎么处理的？

  • 这是考虑到缓存区间，避免红黑树和链表来回的转换
  • 处理思路：迁移时无非迁移到两个新桶的index（在原先位置的和需要迁移的），那么只需判断是否要保持红黑树结构，如果其中一组<6了就退化，挂到新桶，否则重构为红黑树挂到新桶

HashMap 为什么线程不安全？讲一个具体的线程安全问题场景。

  • 因为没有volatile、锁等方式保证变量的可见性，比如：一个线程A在修改变量的时候，另外一个线程B已经写入该变量，但是修改的变量没有同步到A，导致线程A仍然用旧值进行修改，导致线程A用脏读的数据进行操作，出现了线程安全的问题
  • 线程不安全核心原因：是没有并发的控制（volatile/锁）
  • 具体场景：JDK1.7中扩容头插法链表成环；多线程put时数据被覆盖（两个线程同时在同一个bucket中判空写入，后者覆盖掉了前者）
    • 原因：计算hash定位桶——插入节点不是原子操作，且没有锁保护

追问：

线程安全的 Map 有哪些？ConcurrentHashMap JDK1.7 和 1.8 的实现原理核心区别是什么？为什么 1.8 放弃分段锁，改用 CAS+synchronized？

  • 线程安全的 Map：Hashtable、Collections.synchronizedMap、ConcurrentHashMap；
  • CHM 1.7 vs 1.8：1.7 用分段锁（Segment），锁粒度是 “段”；1.8 改成 CAS+synchronized，锁粒度细化到 “桶节点”；
  • 弃分段锁原因：分段锁内存开销大，CAS 无锁能提升并发，而且 synchronized 已经做了优化（偏向锁 / 轻量级锁），性能足够。

模块 2：MySQL

InnoDB 为什么用 B+ 树做索引，而不用二叉树、红黑树？

  • 是这样的，B+树适合磁盘存储，因为他的节点都存放在叶子节点，而非叶子节点只需要存储主键/索引以及指针的值，这样非叶子节点在一个数据页内就能存放更多的索引数据了，高度也就相对于红黑树等二叉树更稳定，这会减少I/O开销
  • 范围查询的优势，红黑树适合频繁插入/删除，而B+树叶子节点之间有链表相连，在innoDB中非常适合范围查询的需求。

追问：

InnoDB 聚簇索引和非聚簇索引的区别是什么？非聚簇索引查询的回表问题怎么解决？

  • 聚簇索引（主键索引）：叶子节点存整行数据；非聚簇索引（二级索引）：叶子节点只存主键值；
  • 回表解决：用覆盖索引（查询字段都在索引里，不用回表）

事务 ACID 分别是什么，InnoDB 是怎么实现持久性和原子性的？

  • A是原子性，操作不可分割；C是一致性，每个事务操作前后要保证状态一致；I是隔离性，每个事务之间操作不相互影响；D是持久性，事务操作的结果要持久保存。
  • 【事务保证原子性和持久性核心是有一个MVCC机制，核心就是通过undo log的回滚日志来记录每个事务操作后的版本号，当有新事务进行操作的时候，就会判断当前事务id是否和当前正在执行事务的id冲突，从而决定使用哪个版本号的内容，从而保证多个事务完成后能有一个完整的版本链，从而保证了每个事务的原子性和持久性。——MVCC机制主要关注的是隔离性
  • 原子性：主要依赖undo log（记录修改前的状态，失败后事务整体可以回滚）
  • 持久性：是依赖redo log + WAL机制（日志先行，提交时保证redo log先罗盘，再刷新数据页，保证宕机后可以恢复）

追问：

redo log 的刷盘策略有哪些？InnoDB 两阶段提交是为了解决什么问题？MVCC 结合隐藏字段、undo log 版本链、Read View 的具体实现逻辑是什么？

  • 两阶段提交：解决 binlog 与 redo log 一致性问题，避免宕机数据不一致（基于XID的协商方案，在不一致的 情况下后续判断是最终进行事务提交还是回滚）
  • MVCC 实现：隐藏字段（事务 ID、回滚指针）+ undo log 版本链 + Read View；RC 每次查询生成 Read View，RR 事务首次查询生成；

说下脏读、不可重复读、幻读的区别，MySQL 默认隔离级别是什么？

  • 说明
    • 脏读就是一个事务读取到了另外一个未提交事务写操作后的结果
    • 不可重复读是一个事务两次读，由于在事务过程中其他写事务的提交（更新操作），导致读到的结果不一致
    • 幻读是一个事务两次进行查询，由于在事务过程中其他写事务的提交（插入/删除操作），得到的数据行行数不一致，莫名其妙几行数据发生了变化，好像是出现了幻觉，所以我们叫幻读
  • MySQL默认隔离级别是可重复读RR，mysql设置这个隔离级别的原因出发点是为了解决binlog早期的statement的兼容性，避免主从同步的不一致

例子复现：必须要用间隙/临键锁保证事务B的插入操作在事务A提交后执行（注 insert等修改操作用当前读，而select要用快照读，这导致幻读问题无法完美解决！）

追问：

InnoDB 的 RR 隔离级别怎么 “部分解决” 幻读？RC 和 RR 下 MVCC 的 Read View 生成时机有什么不同？

  • RR 解决幻读：快照读靠 MVCC 避免，当前读（insert/update）靠间隙锁 / 临键锁避免。

模块 3：Redis

Redis 是单线程，为什么性能还这么高？

  • 首先Redis的单线程仅针对命令执行，实际上6.0之后有多线程的优化，主要是针对网络I/O的多线程来解决性能瓶颈
  • 【Redis性能高的关键原因：内存操作，没有磁盘IO的瓶颈，并且单线程避免了线程的上下文切换/锁的竞争开销
  • Redis使用了多路IO复用的技术，相比于其他阻塞IO和非阻塞IO模型，Redis只需要一个线程监听多个连接的epoll指令，然后调用内核完成后，向客户端返回结果，避免了非阻塞IO中Redis频繁发送read指令导致CPU空转的问题，多路复用的意思就是一个线程可以复用多个连接的指令，提高了处理效率。
    • Redis使用epoll 的 ET（边缘触发）模式（而非 LT 水平触发）
  • Redis还提供了高效的数据结构，如Hash、跳表等，在删除插入时提供了高效的时间复杂度，还有一些高效的编码思想，如String的SDS中针对int的编码，还有对列表的ziplist压缩存储的思想

参考图

追问：

Redis 6.0 引入的多线程是做什么的？为什么不把命令执行也改成多线程？epoll 的 LT 和 ET 模式区别是什么，Redis 为什么选 ET？

  • Redis6.0 多线程：只处理网络 IO 读写 / 解析，不执行命令，不做多线程命令：避免锁竞争、上下文切换，保持单线程简单高效
  • epoll LT/ET 补全：LT 水平触发（没处理完一直通知），ET 边缘触发（只通知一次）；Redis 用 ET 减少事件触发次数，更高性能

RDB 和 AOF 两种持久化的区别，你们项目里一般怎么配？

  • RDB保存的是Redis的数据快照，而AOF是一个记录数据库每一个操作增量记录。

    • RDB有一个写时复制COW策略（fork子进程做快照，共享同一份页表，加快fork效率，COW策略也保证父进程可以正常处理请求），缺陷是快照间隔内的数据丢失
    • AOF的话内存占用少，但是恢复需要时间，并且根据不同的AOF写回策略可能存在数据丢失的风险（always/everysec/no，默认 everysec）。

  • 我们项目里一般会用Redis6.0更新后的一个方式，RDB+AOF都使用的策略，首先用RDB保存当前的数据快照，AOF保存在数据快照保存过程中的指令，并且用立即写入系统缓存，调用fsync的always写回策略尽可能保障数据不丢失，最终在AOF文件中前半部分为RDB快照，后半部分是AOF日志，同时兼顾了恢复性能和数据完整性

追问：

Redis fork 子进程做 RDB 时，COW 的具体过程是什么？为什么 fork 会有短暂阻塞？AOF 重写的触发条件和原理是什么，和 RDB 的 fork 有什么区别？

  • RDB fork COW 补全：父进程写时复制页，fork 阻塞是因为拷贝页表
  • AOF 重写：触发（体积 / 比例阈值），原理：fork 子进程读取当前内存数据重写新 AOF，避免历史冗余

Redis 有哪些数据淘汰策略？实际业务用哪个最多？

分成两种，一种是带过期时间的缓存，还有一种是不带的。

  • 带过期时间的，一般是有LRU淘汰最近未使用的，LFU淘汰最不经常访问的，还有ttl固定的跳数，random随机时间
    • volatile-lru/volatile-lfu/volatile-ttl/volatile-random
  • 不带过期时间的，一般是有LRU淘汰最近未使用的，LFU淘汰最不经常访问的，
    • allkeys-lru/allkeys-lfu/allkeys-random/
  • 另外还有一种还有noeviction内存满拒绝写入的策略
  • 实际业务中我们根据业务数据访问是否均匀采用（allkeys-lru/volatile-lru），LFU可以 解决 LRU 的缓存抖动问题

追问：

Redis 的 LRU 是 “近似 LRU” 而非严格 LRU，为什么这么设计？近似 LRU 的实现原理是什么？

  • 近似 LRU 补全：省内存、效率高，采样实现足够接近严格 LRU（思路是通过24位时间戳+随机采样替代双向链表高并发瓶颈，尽可能通过局部最优逼近全局最优）

第二阶段：二面（源码 + 场景 + 实战，偏深度 + 工程能力）

二面对标大厂 “核心岗” 要求，侧重源码理解、业务场景落地、问题排查，每题需结合原理 + 实战回答：

模块 1：Java 进阶 + 源码

结合源码说下 ConcurrentHashMap JDK1.8 的 put 流程，重点说明 CAS+synchronized 的使用时机和目的；

  • 因为并发集合类为了避免null在并发时的潜在问题，put时首先会判断key和value是否为空，为空直接NPE（get方法同理）
    • 避免null是为了消除歧义（get返回null只能说明key不存在），并简化实现（不需要二次判断key是否存在等）
  • 接下来就是定位到桶的下标找key了，有一个spread 方法，避免高位未参与哈希
    • 桶为空：CAS 无锁插入，失败则自旋重试。
    • 否则，对桶的首节点加synchronized进行阻塞（锁的细粒度），再判断根据链表 / 红黑树，判断key是否替换还是新增node即可
  • CAS更新size，到达阈值则红黑树转换/扩容（LongAdder思想，每个线程在put等操作的时候会记录自己对应下标的CounterCell对象，最终统一）

追问：

ConcurrentHashMap的渐进式扩容？

① 渐进式扩容：

  • 会维护一个transferIndex的volatile 变量，来的线程会死循环CAS争夺下标（线程A争夺设置为16，处理3216，线程B争夺设置为0，处理150），如果下标分配完了，就不搬迁，否则就搬运对应的node
    • 【如数组初始长度16，扩容后长度32，3216的node由线程A搬迁，150的node由线程B搬迁。】；
  • 迁移时用 ForwardingNode 标记桶，新写入走扩容逻辑、读请求转发到新表；
维度 ConcurrentHashMap (JDK1.8) Redis HashMap（单机 / Cluster）
扩容触发 put 时检测容量阈值，主动触发 内存满 / 负载因子达标，或手动 resize；Cluster 需槽位均衡
迁移单位 数组的单个 “桶（bucket）” 单机：哈希表槽位（dictEntry）；Cluster：16384 个哈希槽
扩容标识 迁移中桶标记为 ForwardingNode，请求转发 单机：rehashidx 标记进度；Cluster：槽位状态（迁移中 / 完成）
并发控制 桶级别 synchronized + CAS 保证线程安全 单线程执行迁移，无锁；Cluster 靠 Gossip 协议同步槽位
扩容范围 单实例内的数组扩容 单机：单哈希表 rehash；Cluster：跨节点迁移槽位（分布式）

ConcurrentHashMap 的 size() 方法在 JDK1.8 如何保证准确性？为什么 1.8 用 synchronized 而非 ReentrantLock？

size () 准确性：先累加 baseCount + CounterCell 数组，并发修改时加锁重算；

1.8 用 synchronized 而非 ReentrantLock：synchronized 已优化（偏向 / 轻量级锁），减少 Lock 对象内存开销，且锁粒度到桶节点，效率足够；

说下 ThreadLocal 的实现原理，为什么会出现内存泄漏？怎么避免？

  • 每个Thread线程会维护一个ThreadLocalMap，key是当前线程的ThreadLocal对象，value则是业务数据
  • 内存泄露原因：key是弱引用，但value仍被Thread强引用，而Thread在线程池中长期存活，无法回收value
  • 为了避免内存泄露，需要手动执行threadLocal.remove()，以及考虑线程池场景下的特殊处理（线程复用需手动清理）。

追问：

ThreadLocal 内存泄漏在线程池场景下会更严重，为什么？InheritableThreadLocal 解决了什么问题，有什么坑？

① 线程池场景更严重：线程池线程复用，Thread 对象长期存活，ThreadLocalMap 的 value 强引用无法回收，累计泄漏；

② InheritableThreadLocal：解决子线程无法继承父线程 ThreadLocal 值的问题；

坑：线程池复用线程时，子线程继承的是首次创建时的父线程值，非当前父线程值，需手动清理；

谈谈你对 Java 内存模型（JMM）的理解，volatile 是怎么保证可见性和禁止指令重排序的？

  • JMM：Java 内存模型，定义主内存 & 线程工作内存，解决可见性、原子性、有序性。
  • volatile 作用：
    1. 可见性：写立即刷回主内存，读强制从主内存拉。
    2. 禁止指令重排：靠内存屏障。
  • 注意：volatile 不保证原子性（i++ 不安全）。

追问：

volatile 能保证 i++ 的原子性吗？为什么？synchronized 和 volatile 的核心区别，分别适用于什么场景？

① volatile 不能保证 i++ 原子性：i++ 分读 - 改 - 写三步，volatile 仅保证可见性 / 有序性，无法保证三步原子执行；

② 核心区别：synchronized 保证原子性 + 可见性 + 有序性（重量级、阻塞），适用于多线程修改共享变量（如 i++、复杂业务逻辑）；volatile 仅保证可见性 + 有序性（轻量级、非阻塞），适用于单线程写、多线程读（如状态标记位）。

模块 2：MySQL 实战 + 优化

实战场景：一个电商订单表（order_id 主键，user_id 普通索引，create_time 无索引），执行 select * from order where user_id = 10086 and create_time > '2026-01-01' 时慢查询，怎么优化？请从索引、SQL、执行计划三个维度分析；

  • create_time没有命中索引，保证最左匹配原则下，可以考虑使用联合索引(user_id,create_time
    • 这里可以有一层索引下推机制，在存储引擎层提前先过滤掉数据
  • SQL优化：SELECT *是不建议的，只需要返回需要的字段，如果刚好只有索引中的字段，满足覆盖索引条件，不需要回表了，减少IO
  • 分区表优化（按 create_time 分区，减少扫描数据量）
  • 执行计划 EXPLAIN 看 type 从 ALL → range/ref，Extra 出现 Using index、Using where。

追问：

若 user_id 基数极低（如 90% 订单的 user_id 都是 10086），联合索引还生效吗？怎么进一步优化？

user_id 基数极低（90% 都是 10086）时联合索引失效（MySQL 优化器认为全表扫描更高效，此时加了索引反而变慢了）；进一步优化（核心就是减少扫描/索引成本）：

强制走索引（临时兜底）

SQL 加 force index(联合索引名)，强制优化器走索引；若只查少量数据（如近 7 天订单），搭配 limit 限制行数，降低索引遍历成本。

分区表（核心优化）

按 create_time 分区（如按天 / 月），即使全表扫也只扫目标分区，扫描数据量骤减，比全表快 10 倍以上。

覆盖索引（消除回表）

若查询字段（如 order_id、create_time）都在 (user_id, create_time) 联合索引中，优化器会优先走索引（无需回表，成本低于全表）。

数据拆分（长期方案）

把高频 user_id=10086 的订单单独拆表，其余数据放主表，查询时直接定位到拆分表，彻底避免扫大量无效数据。

说下 MySQL 索引失效的常见场景（至少 5 个），并说明如何避免；

  • 不满足最左匹配原则
    • 但在特定的场景下（覆盖索引）还是会用到索引的（type=index）
  • 索引中使用函数/运算表达式
    • 隐式类型转换（如int 传字符串函数，非常隐蔽！）
  • 使用like %?这样的表达式
  • 不同的参数导致索引的失效：因为需要命中索引，这就导致在一些复杂查询（联表 子查询 回表场景）可能选择全表查询更好
    • 如商品从订单表查询，传入的商品id是热门商品且占据了80%的销量，那么这个查询很有可能是全表查询，相反，如果是冷门商品，可能就是走索引查询，同上一节定时任务的场景。
  • or 连接非索引列
  • 不同字段比较
  • order by后面不是主键，或者不是覆盖索引

追问：

为什么 “隐式类型转换” 会导致索引失效？举个具体例子说明（如字段 id int，查询 where id = '123'）。

隐式类型转换失效原因：MySQL 会对索引字段做函数转换（id int 查 id=’123’ → cast (id as char)=’123’），索引字段参与函数运算导致失效；例子：id int 类型，查询 where id=’123’ 索引失效

分库分表的核心痛点是什么？你会怎么选择分片键？分表后如何解决跨分片查询问题？

  1. 首先要分析业务要求：分析数据量和增长的趋势，评估分库分表的必要性（可以有适度预判，但不要过度设计）
  2. 技术选型：设计分库分表方案
    1. 分片键：确定业务意义明确，查询高频，数据分布均匀的字段（user_id、order_id）
    2. 分库分表方式：range、hash、一致性hash、或结合上述方案，确定分片数量（要考虑未来拓展性）
    3. 确定路由逻辑：设置在Client端、Proxy层（ShardingSphere）还是自研中间件
  3. 数据迁移考虑：分批导入到新的分库分表结构中，可通过ETL工具或脚本完成
  4. 部分SQL语句和业务逻辑是否需要调整：如业务的链表查询，或者本地事务修改为分布式事务，要列举出来一一修改

痛点：

  • 事务问题：分库后单机事务必须转变为分布式事务，大部分分布式事务只能保证最终一致性，业务上会引发数据不一致的场景

  • 连表join问题：跨库后就无法使用join了

    • 解决方案1：在业务代码中关联，先差一个表的结果，再通过该结果差另外一个表，通过代码关联
    • 解决方案2：适当引入冗余字段，如以前的表只有一个关联ID，但业务经常需要返回Name或其他字段，这时就可以把对应字段冗余到当前表，去除关联操作
    • 解决方案3：基于宽表的形式查询，如数据全量存储在ES中，ES来查询问题

  • 全局ID的唯一性：原先的自增ID通过分库分表后，自增ID可能会出现重复主键，此时需要通过雪花算法或其他唯一ID发号器生成唯一ID

  • 排序问题：原先的order by即可，分库分表后无法排序，需要在中间件进行汇总排序，或者在业务代码中排序，或者通过ES存储全量数据排序访问

  • count的问题：和排序问题类似，单表可以count，分库分表后无法支持，只能多标count在业务中累加，或者在中间层维护，要么就是上ES维护

  • 深分页问题

    • 记录id：每次分页查询返回最大id，后续查询带上该id，通过id>maxid过滤即可

      • 场景：仅适合连续查询情况，如滚动翻页

    • 子查询/JOIN过滤

      • SELECT * FROM mianshiya WHERE name = 'yupi' ORDER BY id LIMIT 99999990, 10; ；可以优化如下，避免直接扫描主键索引，而是先通过二级索引过滤掉数据，通过子查询/JOIN再通过少量的id去查询，数据量更小，速度更快：

      •   SELECT * FROM mianshiya 
  WHERE name = 'yupi' 
    AND id >= (
      SELECT id FROM mianshiya 
      WHERE name = 'yupi' 
      ORDER BY id 
      LIMIT 99999990, 1
    )
  ORDER BY id 
  LIMIT 10;

    • elasticsearch：搜索引擎解决，但es本身也有深度分页问题

合并后的结果不是95和90，没进行优化的话只能limit 0,3 然后取最后两个

追问：

分库分表后，订单表（按 user_id 分片）和订单详情表（按 order_id 分片）需要 join，怎么解决？分布式事务怎么保证？

① 双分片键冗余；

② 先查订单表拿 order_id 再查详情表；

③ 引入 ES 关联；

分布式事务：① 最终一致性（消息队列 + 补偿）；② TCC/Seata AT

模块 3：Redis 实战 + 问题排查

实战场景：Redis 作为缓存，出现 “缓存穿透、缓存击穿、缓存雪崩”，分别是什么问题？各自的解决方案是什么？请结合代码 / 配置举例；

  • 缓存穿透：访问不存在key，导致直接请求数据库，增加数据库压力
    • 缓存null值
    • 布隆过滤器等过滤恶意请求（误判率）
  • 缓存击穿：key到期了，然后有多个请求打进来，打到数据库
    • 使用分布式锁机制保证只有一个线程访问并设置缓存（缓存击穿的分布式锁代码、布隆过滤器配置）
    • 加入多级缓存，如caffine本地缓存
  • 缓存雪崩：多个key到期，连带着大量请求打进数据库，数据库压力陡增
    • 随机过期时间
    • 分布式锁机制保证只有一个线程访问并设置缓存
    • 加入多级缓存，如caffine本地缓存
    • 集群、限流降级。（限制数据库流量）

追问：

布隆过滤器的误判率怎么降低？缓存击穿的分布式锁若超时释放，会导致多个线程更新缓存，怎么解决？

布隆过滤器降低误判率：增大位图大小、增加哈希函数数量；

缓存击穿锁超时：① 锁续约（看门狗）；② 缓存过期时间＞锁超时；③ 双删策略

Redis 集群（主从 + 哨兵 / Cluster）的核心架构是什么？哨兵的故障转移流程是怎样的？

  • 架构：一主多从，一般主节点负责写入，从节点负责读取，哨兵负责监控主节点的状态，如果无法处理就即使选举主节点，cluster则是按照16384方槽进行分布，每个节点对应一段槽的处理，缓解多个key 的压力

  • 哨兵转移流程：

    ① 哨兵检测主节点不可达→主观下线；

    ② 多数哨兵确认→客观下线；

    ③ 选举 leader 哨兵；

    ④ leader 哨兵选举新主（RAFT “候选人请求投票→多数节点同意则成为 leader”；）；

    ⑤ 通知从节点复制新主、通知客户端；

追问：

Redis Cluster 的 “脑裂” 问题是什么？怎么解决？哨兵的 quorum 参数作用是什么？

  • Redis Cluster 脑裂：主节点隔离后从节点成新主，原主恢复出现双主；解决：
    • 设置 min-replicas-to-write（主库至少有 N 个从库同步成功才接受写）+ min-replicas-max-lag（同步延迟阈值）；
  • 哨兵 quorum：确认主节点客观下线的最少哨兵数量

线上 Redis 出现慢查询，你会怎么排查？从 “命令层面、配置层面、架构层面” 分别说明。

  • ① 命令层面：用 slowlog get 查看慢查询、redis-cli --bigkeys 检测 big key、分析耗时命令（如 hgetall、keys *）；
  • ② 配置层面：调整 slowlog-log-slower-than（默认 10ms）、检查 maxmemory-policy（避免频繁淘汰）、优化持久化（AOF 重写 / RDB 避免高峰执行）；
  • ③ 架构层面：热点 key 拆分、集群扩容、主从分离；

注意考虑CPU 使用率、网络延迟、持久化阻塞导致的慢查询。

追问：

Redis 删除 big key 为什么会阻塞？怎么异步删除 big key（unlink 命令）？

big key 删除阻塞：单线程遍历删除耗时久；异步删除：用 unlink 命令（后台线程删除）

模块 4：项目 + 工程能力（校招核心）

你在项目中用 Redis 做过什么？如果遇到 Redis 主从同步延迟，导致从库读不到最新数据，怎么解决？

  • 主从同步延迟无具体解决方案：

    ① 热点 key 读主库；

    ③ 增大 repl-backlog-size（同步积压缓冲区）；

    ④ 避免主库大事务；

分布式锁未提实现细节（如 Redlock、过期时间 + 续约）。

追问：

Redis 分布式锁怎么避免死锁？读主库解决同步延迟会带来什么问题（主库压力）？

分布式锁避免死锁：① 设过期时间；② 加锁存唯一标识；③ 看门狗续约；读主库问题：主库压力增大，可加本地缓存 + 限流

你项目中有没有遇到过数据库死锁？怎么排查的？死锁的产生条件是什么？

  • 有，会遇到多个事务相互等待的情况

    • 排查：show engine innodb status\G 看死锁日志。
    • 场景（事务 1 排着队等事务 2 释放 X 锁，而事务 2 又需要 S 锁，申请 S 锁的时候发现等锁的队伍前还有个 X 锁，这样一来就形成了相互依赖。）
      • 事务1请求X锁：delete from test where a = 2，等待释放索引a的X锁（index a ... trx id 2A8BD lock_mode X waiting）
      • 事务2持有X锁：insert into test (id,a) values (10,2)
      • 事务2请求S锁：由于 a 建立了唯一索引，因此需要 S 锁来进行重复 key 判断，因此日志显示等待释放X锁：(index a ... trx id 2A8BC lock mode S waiting
    • 解决：统一事务加锁顺序、小事务、避免同时锁多张表。

  • 死锁 4 条件（必须背全）：

    1. 互斥条件：每个资源只能被一个线程占用。
    2. 占有和等待：线程在持有至少一个资源的同时，等待获取其他资源。
    3. 不可抢占：线程所获得的资源在未使用完毕之前不能被其他线程抢占。
    4. 循环等待：多个线程形成一种头尾相接的循环等待资源关系。

追问：

怎么从业务层面避免数据库死锁？举个实际场景的优化例子（如订单表和库存表的事务顺序）。

  • 避免互斥
    • 如用版本号的乐观锁代替互斥锁
  • 避免占有和等待
    • 尽量减少锁的范围：将锁的粒度尽可能缩小，减少持有锁的时间。可以通过拆分锁或使用更细粒度的锁来实现。
    • 使用尝试锁机制：使用 ReentrantLock 的 tryLock 方法，尝试在一段时间内获取锁，如果无法获取，则可以选择放弃或采取其他措施，避免死锁。
    • 避免嵌套锁：尽量避免在一个锁的代码块中再次尝试获取另一个锁。
  • 避免不可抢占
    • 设置超时等待时间：为锁操作设置超时，防止线程无限期地等待锁。
  • 避免循环等待：
    • 按序申请资源：确保所有线程在获取多个锁时，按照相同的顺序获取锁。

谈谈你对 “高可用” 的理解，结合你学的技术，怎么设计一个高可用的电商订单系统（核心模块：下单、支付、库存）？

高可用理解错误：高可用是系统在故障下仍能提供服务（可用性 = MTBF/(MTBF+MTTR)），而非 “每次请求有回应”；

CAP 中可用性与一致性是权衡维度，但高可用核心是 “服务不中断”；

设计思路未体现高可用核心：

  • 服务层面：集群部署，加熔断、降级、限流（避免服务雪崩）；
  • 数据层面：库存预扣减、保证分布式事务最终一致性、实现接口幂等性；
  • 监控层面：链路追踪，针对库存不足、支付超时等场景设置告警；
  • 容错层面：支付结果通知失败重试，库存服务故障时临时降级关闭下单。

追问：
订单系统的幂等性怎么保证（如用户重复支付）？库存预扣减场景下，怎么避免超卖？

  • 订单幂等性（避免重复支付）：① 用唯一订单号 / 支付流水号校验；② 分布式锁 + 订单状态机（防止重复修改）。
  • 库存预扣减防超卖：① 分布式锁扣减；② 乐观锁（SQL 加 where stock>0）；③ 预扣减后入消息队列异步确认。
