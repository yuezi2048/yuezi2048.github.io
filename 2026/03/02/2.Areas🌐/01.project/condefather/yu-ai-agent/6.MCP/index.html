MCP

概念说明

**MCP（Model Context Protocol，模型上下文协议）**是一个开放的标准，旨在让AI能与外部系统交互标准化，实现AI与现有系统集成的效果。

根据官方定义，可以将MCP想象成AI应用的USB接口。就像USB为设备连接各种外设和配件提供了标准化方式一样，MCP为AI模型连接不同的数据源和工具提供了标准化的方法。

注意MCP是一个协议，类比HTTP的get/post请求方式的标准、

这样一来只需要官方支持该协议，我们就很方便调用，标准造就生态，类比NPM包、Maven、Docker镜像

好处：

  • 增强AI能力
  • 统一标准，降低使用和理解成本
    • 进而打造服务生态，造福开发者

实际上按照官方的说法，MCP总共有6大核心概念。简单了解即可，除了Tools工具之外的其他概念都不是很实用，如果要开发MCP服务，只需要关注前三个概念，不过大多数客户端还是目前只支持Tools工具。
1.Resources资源:让服务端向客户端提供各种数据，比如文本、文件、数据库记录、API响应等，客户端可以决定什么时候使用这些资源。使AI能够访问最新信息和外部知识，为模型提供更丰富的上下文。
2.Prompts提示词:服务端可以定义可复用的提示词模板和工作流，供客户端和用户直接使用。它的作用是标准化常见的AI交互模式，比如能作为UI元素(如斜杠命令、快捷操作)呈现给用户，从而简化用户与LLM的交互过程。
3.Tools工具:MCP中最实用的特性，服务端可以提供给客户端可调用的函数，使AI模型能够执行计算、查询信息或者和外部系统交互，极大扩展了AI的能力范围。
4.Sampling采样:允许服务端通过客户端向大模型发送生成内容的请求(反向请求)。使MCP服务能够实现复杂的智能代理行为，同时保持用户对整个过程的控制和数据隐私保护。
5.Roots根目录:MCP协议的安全机制，定义了服务器可以访问的文件系统位置，限制访问范围，为MCP服务提供安全边界，防止恶意文件访问。
6.Transports传输:定义客户端和服务器间的通信方式，包括Stdio(本地进程间通信)和SSE(网络实时通信)，确保不同环境下的可靠信息交换。

MCP架构

宏观上来说，MCP就是一个客户端的架构，MCP客户端可以连接多个服务器。

注：客户端是Calude Desktop、IDE、AI工具或者我们部署在服务器上的项目

Java SDK

而如果在程序中使用MCP，我们最优先考虑的是SDK（Java SDK），而在官方文档中，MCP SDK的架构是三层：

  • 客户端/服务器层：MCP Client处理客户端，MCP Server处理服务端协议，两者基于MCP Session进行通信管理
  • 会话层(McpSession)：通过DefaultMcpSession 管理通信模式和状态。
  • 传输层(McpTransport)：处理JSON-RPC消息序列化和反序列化，支持多种传输实现，比如Stdio 标准IO流传输、HTTP SSE远程传输。

客户端和服务端连接的过程，类似于三次握手，建立连接后即可通信

客户端

MCP Client负责和服务器建立连接并通信：匹配服务器的协议版本、确认可用功能、进行数据传输和JSON-RPC交互。同时也可以发现并使用工具、管理资源、和提示词系统交互，以及额外特性：如根管理、采样控制，以及同步或异步操作等。

为适应不同场景，提供多种数据传输方式（本地 / 远程调用，具体传输方式示意如图）：

  • Stdio标准输入/输出:适用于本地调用

  • 基于Java HttpClient和WebFlux的SSE传输：适用于远程调用

服务端

服务端主要是给客户端提供工具、资源、功能支持。

服务端主要负责处理客户端请求（解析协议、提供工具、资源管理、处理交互信息），同时本身也能记录日志、发送通知，可以对多个客户端同时连接高效通信协作。

和客户端一样，它也由多种方式进行数据传输（Stdio标准输入/输出、基于Servlet/WebFlux/WebMVC的SSE传输），满足不同应用场景。

这种设计使得客户端和服务端完全解耦，任何语言开发的客户端都可以调用MCP服务。两种服务端通信模式如图

MCP最佳实践

  • 慎用MCP：MCP不是银弹，其本质就是工具调用，只不过统一了标准、更容易共享而已。如果我们自己开发一些不需要共享的工具，完全没必要使用MCP，可以节约开发和部署成本。我个人的建议是能不用就不用，先开发工具调用，之后需要提供MCP服务时再将工具调用转换成MCP服务即可。
  • 传输模式选择：Stdio模式作为客户端子进程运行，无需网络传输，因此安全性和性能都更高，更适合小型项目;SSE模式适合作为独立服务部署，可以被多客户端共享调用，更适合模块化的中大型项目团队。
  • 明确服务：设计MCP服务时，要合理划分工具和资源，并且利用@Too1、@ToolParam注解尽可能清楚地描述工具的作用，便于AI理解和选择调用。
  • 注意容错：和工具开发一样，要注意MCP服务的容错性和健壮性，捕获并处理所有可能的异常，并且返回友好的错误信息，便于客户端处理。
  • 性能优化：MCP服务端要防止单次执行时间过长，可以采用异步模式来处理耗时操作，或者设置超时时间。客户端也要合理设置超时时间，防止因为MCP调用时间过长而导致AI应用阻塞。
  • 跨平台兼容性：开发MCP 服务时，应该考虑在Windows、Linux和macOS等不同操作系统上的兼容性。特别是使用 stdio传输模式时，注意路径分隔符差异、进程启动方式和环境变量设置。比如客户端在Windows系统中使用命令时需要额外添加.cmd 后缀。

MCP部署方案

MCP分为两种传输模式：stdio（本地）和SSE（远程），对应的就是本地部署和远程部署。

本地部署：将MCP Server打包后，上传到MCP Client可访问的路径下，编写MCP配置即可启动，只适合小型项目

缺点就是每有一个MCP服务就要单独部署，既然如此，为什么不直接在后端中开发工具调用，何必要专门开一个项目开发MCP？

远程部署：同后端web项目部署，参考快速上线项目的方法汇总

值得说明的是，MCP一般是职责单一的项目，因此很适合部署到Serverless平台，按量使用付费，减少运维难度和成本，百炼部署MCP指南

同时可以将MCP服务提交到第三方市场，让其他人也方便使用

扩展

MCP安全机制

安全风险：隐私泄露、服务器权限泄露、恶意脚本攻击等。

为什么？

  • 信息不对称：用户不关注内部实现，而AI是能看到完整工具描述，那么是可以通过描述来恶意控制AI行为的。
  • 上下文混合与隔离不足：所有MCP描述会加载到同一上下文中，那么一个恶意的MCP工具就可能会影响其他MCP
  • 大模型本身的安全意识不足：大模型以精确为主导，而缺乏有效的安全指令
  • MCP缺乏严格的版本控制和更新通知机制：这意味着用户无法感知内部是否添加了恶意代码
  • 敏感操作缺乏严格的多重授权机制，难以控制工具的行动范围

举例：

  • 潜伏：表面上只有一个技巧提示的功能，但有一个隐藏的不被用户可见的指令
  • 注入指令：MCP会将恶意指令加入到工具描述中，这样用户看到私信，就会将内容自动邮件发送给攻击者
    • 触发攻击：用户触发了特定的行为，AI触发指令发送给攻击者，用户对该过程完全不知情
    • 数据窃取：攻击者得到了敏感信息

解决思路：

  • 沙箱环境：隔离环境中使用，限制网络和文件权限
  • 检查参数和行为：检查源码的所有参数，特别注意网络请求和文件操作
  • 优先使用可信来源：仅安装知名企业的服务，避免未知来源第三方工具，优先选文档详细、大厂维护、知名度高的产品

改进思路：优化MCP的功能描述（什么时候调用）和执行指令（实际的prompt）的区分，建立最小权限原则、安全检测、MCP生态规范

而对于SSE：网上几乎没有解决方案和实践，但是我们可以思考
