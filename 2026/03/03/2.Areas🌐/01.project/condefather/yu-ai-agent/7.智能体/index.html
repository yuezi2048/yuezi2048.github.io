智能体

概念

智能体（Agent）是能自主决策、指定计划，从而实现特定目标的AI系统，集成 记忆+知识库+工具 –> 记忆能力、决策能力、执行能力，就好像人一样。

Agent和一般的大模型区别：

  • 感知环境：可以多模态地理解用户需求和环境状态
  • 自主规划任务步骤：将复杂任务分解为可执行的子任务，并设计执行顺序
  • 主动调用工具完成任务：根据需要选择并使用各种外部工具和API，扩展自身能力边界
  • 进行多步推理：通过思维链(Chain of Thought)逐步分析问题并推导解决方案
  • 持续学习和记忆过去的交互：保持上下文连贯性，利用历史交互改进决策
  • 根据环境反馈调整行为：根据执行结果动态调整策略，实现闭环优化

分类：

  • 反应式智能体：简单的聊天机器人，没有真正的规划
  • 有限规划智能体：简单分为多步执行，但执行路径一般是预设的（只能完成简单的事情，如联网搜索，调用知识库和工具），如24年流行的ChatGPT+Plugins
  • 自主规划（目标导向）智能体：能真正地根据任务自主分解任务、制定计划、选择工具一步步执行，直到完成任务，如25年流行的Manus项目，亮点就是自主能力
    • 讲到Manus，就必须得说到小团队复刻的OpenManus，实现“思考——行动——观察”循环的工作模式，直至完成工作目标

注：自主规划智能体是智能体发展方向，但并不是所有场景都需要自主规划能力，有时候限制自主性可以提高效率、安全

技术

COT

COT(Chain of Thought) 思维链，就是处理复杂推理类问题时，先按步骤思考再执行效果会更好（也可以展示推理过程，便于理解和优化AI）

实现原理：

  • 输入Prompt后，给模型增加额外的提示和引导【让我们一步一步思考这个问题】，从而让模型以推理的方式生成回答。
  • 通过few shot的prompt优化技术给模型提供包含思维链的示例问题和答案，让模型学习如何构建思维链

OpenManus早期的提示词：

可以看到，就会输出我们想要的思考过程和结果，得到更准确的答案。

Agent Loop

Agent Loop执行循环是最核心的机制：在用户没有输入的情况下，能自主推理和工具调用

传统模型问题：用户问完问题后，AI回复完就结束了

Agent Loop：用户问完问题，AI要自主规划后续动作（调用工具、处理结果、继续推理），形成自主执行的闭环，直到任务完成（或超过预设最大步长）

ReAct模式

ReAct（Resoning + Acting）意思是：推理+执行，模仿人类的“思考-行动-观察”的循环，目的是通过交互式的决策来解决任务，是目前最常用的Agent工作模式之一

思路：

  • Reson 推理：拆分多个子任务，明确执行步骤
  • Act 行动：调用外部工具执行动作
  • Observe 观察：得到工具结果，反馈给智能体进行下一步决策
  • 循环迭代：不断循环迭代，直到任务完成或终止条件

技术支持

  • AI大模型：提供了思考、推理和决策的核心能力，越强的AI大模型通常执行任务的效果越好。
  • 记忆系统：智能体需要记忆系统来存储对话历史、中间结果和执行状态，这样它才能够进行连续对话并根据历史对话分析接下来的工作步骤。之前我们学习过如何使用SpringAI的ChatMemory实现对话记忆。
  • 知识库：尽管大语言模型拥有丰富的参数知识，但针对特定领域的专业知识往往需要额外的知识库支持。之前我们学习过，通过RAG检索增强生成+向量数据库等技术，智能体可以检索并利用专业知识回答问题。
  • 工具调用：工具是扩展智能体能力边界的关键，智能体通过工具调用可以访问搜索引擎、数据库、API接口等外部服务，极大地增强了其解决实际问题的能力。当然，MCP也可以算是工具调用的一种。

这里分享一种特殊的工具调用–Compute Use，它允许智能体直接与计算环境交互，比如执行代码、操作文件系统等。目前Claude等平台已提供此类能力:

  • Claude 支持 Compute Use: https://docs.anthropic.com/en/docs/agents-and-tools/computer-use
  • Compute Use 实现示例: https://github.com/anthropics/anthropic-quickstarts/tree/main/computer-use-demo
  • Compute Use 开源实现: https://github.com/e2b-dev/open-computer-use、 https://github.com/showlab/computer_use_ootb

注意：这些操作会在我们自己的系统上执行，建议在虚拟环境中运行，防止意外操作影响实际系统。

综合上面4类技术，并且结合CoT、Agent Loop、ReAct等机制(可以总称为“规划执行机制”)，我们就可以构建完整、有自主规划能力的智能体。

OpenManus实现原理

OpenManus

整体架构

agent

OpenManus的代理架构主要包含以下几层:

  • BaseAgent：最基础的代理抽象类，定义了所有代理的基本状态管理和执行循环
  • ReActAgent：实现ReAct模式的代理，具有思考(Think)和行动(Act)两个主要步骤
  • ToolCallAgent：能够调用工具的代理，继承自ReActAgent并扩展了工具调用能力
  • Manus：具体实现的智能体实例，集成了所有能力并添加了更多专业工具

架构设计参考：Base管理全局的state+loop，ReAct在循环内部添加Think+Act，ToolCallAgent则在ReAct基础上扩展excute能力，Manus/MCP则是工具的具体实现

tool

tool内定义了网页搜索、文件操作、寻求用户帮助、代码执行器等工具

prompt

prompt内部则是提示词，比较专业。

系统支持

  • schema.py
    • 记忆系统：使用Memory类存储对话历史和中间状态
    • 工具系统：提供BaseTool和ToolCollection类扩展智能体的能力边界
    • 流程控制：通过AgentState和执行循环管理状态转换和任务流程
  • llm.py
    • LLM大模型：通过LLM类提供思考和决策能力

内部细节（agent）

BaseAgent（模板设计模式）

  • agent\base.py下，核心就是状态管理+Loop的机制，并在其中加入了stunk检查和步数限制的逻辑
  • 模板设计模式，具体的step交由子类实现

ReActAgent

在agent\react下，实现了step方法：体现为Thinking+Acting的过程，体现了“思考——行动——观察”的逻辑，而Thinking和Acting具体实现仍然交由子类

ToolCallAgent

agent\toolcall目录下，具体实现了Thinking和Act，大体思路是思考的时候是请求LLM使用哪些工具，然后Act的时候来调用工具。

Manus

是OpenManus核心实例，主要是一个集成，

  • 默认提供了python执行、浏览器（demo视频上的）、编辑、AskHuman、Terminate的工具。
  • 并提供了远程工具调用的MCP客户端实例，在create方法中通过反射进行初始化。
  • thinking重写，动态决策下一步要做什么（亮点：在原先的父类thinking前面临时对浏览器的场景做了动态增强，不影响其他流程）

优秀思想

  • 所有工具抽象为BaseTool抽象基类（tool\base.py），统一了接口和行为（excute，to_params方法等）
  • 设计了特殊工具，如Terminate终止工具，并允许智能体通过LLM自主决定何时结束任务，避免无限循环或者过早结束
    • 我们在manus源码中发现special_tool_names字段（工厂方法中就默认加载了Terminate），而该字段在toolcall中对special_tool_names对特殊工具进行具体定义

  • AskHuman工具：允许在遇到无法解决的问题时向人类寻求帮助，让用户可以更好干预过程，提高智能体实用性和安全性
  • 工具集合（ToolCollcetion）：设计该类（tool\toolcollcation）管理多个工具实例，提供统一的工具注册和执行接口，实现工具的可插拔性（类似我们之前基于适配器模式写的工具注册类）
  • MCP集成至工具系统：MCP本质是工具调用，设计者就复用了工具的系统，对MCP进行设计，具体体现在MCPClients继承ToolCollection，拥有同样的excute、to_parmas等方法
    • MCP客户端的动态代理：动态创建MCPClientTool实例作为每个远程工具的代理

  • MCP集成至Manus：智能体Manus通过工具调用同样实现了MCP的无缝集成，在源码中manus就是将MCP提供的工具动态添加到工具集合中

  • 代码执行沙盒

  • 巧妙的状态管理和上下文管理机制
  • 工具结果表示抽象统一（ToolResult类），支持结果组合，更加灵活，特别是对于多个工具结果或者异常情况的处理

扩展知识

智能体工作流

当我们面对复杂任务时，单一智能体可能无法满足需求。因此智能体工作流(Agent Workflow)应运而生，

智能体工作流编排的精髓在于将复杂任务分解为连贯的节点链，每个节点由最适合的智能体处理，节点间通过条件路由灵活连接，形成一个高效、可靠的执行网络。

  • Anthropic 《Building effective agents》

Prompt Chaining 提示链工作流

是最常见的工作流之一。它的核心思想是将一个复杂任务拆解为一系列有序的子任务，每一步由LLM处理前一步的输出，逐步推进任务完成。

  • 结构清晰、易于调试，适合可拆解为多个阶段的场景

比如在内容生成场景中，可以先让模型生成大纲，再根据大纲生成详细内容，最后进行润色和校对。每一步都可以插入校验和中间检查，确保流程正确、输出更精准。

Routing路由分流工作流

Routing工作流模式则更像是一个智能的路由器。系统会根据输入内容的类型或特征，将任务分发给最合适的下游智能体或处理流程。

  • 适合多样化输入和多种处理策略的场景。

比如在客服系统中，可以将常见问题、退款请求、技术支持等分流到不同的处理模块;

在多模型系统中，可以将简单问题分配给小模型，复杂问题交给大模型。这样既提高了处理效率，也保证了每类问题都能得到最优解答

Parallelization并行流

在Parallelization并行化模式下，任务会被拆分为多个可以并行处理的子任务，最后聚合各自的结果。

比如在代码安全审查场景中，可以让多个智能体分别对同一段代码进行安全审查，最后“投票”决定是否有问题。又比如在处理长文档时，可以将文档分段，每段由不同智能体并行总结。这种模式可以显著提升处理速度，并通过”投票”机制提升结果的准确度

Orchestrator-Workers 协调器-执行者工作流

对于复杂的任务、参与任务的智能体增多时，我们可以引入一位“管理者”，会根据任务动态拆解出多个子任务，并将这些子任务分配给多个”工人”智能体，最后再整合所有工人的结果。这种中央协调机制提高了复杂系统的整体效率，

  • 适合任务结构不确定、需要动态分解的复杂场景。

Evaluator-Optimizer评估-优化循环工作流

Evaluator-Optimizer模式模拟了人类“写=>评=>改”的过程。一个智能体负责生成初步结果，另一个智能体负责评估和反馈，二者循环迭代优化输出。

  • 这种模式特别适合需要多轮打磨和质量提升的任务。

举个例子，在机器翻译场景中，先由翻译智能体输出，再由评审智能体给出改进建议，反复迭代直到达到满意的质量。

工作流编排框架

如果要实现上述工作流，我们可以采用Activiti、LiteFlow这种工作流框架，把智能体处理当做工作流的一个节点。也可以直接使用专门的AI智能体工作流编排框架，比如LangGraph和Spring AI Alibaba Graph。这两个框架大家简单了解一下，直到怎么组合工作流即可，实际学习成本都不高。

  1. LangGraph
    LangGraph是LangChain团队开发的前沿工作流编排框架，专为大语言模型应用设计，是构建复杂AI系统的首选工具。LangGraph的核心理念是将智能体工作流表示为状态转换图，每个节点可以是函数、智能体或子工作流，边则代表状态转换条件。
    相比于传统的工作流框架，LangGraph的独特之处在于它对大语言模型工作流的深度优化。框架提供了丰富的功能来满足大模型应用开发场景，比如动态分支和并行执行、思维链支持、对话管理、内置监控与可视化等。开发者可以轻松实现复杂的推理路径，让模型在需要时进行反思、规划和纠错。
    前面讲到的几种工作流模式，以及普通的工具调用代理场景，使用LangGraph都能轻松完成，具体用法和代码示例可以参考 官方文档。

OWL框架

OWL: Optimized Workforce Learning是面向多智能体协作与真实世界任务自动化的前沿框架，可以帮助我们终端命令、运行代码、使用各种开发工具等。

其实OWL的本质就是利用AI来增强传统的自动化办公场景，像自动化数据分析和处理、自动化网页信息检索，都能够用OWL轻松完成。

A2A协议

简单来说，A2A协议（Agent toAgent）就是为“智能体之间如何直接交流和协作”制定的一套标准。

A2A协议的核心，是让每个智能体都能像“网络节点”一样，拥有自己的身份、能力描述和通信接口。它不仅规定了消息的格式和传递方式，还包括了身份认证、能力发现、任务委托、结果回传等机制。

这样一来，智能体之间就可以像人类团队一样，互相打招呼、询问对方能做什么、请求协助。

可以把A2A类比为智能体世界里的HTTP协议，HTTP协议让全球不同服务器和电脑之间能够交换数据，A2A协议则是让不同厂商、不同平台、不同能力的智能体能够像团队成员一样互相理解、协作和分工。如果说HTTP协议让互联网成为了一个开放、互联的世界，那么A2A协议则让智能体世界变得开放、协作和高效。

A2A协议的应用非常广泛，总结下来4个字就是开放互联。

比如在自动驾驶领域，不同车辆的智能体可以实时交换路况信息，协同避障和规划路线;

在制造车间，生产线上的各类机器人智能体可以根据任务动态分工，互相补位;

在金融风控、智能客服等场景，不同的智能体可以根据自身专长协作处理复杂业务流程。

我们还可以大胆想象，未来开发者可以像调用云服务一样，按需租用或组合不同的智能体服务，甚至实现智能体之间的自动交易和结算。

目前其实就有很多智能体平台，只不过智能体之间的连接协作甚少。

和MCP协议的区别：虽然A2A和MCP都算是协议(或者标准)，但二者存在本质上的区别。

  • MCP协议是智能体和外部工具之间的标准，它规定了智能体如何安全、规范地调用外部的数据库、搜索引擎、代码执行等工具资源。你可以把MCP理解为”智能体-工具”的HTTP协议。

  • 而A2A协议则是智能体之间的通信协议。它更像是让不同的AI角色之间可以直接对话、协作和分工。

  • 从安全角度看，MCP和A2A处理的是不同层面的安全问题:

    • MCP的安全关注点：主要集中在单个智能体与工具之间的安全交互，主要防范的是工具滥用和提示词注入攻击。

    • A2A的安全关注点：更关注智能体网络中的身份认证、授权和信任链。A2A需要解决”我怎么知道我在和谁通信”、“这个智能体有权限请求这项任务吗”、“如何防止恶意智能体窃取或篡改任务数据”等问题。(这些问题也是HTTP协议需要考虑的)

    • 显然，A2A面临的安全挑战更加复杂，因为它处理的是跨网络、跨平台、多方协作的场景。

对于一个成熟的智能体系统，可能会同时运用MCP和A2A，MCP负责某个智能体内部调用工具完成任务，A2A负责智能体之间协同完成任务，A2A 和 模型上下文协议 (MCP)是构建健壮代理应用的互补标准。

  • 补充阅读：Agent2Agent (A2A) 协议是什么？

如果你能想象在亚马逊上买书的流程，你就已经理解了 A2A 在网络上传输的四种数据形式。

  1. 浏览列表：客户端获取一次智能体卡片。如果“功能”（能力）和“结账”（认证）看起来不错，就继续。
  2. 下订单：客户端发送createTask JSON-RPC 请求（就像点击“立即购买”）。远程智能体用task_id回复，这是你该任务的订单号。
  3. 查看跟踪邮件：远程智能体通过服务器发送事件流式传输消息：“待处理”、“处理中”，可能还有“需要输入”（一个“需要签名”的时刻）。客户端可以用addInput回复，就像你更新送货说明一样。
  4. 收到包裹：当状态变为“已完成”时，工件事件传递有效负载——PDF 报告、PNG 资产、JSON 数据或任何承诺的内容。
  5. 闭环：如果任务失败或被取消，远程智能体将其标记为“已失败”或“已取消”，并且不发送工件（就像亚马逊退还未完成的订单）。

通过这样构建交互流程，你可以明白为什么 A2A 保持规范简洁：它只定义了每个购物者（客户端）和卖家（远程）绝对需要的东西——目录、订单、跟踪、交付——同时将“仓库内部”（模型提示、工具模式）留给 MCP 或卖家选择的任何其他机制。
