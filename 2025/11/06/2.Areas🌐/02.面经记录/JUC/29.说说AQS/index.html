29.说说AQS

发表于2025-11-06|更新于2025-11-06|2.Areas🌐02.面经记录JUC
|总字数:2.2k|阅读时长:7分钟|浏览量:

如果面试官问为什么需要 AQS ，简单回答作用即可，这个回答太细容易绕进去：

AQS（Abstract Queue Syncronized)，那么就是将JUC中排队、入队、加锁、中断等方法进行抽象，从而便于其他JUC锁使用。而具体的加锁、入队等策略则交由具体的实现类控制，那么在AQS中，为了管理线程对共享资源的访问，核心定义了两个内容：

  • State（volatile修饰，共享状态）：如独占锁，0——未占用，1——已占用
  • 队列（FIFO）：线程获取资源失败，则加入AQS等待队列（是CLH队列的变体），通过双向链表结构相互连接
    • 节点内部包括线程引用、等待状态，以及前后驱指针

实现类：ReentrantLock、CountDownLatch、Samphore等

追问：让你实现一个AQS类，你会怎么实现？

那么参考ReentrantLock的实现思路，首先要实现AQS规定的方法：

  • 独占模式下
    • 获取同步状态：boolean tryAccuire(int arg)，通过AQS（0 + 1 = 1），设定成功返回true
    • 释放同步状态：boolean tryRelease(int arg)，判断：当前state是否为0，设置独占线程为null，设置state为0，返回true
  • 共享模式
    • 获取同步状态：int tryAccuireShared(int arg)
    • 释放同步状态：boolean tryReleaseShared(int arg)
  • 判断当前是否是独占模式的持有状态：boolean isHeldExclusively()：判断state是否为0
  • Condition的新建：newCondition

追问：ReentrantLock 原理？

源码中可以发现同样是定义一个 Sync 继承 AQS 自定义 tryAcquire、tryRelease、isHeldExclusively 等方法。

首先按到如何获取锁（默认非公平锁），可以看到是CAS修改state，实现了可重入：

  • state = 0：说明没人占用，此时将state设定为申请的数量，成功设定后返回true
  • state != 0
    • 判断占用的线程是不是自己，如果是自己，叠加申请数量，同时判断int溢出情况，成功设定后返回true
    • 否则，没抢到，返回false

来到 tryRelease 尝试释放锁的方法：

  • 预期state数量c：state - release

  • 判断当前线程是否占用锁进程，如果不是，报错

  • 如果c = 0：释放锁

  • 设定c后，返回当前线程是否空闲（为0就返回true），

    可以看到是可重入锁的逻辑：如果有重入或者共享状态，那么来一个线程则可以将 state + 1，同理释放的时候将 state - 1，如果 state 为 0 说明线程已经不需要这把锁了，此时尝试释放锁即可。

追问：源码中 selfInterrupt 的作用是什么?

线程阻塞等待锁过程中被中断唤醒后，如果前面有排队抢锁的线程，那么被中断的线程会因为抢不到锁而再次阻塞，那么就无法响应中断。

因此 acquireQueued 通过 interrupted 变量记录了之前产生过的中断，然后 acquire 方法判断，如果 interrupted = true 说明之前被中断过，则自己主动触发一次中断补上之前的中断！

追问：AQS 中变型的 CLH是什么？

因为CLH本身会占用CPU，优化这个问题，AQS将前置节点改成了阻塞线程（但也会导致后续线程无法主动发现前面线程释放锁，所以如果释放了就后向通知-1信号）

根据不同情况，线程的等待状态有以下几种：

  • 0，初始化的时候的默认值
  • CANCELLED，值为 1，由于超时、中断或其他原因，该节点被取消
  • SIGNAL，值为 -1，表示该节点准备就绪，正常等待资源
  • CONDITION，值为 -2，表示该节点位于条件等待队列中
  • PROPAGATE，值为 -3，当处在 SHARED 情况下，该字段才有用，将 releaseShared 动作需要传播到其他节点

AQS 扩展知识

AQS 的核心机制

1）状态（State）：

AQS 通过一个 volatile 类型的整数 state 来表示同步状态。

子类通过 getState()、setState(int) 和 compareAndSetState(int, int) 方法来检查和修改该状态。

状态可以表示多种含义，例如在 ReentrantLock 中，状态表示锁的重入次数；在 Semaphore 中，状态表示可用的许可数。

2）队列（Queue）：

AQS 维护了一个 FIFO 的等待队列，用于管理等待获取同步状态的线程。每个节点（Node）代表一个等待的线程，节点之间通过 next 和 prev 指针链接。

当一个线程获取同步状态失败时，它会被添加到等待队列中，并自旋等待或被阻塞，直到前面的线程释放同步状态。

3）独占模式和共享模式：

  • 独占模式：只有一个线程能获取同步状态，例如 ReentrantLock。
  • 共享模式：多个线程可以同时获取同步状态，例如 Semaphore 和 ReadWriteLock。

关键方法介绍，子类通过重写这些方法来实现特定的同步器：

1）tryAcquire(int arg)：

尝试以独占模式获取同步状态。由子类实现，返回 true 表示获取成功，返回 false 表示获取失败。

2）tryRelease(int arg)：

尝试以独占模式释放同步状态。由子类实现，返回 true 表示释放成功，返回 false 表示释放失败。

3）tryAcquireShared(int arg)：

尝试以共享模式获取同步状态。由子类实现，返回一个非负数表示获取成功，返回负数表示获取失败。

4）tryReleaseShared(int arg)：

尝试以共享模式释放同步状态。由子类实现，返回 true 表示释放成功，返回 false 表示释放失败。

5）isHeldExclusively()：

判断当前线程是否以独占模式持有同步状态。由子类实现，返回 true 表示当前线程持有同步状态，返回 false 表示没有持有。

简单基于 AQS 实现一个独占锁

定义一个 Sync 继承 AQS 自定义 tryAcquire、tryRelease、isHeldExclusively 即可实现一个独占锁，非常简单。所以说 AQS 把一切都封装的很好，基于 AQS 可以便于其他相关 JUC 锁的使用！
