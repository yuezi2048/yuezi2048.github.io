38.Volatile对比Synchronized

发表于2025-11-06|更新于2025-11-06|2.Areas🌐02.面经记录JUC
|总字数:586|阅读时长:1分钟|浏览量:

volatile和synchronized作用和场景不同：

  • volatile：
    • 应用于字段变量，保证变量可见性（每次读取volatile变量，会从主内存读取最新值，写入则会立即将其刷到内存），性能开销小（不加锁，只保证线程间可见）
    • 但不保证原子性（如i++会有线程安全问题）
  • synchronized
    • 应用于方法 / 代码块，保证可见性（加锁的方式从主存读取最新值，将数据写入主存后释放锁）
    • 保证原子性（锁保证同一时间只能有一个线程执行被保护的代码块），性能开销大（需要获取和释放锁，会造成线程阻塞）
特性 volatile synchronized
适用范围 单个变量 方法或代码块
可见性 保证 保证
原子性 不保证 保证
性能开销 较小 较大
典型场景 状态标志位 复杂操作或多个变量的保护

扩展知识

volatile 如何保证可见性？

volatile 关键字在 Java 中通过特定的内存模型规则来保证变量的可见性。它确保一个线程对 volatile 变量的修改对其他线程是立即可见的。

1）Java 内存模型（JMM）

Java 内存模型定义了线程如何与主内存以及线程的工作内存进行交互。

  • 每个线程都有自己的工作内存，用于存储线程本地的变量副本。
  • 线程对变量的操作（读取、写入）通常发生在工作内存中，而不是直接操作主内存。
  • 如果没有特殊的同步机制，线程 A 对共享变量的修改可能不会被线程 B 看到，因为线程 B 可能仍然使用其工作内存中的旧值。

2）volatile 的作用机制

  • 写操作时强制刷新到主内存。
  • 读操作时强制从主内存读取。

3）Happens-Before 原则

volatile 的可见性还依赖于 Java 内存模型中的 Happens-Before 原则中的一部分。这个原则定义了操作之间的顺序关系，确保某些操作的结果对其他线程可见。

  • 对 volatile 变量的写操作 Happens-Before 后续对该变量的读操作。
  • 在同一个线程内，按照程序代码的顺序，前面的操作 Happens-Before 后面的操作。
