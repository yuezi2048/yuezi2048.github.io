30.ReentrantLock原理

发表于2025-11-06|更新于2025-11-06|2.Areas🌐02.面经记录JUC
|总字数:2.1k|阅读时长:6分钟|浏览量:

ReentrantLock 是基于AQS实现的可重入锁，支持公平锁和非公平锁，具体地：

  • 根据State变量和两个等待队列（同步队列 + 等待队列），并基于CAS修改State来争抢锁
    • 争抢不到锁 ：进入同步队列（本质是双向链表）
    • Condition不满足：进入等待队列（单向链表）
  • 公平锁的划分：线程获取锁是加入到同步队列尾部，还是直接利用CAS争抢锁

追问：CAS中的自旋锁了解吗？在高并发下如何解决锁饥饿和总线风暴的问题？

CAS是轻量级锁的级别，获取失败不会阻塞，而自旋锁的意思就是尝试循环获取锁，直到拿到为止，很明显是牺牲CPU计算 换来 线程切换的开销，同时也会导致可能一些线程永远拿不到锁，导致锁饥饿。解决这个也很简单，维护一个FIFO的关系，然后计算本地变量即可，也同时避免了总线风暴

改进为基于队列的自旋锁（ CLH，Craig, Landin, and Hagersten），从而适应高并发场景，具体：

  • 维护隐式队列（ AtomicReference + ThreadLocal 实现的逻辑队列，不是真正的指针），当线程等待锁时，自旋本地变量（前面线程的状态），从而减少对共享变量的征用和缓存一致性流量
    • 争抢的线程进入队列，通过排序按需抢锁，每个线程自旋判断前面线程的状态，如果前面释放了，则后续线程争抢锁，避免了锁饥饿 总线风暴的问题

但是也有缺点：自旋本身仍会占用CPU，适合锁等待时间短的场景。

追问2： 让你来改进CLH自旋问题你会怎么解决？ / AQS对CLH的改进了解吗

阻塞 + 通知机制。隐式队列维护的也不是单向关系，而是双向关系，当释放锁后，则通过向后续节点通知变量，从而避免自旋时CPU占用的问题.

通知的情况包括：初始化默认0、超时CANCELED、就绪SIGNAL、在条件等待队列中CONDITION、SHARED情况下需要传播给其他节点PROPOGATE

扩展知识

CAS

什么是 Java 的 CAS（Compare-And-Swap）操作？

CAS 是一种无锁编程技术中的基本操作，它提供了一种无需操作系统级别锁就可以执行原子更新的方式。CAS 操作通常包括三个操作数：

\1. 内存位置（V）

\2. 预期原值（A）

\3. 新值（B）

CAS 操作会检查 V 位置的值是否与 A 相等，如果相等，则将 V 位置的值原子地更新为 B。如果值不相等，则不会进行更新，并且操作返回失败。这种操作通常是原子的，由处理器直接支持。

CAS 通常用于实现非阻塞的数据结构或算法，如无锁队列、栈等。Java 中的 java.util.concurrent.atomic 包下的原子类（如 AtomicInteger）都利用了 CAS 操作。

AQS 是一个用来构建锁和其他同步组件的框架。它是一个位于 java.util.concurrent 包内的抽象类，提供了内部的 FIFO 等待队列来管理线程以及同步状态的管理。

AQS 定义了一套多线程访问共享资源的同步器框架，许多同步类如 ReentrantLock, Semaphore, CountDownLatch 等都是基于 AQS 实现的。AQS 中的关键抽象包括同步状态（state）和条件队列（CLH 锁队列）。

AQS 支持两种主要类型的同步器：

\1. 独占模式：只有一个线程可以执行，例如互斥锁。

\2. 共享模式：多个线程可以同时执行，例如信号量。

当线程试图获取同步状态失败时，AQS 会将该线程插入到等待队列中，并阻塞线程，直到同步状态可用。

总结

- CAS 更适用于实现轻量级的同步操作，特别是那些可以通过无锁方式避免使用重量级锁的情况。

- AQS 提供了一个更高级别的抽象，用于构建需要等待队列和复杂同步状态管理的锁和同步组件。

两者经常一起使用，例如在 ReentrantLock 中，CAS 用于实现锁的获取和释放操作，而 AQS 则管理锁的状态和等待队列。

自旋锁

自旋锁（Spinlock） 是一种轻量级锁机制。线程在获取锁失败时不会立即进入阻塞状态，而是会在循环中反复尝试获取锁，直到成功。

这种方式避免了线程的上下文切换开销，所以称之为轻量级锁，适用于锁等待时间较短的场景。

  • 锁饥饿问题：高并发场景，可能存在某个线程一直 CAS 失败，争抢不到锁。
  • 性能问题：多核处理器如果对同一变量高并发进行 CAS 操作，会导致总线风暴问题（参见 CAS 扩展知识）。

CLH

针对自旋锁的问题，演进出一种基于队列的自旋锁即 CLH（Craig, Landin, and Hagersten），它适用于多处理器环境下的高并发场景。

原理是通过维护一个隐式队列，使线程在等待锁时自旋在本地变量上，从而减少了对共享变量的争用和缓存一致性流量。

它将争抢的线程组织成一个队列，通过排队的方式按序争抢锁。且每个线程不再 CAS 争抢一个变量，而是自旋判断排在它前面线程的状态，如果前面的线程状态为释放锁，那么后续的线程则抢锁。

因此，CLH 通过排队按序争抢解决了锁饥饿的问题。通过 CAS 自旋监听前面线程的状态避免的总线风暴问题的产生。

不过 CLH 还是有缺点的：

  • 占用 CPU 资源：自旋期间线程会一直占用 CPU 资源，适用于锁等待时间较短的场景。

注意！上面说了 CLH 是通过隐式队列实现的，这里的隐式指的是不同线程之前是没有真正通过指针连接的， 仅仅是利用 AtomicReference + ThreadLocal 实现了隐式关联。

AQS 对 CLH 的改造

因为 CLH 有占用 CPU 资源问题，因此 AQS 将自旋等待前置节点改成了阻塞线程。

而后续的线程阻塞就无法主动发现前面的线程释放锁，因此前面线程需要需要通知后续线程锁被释放了。

所以 AQS 的变型版 CLH 需要显式地维护一个队列，且是一个双向列表实现，因为前面线程需要通知后续线程。

且前面线程如果等待超时或者主动取消后，需要从队列中移除，且后面的线程需要“顶”上来。

在 AQS 中，线程的等待状态有以下几种：

  • 0，初始化的时候的默认值
  • CANCELLED，值为 1，由于超时、中断或其他原因，该节点被取消
  • SIGNAL，值为 -1，表示该节点准备就绪，正常等待资源
  • CONDITION，值为 -2，表示该节点位于条件等待队列中
  • PROPAGATE，值为 -3，当处在 SHARED 情况下，该字段才有用，将 releaseShared 动作需要传播到其他节点
