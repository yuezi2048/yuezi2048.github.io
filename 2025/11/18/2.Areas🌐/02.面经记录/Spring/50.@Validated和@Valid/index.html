50.@Validated和@Valid

发表于2025-11-18|更新于2025-11-18|2.Areas🌐02.面经记录Spring
|总字数:1.4k|阅读时长:5分钟|浏览量:

@Validated 和 @Valid 是Spring用于对象验证的注解，这两者场景和特性有区别：

  • @Valid：是标准的Java Bean Validation 注解（ javax.validation 包），作用于方法参数或者类的字段
    • 如@NotNull、@Size 等。在 Spring 中，它可以用于验证单个对象或嵌套对象。
  • @Validated：是Spring特有注解（ org.springframework.validation.annotation.Validated 包），主要是可以支持分组验证（Group Validation）
    • 允许开发者根据不同的场景定义不同的验证逻辑。它也可以用在类级别、方法参数上，触发不同验证组的规则。

总结

  • @Valid：用于标准的 Java Bean 验证，适用于单一对象或嵌套对象，不能支持分组验证。
  • @Validated：Spring 提供的注解，支持分组验证，允许根据不同的场景执行不同的验证逻辑。

分组验证解释：在企业级应用中，通常需要针对不同操作定义不同的验证逻辑。如创建用户对象，我们希望在用户创建时，验证用户名和密码不能为空，而在更新用户信息时，用户名可选，但用户 ID 必须提供。

扩展知识

@Valid 的使用

@Valid 是 Java Bean Validation 标准的一部分，广泛用于验证方法参数、对象属性。它会根据类中的验证注解（如 @NotNull、@Min、@Size 等）自动进行验证。

使用场景：

  • 在表单提交时，验证输入的用户数据是否符合要求。
  • 在 API 请求中，验证请求体的参数格式是否正确。

主要特点：

  • 嵌套对象验证：如果对象中包含嵌套对象，@Valid 还会递归验证嵌套的子对象。
  • 不支持分组验证：@Valid 只能触发默认的验证组，无法根据场景区分验证逻辑。

@Validated 的使用

@Validated 是 Spring 框架特有的注解，它除了支持所有 @Valid 的功能之外，还允许开发者通过分组来进行不同场景下的验证。这在一些场景中非常有用，例如区分创建和更新操作的验证规则。

使用场景：

  • 需要在不同场景下执行不同的验证规则，如创建时要求所有字段必须提供，而更新时可能只需要验证部分字段。

  • 创建用户时，只需要验证 name 字段。
  • 更新用户时，需要验证 id 字段是否存在。

主要特点：

  • 分组验证：@Validated 支持通过 groups 参数指定验证组，可以根据不同的场景（如创建、更新、删除等）触发不同的验证逻辑。
  • 适用范围广：除了方法参数，还可以用于类级别的注解，尤其适用于 @ConfigurationProperties 等配置类的校验。

分组验证的具体场景

在企业级应用中，通常需要针对不同操作定义不同的验证逻辑。通过 @Validated 的分组功能，可以实现对同一个实体类在不同操作下执行不同的验证规则。

示例：区分创建和更新操作的验证

假设有一个用户对象，我们希望在用户创建时，验证用户名和密码不能为空，而在更新用户信息时，用户名可选，但用户 ID 必须提供。可以通过分组验证来实现这一需求。
