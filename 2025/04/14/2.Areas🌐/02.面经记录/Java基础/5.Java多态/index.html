5.Java多态

发表于2025-04-14|更新于2025-04-14|2.Areas🌐02.面经记录Java基础
|总字数:2.3k|阅读时长:8分钟|浏览量:
  • 多态：一个接口/父类引用的变量可以指向不同的对象实例，可以根据实际的对象实例来执行对应不同的方法
    • 也就是说，多态允许在一个方法能表现出多种不同的行为，是OOP的核心特性之一
    • 优势：能够灵活处理不同类型的对象，在新增子类/实现类时，无需修改原有代码，只需通过父类/接口的引用即可调用，符合开闭原则，减少了代码耦合，增强可扩展性

拓展：

  • 多态的分类
    • 编译时多态：方法的重载，编译时确定方法的调用
    • 运行时多态：方法的重写，运行时确定方法的调用
  • 动态绑定机制：
    • 调用对象的方法时，方法和该对象的内存地址/运行类型绑定
    • 调用对象的属性时，没有动态绑定机制，哪里声明，哪里调用

什么是耦合？

案例1：JDBC，当我注册驱动类这样写，乍一看没问题，但是从耦合角度分析，耦合度是非常高的，不利于产品的迭代升级。

如果不小心注释了pom.xml相关注解，那么就会直接编译不通过，这是我们不愿意看到的，追其原因就是驱动类高度依赖pom.xml配置，耦合度太高了。

实际开发中， 我们应做到在编译期不依赖，而在运行期减少依赖

这里主要是说明耦合，该案例的解决方案有很多：读取配置文件反射读取类、工厂解耦，这么做就是为了降低依赖关系。

AutoWired：修饰成员变量

  • 自动注入的原理
    • 先按照类型进行注入：此时IOC必须存在唯一且与数据类型匹配的bean
      • Q: 为什么我定义的是接口，而实际注入的是实现类？答：请参阅接口多态相关章节
    • 再按照名称进行注入：如果有多个数据类型匹配的bean，再按照名称注入，名称注入具体的做法如下：
      • 将@AutoWired注解修饰的变量的名称和容器中的bean的id进行匹配
      • 如果匹配成功，则按照名称注入成功，否则注入失败： No qualifying bean of type available

Q: 如果IOC中有多个类型匹配，又该如何处理？

我们接下来就定义两个实现类观察spring究竟是怎么处理的

  • 发现直接报错：Could not autowire. There is more than one bean of ‘AccountDao’ type. 看来就是类没有匹配上，名称也没有匹配上了

  • @Qualifier：

    • 作用：在按类型注入的基础上指定名称注入，给类成员注入时不能单独使用；后续会讲到该注解可以给方法使用

    • 属性：value，指定bean的id

  • @Resource：我们单独使用@Autowired时需要改变变量名；使用@Qualifier时又不能单独使用，我嫌麻烦，因此引出@Resource集成

    • 作用：默认情况下按照名称装配，可以独立使用（非spring官方注解，是javax拓展包下的注解）
    • 属性：name指定bean的id，type按照类型进行注入
    • 说明：默认优先按照名称装配
      • 如果同时指定了name和type，则从Spring上下文中找到唯一匹配的bean进行装配，找不到则抛出异常
      • 如果指定了name，则从Spring上下文中查找名称（id）匹配的bean进行装配，找不到则抛出异常
      • 如果指定了type，则从Spring上下文中找到类型匹配的唯一bean进行装配，找不到或找到多个，都抛出异常
      • 如果既没指定name，也没指定type,则自动按照byName方式进行装配。
    • 与@AutoWired区别：@AutoWired默认按类型装配，Resource默认按名称装配

  • @Value：以上的注解都无法注入基本数据类型，且集合类型的注入只能通过XML实现，引出@Value

    • 作用：用来注入基本数据类型和String类型的数据，可以使用SpEL表达式（Spring的EL表达式 ${}）
