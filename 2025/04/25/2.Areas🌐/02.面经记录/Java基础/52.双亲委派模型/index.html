52.双亲委派模型

发表于2025-04-25|更新于2025-04-25|2.Areas🌐02.面经记录Java基础
|总字数:890|阅读时长:2分钟|浏览量:
  • 双亲委派模型是类加载机制的设计模式之一，核心思想：
    • 类加载某个类时，会先委派给父类加载器加载
    • 当父类加载器无法加载时，才由子类加载器加载

工作过程

  • 类加载器（如自定义类加载器）试图加载某个类时，会先向上委派给父类加载器，父类再委派给他的父类，知道根加载器（Bootstrap ClassLoader）

拓展

JDK9之前，Java的3种类加载器

  • BootStrap ClassLoader（启动类加载器）：属于虚拟机一部分，C++实现。负责加载JAVA_HOME\lib目录或者被-Xbootclasspath指定的路径中被虚拟机识别的文件，是所有类的父加载器。
  • Extension ClassLoader（拓展类加载器）：Java实现，独立于JVM。加载JAVA_HOME\lib或者被java.ext.dirs系统变量指定的类库
  • Application ClassLoader（应用程序类加载器）：Java实现，独立于JVM。加载用户类路径classPath上的类库，如果没有实现自定义类加载器就会用程序中的默认加载器。

顺序：自定义ClassLoader–> Application ClassLoader–> Extension ClassLoader–> BootStrap ClassLoader

双亲委派机制的意义

  • 让类有层次的划分：如java.lang.Object、加载该类是通过一层层委托直到BootStrap ClassLoader加载，即让启动类加载器找\lib中rt.jar中java.lang.Object加载到JVM中
  • 核心类保护机制：如果有坏人造了一个java.lang.Object，嵌入了不受欢迎的代码，如果按照双亲委派模型，最终还是会加载\lib中rt.jar中的东西

一致性：保证一个类在JVM中只被加载一次，确保在整个应用程序中是同一个类

安全性：避免重复加载，防止恶意代码加载不受信任的类替代核心类（上层优先）

有没有不遵循双亲委派的案例

案例：JDBC

  • JDBC接口是类库定义的，但是实现是通过各个厂商去定义的，那么类启动器是找不到实现类的，所以需要应用程序类加载器来完成，违反了自下而上的委托机制
  • 具体实现：定义了一个线程上下文类加载器，通过setContextClassLoader()默认设置应用程序类加载器，通过Thread.current.currentThread().getContextClassLoader()获取类加载器加载

实际上SPI机制都违背了双亲委派模型，SPI允许开发者在类路径中自定义服实现，通过线程上下文类加载器加载SPI实现类，绕过了父类加载器。

除此之外，Java EE的容器（Tomcat WebLogic等），每个容器内有自己的类加载器，应用程序级别应优先加载类库，而不是父类加载器，违背了双亲委派机制。

为什么不能直接在最顶端自上而下加载？

  • 类加载器本身是组合关系，即子加载器只记录父加载器，父加载器是找不到子加载器的
  • 其次如果父加载器直接传给子加载器，假设有多个子加载器，应该传给谁加载？如果一个个试过去，效率就低了
