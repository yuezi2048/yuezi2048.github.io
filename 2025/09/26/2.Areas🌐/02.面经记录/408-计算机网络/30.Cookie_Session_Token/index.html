30.Cookie_Session_Token

发表于2025-09-26|更新于2025-09-26|2.Areas🌐02.面经记录408-计算机网络
|总字数:1.1k|阅读时长:3分钟|浏览量:
  • Cookie存储在浏览器端，跟踪和保存用户的状态（用户登录状态、跟踪用户行为、存储用户偏好等）
  • Session存储在服务端，保存用户状态的机制，每个会话对应唯一的Session ID（跟踪服务器的状态信息，如登录状态、购物车等）
  • Token存放在客户端，本质是加密的字符串用于身份验证和鉴权，后端认证后返回token存储在客户端中，后续客户端访问服务端带上token

使用场景（Cookie 和 Session用于单次会话认证和状态管理，Token适合跨会话的认证和状态管理）

  • Cookie是客户端简单的存储和追踪
  • Session是服务器端复杂状态管理，特别是存储大量会话数据时
  • Token用于无状态认证和鉴权，特别是分布式和跨域环境下

拓展

Cookie

一开始HTTP的设计是只要在服务器端拿数据，因此只有GET的需求，此时服务器是无状态的。

但随着购物等交互的需要，服务器需要追踪请求的发起者，此时就需要让请求有状态，衍生出Cookie，本质上是kv，存储在客户端本地，请求的时候携带。后端收到cookie后就知道发起者相关的信息。

注意:Cookie有域的划分，所以在跨域的情况下，cookie是单独分离的

示意图

Session

Cookie的缺陷在于明文存在用户本地，携带用户敏感信息，同时请求携带过多的cookie也是消耗带宽。

衍生出Session（会话），将会话信息存在服务端，然后向用户发放sessionID，后续前端请求带sessionID（一般会以cookie的形式存在本地），服务器就能识别到用户信息。

Session的缺陷也有，这种做法让服务器有状态了，换句话说，部署上线的时候（一般会部署多个服务器负载均衡做一个backup），session如果在一个服务器上，那么请求其他负载服务器的时候就找不到session信息了。

  • 一种直观的想法是session复制，但是会有额外开销。
  • 另外一种做法是session sticky，就是将请求黏在一台服务器上，一开始指派的服务器是谁，后续就一直请求这个服务器，但是服务器如果出问题了，会话还是会丢
  • 更保险的做法是交给第三方存储，如redis，这样服务器就没状态了，这意味着可以无限伸缩，服务集群可以随着业务随时增减设备。核心只要保证第三方的高可用即可

Token

这是另外一种思路，用户信息不存在服务端，取而代之的是用户凭证（token），基于用户信息生成签名，后续又通过签名来辨别真伪，验证成功后就能知道是谁发的。

这样服务器是真正的无状态，token的管理交给客户端。

  • 服务端更加轻量，减少冗余数据
  • 由于无状态，可以无限拓展。
  • token也可以跨应用使用（如阿里旗下的产品一个token就能全部登录，这就是单点登录）

token：含有身份凭证的令牌，服务器可以识别token并根据令牌的身份做出回应。

可以理解为时间换空间的思想。用本地签名验证的时间来换取不用存储用户信息、无状态。

相比于cookie/session（存在可扩展性、跨域、跨站伪造请求等问题），token在这方面一个很好的点就是可以更加灵活地跨域访问

补充

  • cookie
    • 有过期时间 或者可以设置为会话cookie（关掉服务器就没了）
    • 可以通过secure httpOnly来提高安全性（仅允许http / 禁用js）
    • 通常大小小于4KB，对域名内的cookie数量也有限制
  • session
    • 生命周期和cookie类似
    • Session信息存储在服务器上，通过生成的sessionID以cookie等形式告诉客户端
  • token
    • 一般可以通过jwt和OAuth2.0的实现，主要特性是无状态会话，另外也可以鉴权，身份验证
    • token本身就包含了用户权限信息，所以服务器不需要额外存储
    • 也有过期时间，有获取token的刷新机制，并且由于加密/签名机制，可以保护内容不被篡改
