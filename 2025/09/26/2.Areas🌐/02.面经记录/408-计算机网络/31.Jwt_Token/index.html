31.Jwt_Token

发表于2025-09-26|更新于2025-09-26|2.Areas🌐02.面经记录408-计算机网络
|总字数:906|阅读时长:2分钟|浏览量:
  • JWT(Json Web Token)是用于传递安全信息 的紧凑、URL安全的令牌格式，具体过程：
    • 用户登录后，服务器生成JWT返回给客户端
    • 客户端在后续请求通过HTTP头（Authorization头）发送JWT，服务器验证JWT的有效性以进行用户身份验证。由于他的无状态性，常用在分布式和微服务架构中。
  • 结构主要包含三部分：Header + Payload + Signature
    • Header：令牌元数据，令牌类型 （JWT）+ 签名算法（HMAC / SHA256）
    • Payload：声明（claims），即传递的数据（用户信息等数据），声明类型包括iss发行者、exp过期时间、sub主题等
    • Signature：Header + Payload通过指定算法签名，用来验证JWT真实性和完整性保证传输过程中没有篡改
  • JWT优点：
    • 自包含：JWT包含了用户信息，验证时不需要查数据库
    • 跨语言：JWT基于JSON，兼容大多数编程语言

扩展

如何废除未过期的JWT

JWT是无状态的，服务器一般不保存已签发的JWT，所以无法直接主动撤销签发过的JWT，但可通过以下方式处理：

  • 黑名单BlackList：服务器维护一个黑名单（废弃列表），包含已被废除的JWT标识符（jti声明），后续每次验证的时候，在验证签名等基础信息之外，检查JWT是否在黑名单即可
    • 优势：精准废除，不影响其他合法JWT
    • 缺点：存储管理黑名单，违背了无状态性，并且列表大了之后，查询效率就会出问题
  • 版本控制Token Version： 用户信息加入version字段，每次生成JWT后，版本号就作为JWT的一部分（payload），当废除JWT后，就将用户版本号递增，验证时判断JWT版本号是否与当前版本是否匹配，不匹配则无效
    • 优势：无需维护黑名单，就可以废除特定用户JWT
    • 缺点：一用户多设备的情况下，一个设备废除，所有设备都会失效。并且服务器端需存储管理用户版本号
  • 结合状态信息：在特定场景下（注销 更改密码），服务器结合状态信息判断JWT有效性，在验证JWT基础上，检查这些是否符合要求，不符合的情况下即使JWT有效，也可以拒绝请求
    • 优势：灵活性高，根据业务灵活决定有效性
    • 缺点：服务器端要保存状态信息，违背了无状态设计初衷，且实现复杂。

JWT安全性

  • 签名/加密：签名保证数据完整性。但是payload通常不加密，所以敏感信息不应放在payload中，可以进一步使用JWE(Json Web Encryption)标准加密JWT的payload
  • 密钥管理：签名安全性依赖密钥保护，密钥如果泄露，攻击者可以伪造有效JWT。因此密钥是保证JWT安全性的关键
  • Token泄露和保护：JWT通常在客户端中，如果通过XSS等攻击泄露了JWT，攻击者可以冒充合法用户，可以通过短期过期时间exp和定期刷新token降低风险。

JWT vs Session

  • 无状态认证：JWT无状态认证，即服务端不存储会话数据，而Session认证需要服务端存储用户会话信息
  • 扩展：JWT不依赖服务器存储，因此在分布式具备拓展型。而Session在多服务器中依赖共享存储（或会话黏性 Session Stickness），增加了系统负载和复杂性
