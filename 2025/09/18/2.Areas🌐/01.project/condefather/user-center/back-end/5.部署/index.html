项目部署

多环境理论

多环境：同个项目在不同的阶段（本地开发->测试调试–>部署上线）需要根据情况需要部署到不同的机器上

好处（针对不同环境做不同的事情）：

  • 每个环境互不影响；
  • 区分不同的阶段（开发/测试/生产）
  • 项目优化，有些依赖不需要放到线上
    • 本地日志
    • 精简依赖：lombook、接口文档
    • 项目的环境 / 参数调整（JVM堆内存等参数）

分类：

  • 本地环境：localhost:8080
  • 开发环境（远程开发）：为了开发方便，大家连同一台机器
  • 测试环境（测试 独立数据库 独立服务器）：开发 / 测试 / 产品
    • 性能测试 / 功能测试 / 系统集成测试
    • 沙箱环境（实验环境）：完全隔离的环境，做实验
  • 预发布环境（内测）：基本和正式一致，正式数据库，可以查出更多问题
  • 正式环境（线上，公开对外访问）：不要改动，保证正常运行

前端多环境

请求地址：

  • 开发环境：localhost:8000
  • 线上环境：user-backend.code-nav.cn

前端区分环境，通过env变量传递。

原项目使用了umi框架，build的时候自动传入NODE ENV === PRODUCTION参数，START通过process.env取到环境变量，可以通过alert(process.env.NODE_ENV)校验

  • 启动方式
    • 开发环境：npm run start 本地启动
    • 线上环境：npm run build 项目构建打包，可以使用serve命令（npm i -g serve）在本地启动正式环境

在request.ts中，后续可以考虑打包后怎么更小一些

项目配置

  • 不同的框架有不同的配置文件，比如umi框架的开发环境在config/config.dev.ts和config/config.prod.ts （配置文件后缀区分 config.ts是公共配置），建议多看文档（可以知道优化方法）

  • 在这个配置文件下就可以实现在不同的环境下有不同的效果（如按需加载，静态化实现线上环境优化）。

    • exportStatic：使得页面静态化（如果遇到问题了查看umi文档）
      • 开启后就是根据页面的路由，每个路由都会对应静态文件
      • 如果不静态化，只有一个index.html，后续根据js加载其他页面，此时如果直接访问其他的路由就会报404（要么配nginix，要么静态化解决）

可以在启动项目时传入环境变量，打包时如果不要执行测试，可以选中这个按钮

3. 容器

docker将项目环境、代码（java nginx等）可以打包成镜像，更容易分发和移植，启动项目时不需要敲很多命令，直接启动即可。

  • 官网安装：https://www.docker.com/get-started/
  • 宝塔安装docker管理器

4. 容器平台

直接用别人的容器平台，只要把镜像放上去就行。（大公司用得多）

  • 云服务商容器平台（腾讯云、阿里云）
  • 专门的容器平台（后端微信云托管） 要花钱

前端部署：webify（比容器更傻瓜）

  • 优势：不用命令，代码更新时自动构建
  • 缺点：需要把代码放到托管平台上

好处：

  • 更方便，不如输命令
  • 更傻瓜式，不用控制台
  • 大厂运维，更省心
  • 额外的监控告警、其他（存储、负载均衡、自动缩扩容、流水线）

域名配置

原始的绑定方式（容器也可以直接绑定一个域名）

前端DNS的过程：

用户输入网址——域名服务器（解析为ip地址 / 交给其他域名解析服务）——

（防火墙）nginx（接受请求） —— 找到对应的文件，返回文件给前端——加载js等文件到浏览器中——渲染

后端DNS过程：

用户输入网址 —— 域名服务器 —— 我们的服务器 —— 后端服务端口

  • A——直接指向
  • CNAME：域名——CDN——负载均衡

前端部署时要注意：nginix的servername是否加入，宝塔的域名防火墙是否放行

后端部署要注意：域名默认的是80，直接用域名得加端口号，我们想实现直接用域名转发到端口

nginx反向代理

方案：配置nginx反向代理——替服务器做请求，让nginx帮忙转发请求

实现效果：该nginx服务请求根目录后，会自动转发到8080端口
