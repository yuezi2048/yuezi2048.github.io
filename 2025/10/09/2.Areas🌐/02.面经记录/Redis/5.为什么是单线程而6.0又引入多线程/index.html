5.为什么是单线程而6.0又引入多线程

发表于2025-10-09|更新于2025-10-09|2.Areas🌐02.面经记录Redis
|总字数:897|阅读时长:2分钟|浏览量:

为什么设计为单线程：

  • Redis操作是基于内存的，性能瓶颈一般不是在CPU上
  • 单线程模型，可以在简化的同时减少线程上下文切换的开销
  • 单线程环境下，可以通过I/O多路复用模型提高Redis的I/O利用率

6.0引入多线程：

  • 随着请求量 数据量规模增大，Redis在网络I/O上出现了瓶颈，引入多线程可以提高网络I/O的处理速度

拓展

  • 我们所说的Redis单线程，指的是Redis网络I/O和键值对读写这些操作是一个线程完成的（持久化 集群是另外的后台线程）
  • 后续4.0也引入了异步删除多线程指令，6.0引入了多线程机制（针对网络请求处理多线程）

为什么要引入多线程？

核心还是对Redis性能有更高要求，随着业务更加复杂，对于QPS要求也更大。

  • 可以通过搭建集群提高QPS，但是资源开销过高
  • I/O多路复用模型本质还是同步I/O，网络I/O的瓶颈还是在的
    • 解释：I/O在处理网络请求后，调用select(epoll)后，如果来数据，数据从内核拷贝到用户空间是同步的，这意味着用户线程需要等待数据拷贝完成（此时redis无法处理命令），如果并发量高，这个过程可能是瓶颈
    • 解决方案：利用CPU多核优势，多线程处理网络请求方式提高效率，而对于读写命令，依旧采用单线程

注：redis引入多线程后，本质的读写操作还是单线程，并不会存在线程安全问题

6.0多线程默认是禁用的，大部分公司的并发量用不上这个，开启需要配置io-threads-do-reads为yes

相关概念说明：

  • 五种I/O模型：https://www.mianshiya.com/bank/1804354610222800897/question/1804476796656885762
    • 核心要理解多路I/O复用和真正异步I/O的区别
  • 同步、异步、阻塞和非阻塞：https://www.mianshiya.com/question/1780933295844913153

Q:多路I/O复用的时候，发送select和拷贝到用户空间的过程中，线程不都是同步的吗？这样出现了没有数据的情况，那不就会导致线程阻塞，那为什么redis缓存还是使用多路I/O复用，直接异步I/O不好吗？

A:

  • 当调用select/epoll_wait时，线程会进入内核态阻塞，此时 CPU 会切换到其他进程 / 线程，不会空耗资源（这与 “非阻塞 I/O + 轮询” 运行态/就绪态的无效 CPU 占用完全不同）。只有当注册的 I/O 事件（如网络数据到达、连接建立）就绪时，内核才会唤醒线程，线程再同步处理这些就绪事件（读写数据）。
  • “等待事件的阻塞” 对整体性能影响极小（因为内存处理速度远快于网络传输），反而避免了多线程切换的开销（Redis 的单线程模型正是为了减少线程切换）。
  • 异步 I/O 的设计初衷更偏向磁盘 I/O（如大文件读写），linux等厂商大多是通过模拟多路复用来模拟出异步I/O的效果，主要是异步 I/O 对网络 I/O 支持有限，且会增加复杂度，性价比远低于多路复用。

10.事务
redis支持事务，但是事务的概念与mysql的ACID不同。Redis的事务指的是保证操作原子性，所有命令在原子操作中进行，不被打断。（注：redis的事务不支持回滚） 拓展 开始事务：MULTI开始，之后的命令进入队列 添加命令：添加命令，不会被立即执行，而是存储在队列 执行事务：EXEC原子性执行上述命令 取消事务：DISCARD清空命令，放弃事务 监视：WATCH监视一个/多个键，如果EXEC前被键修改了，EXEC不会执行，保证数据一致性。 Redis不支持回滚：所有命令都会被应用，即使某个步骤失败，依旧会执行，所以在设计时要充分考虑错误。 Redis2.6.5后，服务器在累计命令的过程中检测到错误，会拒绝执行事务，返回错误并丢弃该事务。 但是在执行过程中，仍会继续执行剩下的命令，不会进行回滚（这是和MySQL最不同的点，也不支持隔离级别设置，因为是单线程，只能是串行隔离级别），所以我们理解为是一个残血版的事务。
