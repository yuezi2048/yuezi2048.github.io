需求分析

1. 创建队伍

用户可以创建一个队伍（队长），设置队伍人数、标题、描述、超时时间 P0

队长、剩余的人数

公开/私有/加密房间

根据标签/名称搜索队伍

创建队伍不能超过5个（可以考虑哪些点收费）

  • 请求参数是否为空
  • 是否登录：未登录则不允许创建
  • 校验信息
    • 队伍人数 > 1 且 < = 20
    • 队伍标题 <= 20
    • 描述 <= 512
    • 超时时间 > 当前时间
    • status 是否公开 0 ~ 2（不传默认0）
      • status = 2 且密码存在，密码长度 <=32 （如果有）
    • 校验用户最多创建5个队伍
  • 插入队伍信息到队伍表
  • 插入用户 => 队伍关系表（要引入事务保证ACID）

注：有一些校验框架（复杂逻辑），如果在规则经常变动的场景下，可以用校验框架，固定的话就可以用这个简单一些。

库表设计

队伍表 team

  • id 主键 bigint（简单、连续、url上简短，缺点是爬虫，可以验证机器/限制请求解决）
  • name 队伍名称
  • description 描述
  • maxNum 最大人数
  • expireTime 过期时间
  • userId 创建人id（我们约定创建人id就是队长id）
  • status 0-公开，1-私有，2-加密
  • password 密码
  • createTime 创建时间
  • updateTime 修改时间
  • isDelete 是否删除

最好整理好这些表格，便于共享

两个关系：

  • 用户加了哪些队伍
  • 队伍有哪些用户？

方案：

  • 建立用户-队伍关系表 teamId、userId（便于修改，查询性能高一些）
  • 用户表补充已加入的队伍字段，队伍表补充已加入的用户字段（便于查询，不用多对多代码，可以直接根据用户查用户，根据用户查队伍，但是删除会比较麻烦）

本项目选用方案1.

用户-队伍表 user_Team

字段：

  • id 主键
  • userId 用户id
  • teamId 队伍Id
  • joinTime 加入时间
  • createTime 创建时间
  • updateTime 修改时间
  • isDelete 是否删除
