75.MySQL为什么索引用B+树不用红黑树

发表于2025-10-31|更新于2025-10-31|2.Areas🌐02.面经记录MySQL
|总字数:555|阅读时长:1分钟|浏览量:

MySQL 使用 B+ 树作索引数据结构，而不是红黑树，主要是对性能和存储效率的考虑。

B+树优势：

  • 磁盘I/O少且稳定：B+树节点可以用多个kv（表示较大的数据范围，对应的扇出就高），一般三层B+树就可以存2000w的数据，减少数据高度，也就减少了从根节点到叶子节点的磁盘I/O
  • 范围查找：所有叶子节点在同一层，且每个叶子节点通过双向链表连接，使得范围查询高效，红黑树需要回溯上一节点并找到下一个符合条件节点，效率低。

红黑树优势

  • 内存中的快速插入和删除：红黑树是一种自平衡二叉搜索树，它最坏情况下的操作时间复杂度为O(log n)。对于主要在内存中运行的应用程序，特别是那些对插入、删除操作频率较高的应用，红黑树非常适合。
  • 实现相对简单：相比B+树，红黑树的实现更加直观和直接，尤其是在不需要处理大量数据或不频繁执行磁盘I/O操作的情况下。

使用场景

  • B+ 树：适用于需要频繁进行范围查询、前缀匹配查询等操作的场景，如数据库索引。此外，当数据量较大，无法完全加载到内存中时，B+ 树通过减少磁盘 I/O 次数来提高效率，非常适合用于数据库系统。
  • 红黑树：更适合于内存中的数据结构，比如 Java 中的 TreeMap。这些场景下，数据集相对较小，可以完全驻留在内存中，并且可能需要高效的动态插入和删除操作。

B+ 树更适合磁盘存储和大规模数据存储，红黑树更适合在内存中灵活地操作。
