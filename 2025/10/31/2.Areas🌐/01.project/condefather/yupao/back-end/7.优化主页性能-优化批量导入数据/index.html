7.优化主页性能-优化批量导入数据

|总字数:2.1k|阅读时长:9分钟|浏览量:

批量导入1000w个用户速度问题

编写一次性任务：

  • 用可视化界面：一次性导入、数据量可控
  • 写程序：for循环（建议渐进式导入，不要梭哈），也可以用接口控制，要保证可控、幂等性，同时注意线上和测试环境有区别
  • 执行sql语句，ctrl+r替换（适合小数据）

任务调度

引入任务调度机制

使用spring的stopwatch监控执行时间工具类

通过一个定时任务，但是我们想执行一次，就尝试使用周期更新频率非常长的做法

测试类

不太优雅，尝试使用测试类（import org.junit.jupiter.api.Test;）

时间分析

  • 建立和释放数据库连接（OpenSession）
  • for循环是绝对线性的（多线程）

优化数据库连接

通过mybatisplus封装的，可以看到快了很多，插入10w条，每1w条建立一次连接，用了21s，但是也不是越大越好，10w条建立一次连接，最后用时是19s

多线程优化

尝试分成10个线程，每个线程插入1w个，发现用了20s，并且发现输出有问题：

  • 发现有的线程干了两次活，时间消耗在了这里

我们分成7个线程（CPU核心数 - 1），让每个人都干一次活，16s，更快了一些

（因为我们用的是默认线程池库ForkJoinPool）

多线程优化2——自定义线程池

并发度搞成40，最后的速度其实还是不一定快，因为数据库插入、线程切换等问题

  • 这里可能还是卡点在数据库的I/O上，这里最高就到19s左右了

测试查询

配置分页插件： 3.5.9 版本之后，在配置分页插件的时候，会提示原本的 PaginationInnerInterceptor 找不到这个类，需要导入新的依赖包:https://blog.csdn.net/lrb0677/article/details/144120956

编写分页查询

发现有点延迟（超过了1s），我们可以把数据做一个缓存（这些数据是比较经常访问且固定的）

  • 所以我们可以预先把数据查出来，放到一个更快读取的地方，不用再查缓存了
  • 预加载缓存策略（定时更新缓存——定时任务）
    • 多个机器多个定时任务——分布式锁（同时只允许调用一部分的数量，抢票场景）控制同一时间只有一台机器执行定时任务，其他机器不用重复执行

81.什么是游标Cursor分页_对比传统LIMIT_OFFSET分页的优势是什么
游标（Cursor）分页是一种基于位置标记的分页方式，通过传递一个游标（如时间戳或唯一ID） 来定位下一页数据的起点。 传统 LIMIT OFFSET 分页则是通过跳过前 N 条数据（OFFSET）再取 M 条（LIMIT）实现分页。 举例： 传统分页：SELECT * FROM user ORDER BY id LIMIT 10 OFFSET 10000 游标分页：SELECT * FROM user WHERE id > last_id ORDER BY id LIMIT 10 游标分页的核心优势： 高性能：避免 OFFSET 扫描跳过的大量数据（尤其在大数据量时），查询速度稳定。 数据一致性：不受分页期间数据增删影响（如第2页数据不会因新增数据而“漂移”）。 简单总结一下：游标分页是记住当前位置，直接取下一页，而 LIMIT OFFSET 是从头数到第 N 条再开始取。 建议大家 观看这个视频，快速学习企业级数据同步的实现方案。 扩展知识如何解决 LIMIT OFFSET 在深分页时慢问题？ 630.
