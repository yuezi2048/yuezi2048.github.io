18.MVCC

发表于2025-10-14|更新于2025-10-14|2.Areas🌐02.面经记录MySQL
|总字数:1.7k|阅读时长:5分钟|浏览量:

MVCC(Multi-Version Concurrency Control 多版本并发控制)，允许多个事务同时读取和写入数据库，而无需互相等待，提高数据库并发性能。

MVCC会为每个事务创建一个数据快照，MySQL不会立即覆盖原有数据，而是生成新版本记录，每个记录保存版本号和时间戳。如此一来，每个版本串联起来一个版本连，不同时刻启动的事务可以无锁地获得不同版本的数据（针对的是普通读，当前读是加锁的），此时读（普通读）写不会阻塞。

写操作就是可以继续写，无非就是创建新的数据版本（注意可见性，事务提交后的新版本才对其他事务可见，未提交的事务不会影响其他事务的读取），而历史版本记录可以供已启动的事务读取

拓展

Undo Log

是InnoDB支持事务回滚的日志机制，记录数据修改历史信息，事务失败/撤销操作时，可以恢复到原先的状态。

实际上，不是真的存储多个版本的数据，而是借助undolog记录每次写的反向操作，因此索引只会有一个最新版本，只是可以通过undo log反向操作得到历史版本，所以逻辑上是多个版本。

insert(1, XX)为例，插入后数据页除了ID和name外，还有trx_id(事务ID)和roll_pointer(undo log 指针)两个字段

  • 该事务ID为1，插入后生成undo log，记录的roll_pointer指向该undolog，类型：TRX_UNDO_INSERT_REC表示insert反向操作
  • undo log还存储了主键长度 / 值，方便定位到记录，实现回滚效果，理解为DELETE 1即可

事务1提交后，事务5进行（update NO where id 1），此时的undo log

  • 产生了TRX_UNDO_UPD_EXIST_REC 类型的undolog

注：当事务1提交后，insert的undolog回收了，因为不可能有其他事务会访问更早的版本（新增前的版本）

而update产生的undolog则可以访问到更早的版本，因此后续再更新会叠加到后面

Read View

我们要求事务提交后的新版本才对其他事务可见，未提交的事务不会影响其他事务的读取，所以引出Read View来判断哪个版本对当前事务可见，在ReadView中有四个核心点：

  • creator_trx_id：当前事务ID（非修改操作不生成事务ID）
  • m_ids：当前状态下活跃的事务ID集合（此时活跃意味着事务已启动但未提交）
  • min_trx_id：活跃ID的最小值
  • max_trx_id：此时InnoDB分配给下一个事务的ID值（随时间递增分配）

因此ReadView的目标是：从最新版本追溯老版本，返回符合条件的版本，判断逻辑（当前数据版本为trx_id）：

  • trx_id == creator_trx_id，表示为当前事务，可见
  • trx_id < min_trx_id，表示已提交事务，可见
  • trx >= min_trx_id && trx < max_trx_id
    • 如果trx∈m_ids，说明事务在活跃状态下，未提交，不可见
    • 否则，事务已提交，可见
  • trx_id >= max_trx_id，表示当前事务在生成readView的时候未启动，不可见（事务ID递增）

读已提交隔离级别

在读已提交的隔离级别下，我们假设一种状态：事务1已提交，事务5已执行，但未提交。

此时有一个事务6（当前最大事务ID）要查询name：select name where id 1，那么根据readView定义：

  • create_trx_id=0，不是修改操作不分配事务ID
  • m_id=[5,6]
  • min_trx_id=5
  • max_trx_id=7

查询id为1的记录，此时在的版本下（name=NO但未提交），undolog版本为

此时最新版本trx_id是5，在min_trx_id到max_trx_id范围内，且在m_id列表中，所以不可访问该事务，根据roll_pointer找到上一个版本，得到name=XX

随后事务5提交，再次查询name，分析此时的readview：

  • create_trx_id=0
  • m_id=[6]
  • min_trx=6
  • max_trx_id=7

查询的ID还是为1的记录，此时的数据表还是上述状态没有变，最新版本trx_id=5，小于min_trx_id，说明事务已提交，那么可访问该事务，得到name=NO

这就是读已提交的MVCC过程，可以看到虽然读到的都是已提交的数据，但是同一个事务连续读了两次记录得到了不同结果，这就是不可重复读。

可重复读隔离级别

可重复读的MVCC判断与读已提交的一样，只是readView的生成逻辑不同：

  • 对于重复读来说，每次重复读会复用第一次生成的readview，来保证每次读到同一份数据

套用上面的情况，在第二次执行select name where id 1的时候，使用的是第一版的readview（即使事务5提交了），即trx_id=5，m_id=[5,6]，min_trx=5,max_trx_id=7，此时还是认为事务5未提交，读到的还是undolog里的xx，以此实现了可重复读的隔离级别。

可重复读能解决幻读问题吗？

不能，因为快照读(snapshot Read)和当前读(Current Read)的性质决定了永远会存在幻读的问题。

  • 快照读：事务查询时，不读取当前数据，而是读取数据历史版本（快照）。InnoDB通过MVCC来实现快照读，只返回事务开始时可见数据，即使其他事务修改，也不受影响
  • 当前读：读取数据最新版本，通过加锁保证数据一致性，即使事务2在当前读事务1之后，这个修改也会体现在当前读的结果中（即使事务2的trx_id 大于事务1的trx_id，事务是不可见的，但是当前读是可见数据的修改的）
    • 当前读的意义：例如update的时候，要确保当前行不能delete掉，所以要用当前读保证数据一致性，具体就得通过锁（Next-Key Locking等）锁定读范围内的记录和间隙（记录锁+间隙锁），防止其他事务在对应范围内插入

在这个场景下，由于快照读+当前读混合的读取，即使可重复读隔离级别下，还是出现了幻读的问题

事务开始的时候就要直接用SELECT ... FOR UPDATE来加锁，此时其他事务新增不了数据
