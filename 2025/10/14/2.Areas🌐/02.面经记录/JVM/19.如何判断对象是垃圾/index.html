19.如何判断对象是垃圾

发表于2025-10-14|更新于2025-10-14|2.Areas🌐02.面经记录JVM
|总字数:1.3k|阅读时长:3分钟|浏览量:

两种方式：

  • 引用计数法（Reference Counting）: 每个对象维护一个计数器，当引用技数器为0时，说明不再引用可以回收
    • 优势：简单，实时性高
    • 缺点：循环引用问题，即两个对象相互引用时，计数器永远不会为0
  • 可达性分析算法（Rechablity Analysys）：从一组GC Roots对象出发，遍历所有可达对象，凡是无法通过GC Roots到达的对象，均被视作垃圾
    • 优势：解决循环引用问题
    • 缺点：消耗一定资源进行标记

拓展

GC Roots的来源

  • 线程栈引用：每个线程栈中局部变量、参数…
  • 类的静态变量：类的静态变量（通过类加载器载入方法区）可以视作GC Roots
  • JNI全局引用：JNI可作为GC Roots

引用计数（平摊在运行过程中）

引用计数：每个内存内置计数器，每次被引用就+1，为0的时候就指导该单元无法被引用了（C），可以立即释放内存

深入分析后：

  • 缺陷：额外的计数存储控件，如果本身存储单元小，那么额外占用的空间就会明显

  • 优势：引用内存释放相当于会平摊到应用的日常运行中（内存释放时机更加均匀），适合内存敏感的场景（可达性分析的回收，并不会立马回收垃圾，而是需要等待下次GC）

我们发现这个方法还是很简单的，内存释放时机也很均匀，但痛点就是循环引用问题。

可达性分析（攒一段时间回收）

实际上就是标记-清除(mark-sweep)算法思想，标记可达对象，随后清除不可达对象（此处讨论如何标记）

  • 他是定期 / 内存不足时，进行垃圾回收，在GC Roots（根引用）开始遍历，将所有遍历到的标记为可达，然后回收不可达的对象

根引用包括全局变量、栈上引用、寄存器上的等。

我们发现，在内存不足时进行GC时，此时也是对象最多的适合，从根引用的遍历似乎是比较久的。

这是我们不愿看到的：可达性的思想是将垃圾堆积起来，再一次性清除，导致每次垃圾回收时消耗大量资源，影响了其他用户线程

因此引出了我们上一节那些收集器的思想：分代式垃圾回收、仅先标记根节点直达的对象再并发 tracing的方法

但都是优化方案，无法根本上解决

CPython 也使用引用计数，他怎么解决循环引用的（标记不可达-清除兜底）

首先我们知道像整型、字符串内部是不会引用其他对象的，所以不存在循环引用的问题，因此使用引用计数并没有问题。

那像 List、dictionaries、instances 这类容器对象就有可能产生循环依赖的问题，因此 Python 在引用计数的基础之上又引入了标记-清除来做备份处理。

但是具体的做法又和传统的标记-清除不一样，它采取的是找不可达的对象，而不是可达的对象。

Python 使用双向链表来链接容器对象，当一个容器对象被创建时，它被插入到这个链表中，当它被删除时则移除。

然后在容器对象上还会添加一个字段 gc_refs，现在咱们再来看看是如何处理循环引用的：

  1. 对每个容器对象，将 gc_refs 设置为该对象的引用计数。
  2. 对每个容器对象，查找它所引用的容器对象，并减少找到的被引用的容器对象的 gc_refs 字段。
  3. 将此时 gc_refs 大于 0 的容器对象移动到不同的集合中，因为 gc_refs 大于 0 说明有对象外部引用它，因此不能释放这些对象。
  4. 然后找出 gc_refs 大于 0 的容器对象所引用的对象，它们也不能被清除。
  5. 最后剩下的对象说明仅由该链表中的对象引用，没有外部引用，所以是垃圾可以清除。

最终循环引用的 A 和 B 都能被清理，但是天下没有免费的午餐，最大的开销之一是每个容器对象需要额外字段。

还有维护容器链表的开销。根据 pybench，这个开销占了大约 4% 的减速。

至此我们知晓了引用计数的优点就是实现简单，并且内存清理及时，缺点就是无法处理循环引用，不过可以结合标记-清除等方案来兜底，保证垃圾回收的完整性。

所以 Python 没有解决引用计数的循环引用问题，只是结合了非传统的标记-清除方案来兜底，算是曲线救国。

没有MVCC，系统必须频繁通过读写操作加锁来保证数据正确性（增加了锁的获取和释放开销），这种实现叫作LBCC（Lock-Based-Concurrent-Control） 扩展 场景：一个事务1正在执行，事务2修改了记录A，还未提交事务。 此时事务1想要读取记录A，因为事务2未提交，事务1无法读取到最新的记录A，因为如果事务2回滚了，就脏读了，而如果读修改之前的数据，由于事务2已经修改了，读不到之前的数据（缺少了版本控制的后果） 因此只能用锁阻塞等待事务2的提交，这就是LBCC机制（Locked-based） 如果有多版本的话，就不一样了。 事务 2 修改的记录 A，还未提交，但是记录 A 被修改之前的版本还在，此时事务 1 就可以读取之前的版本数据，这样读写之间就不会阻塞啦，所以说MVCC 提高了事务的并发度，提升数据库的性能。
