20.持久化机制

发表于2025-10-17|更新于2025-10-17|2.Areas🌐02.面经记录Redis
|总字数:1.9k|阅读时长:6分钟|浏览量:

有两种持久化机制：

  • RDB（Redis DataBase）快照：特定时间间隔内保存数据快照
    • 是紧凑的二进制文件，适合灾难恢复/备份，但崩溃时可能丢失最后一次快照后数据
  • AOF（Append Only File）日志：将每个写操作追加到日志文件中
    • 恢复更加精确，但体积更大且重写会消耗更多资源

Redis4.0新增RDB+AOF持久化机制

RDB持久化

RDB通过快照（dump.rdb）保存某个时间点的副本，通过快照可以更方便地进行主从复制

可以通过配置x秒内发生y次key变化，触发RDB持久化

优势：

  • 加载速度快：压缩的二进制，适合备份/灾难恢复
  • 占用资源低：持久化在Redis线程外进行，对主进程影响小

缺点：

  • 数据丢失风险：基于时间间隔快照保存，崩溃后可能丢失上次保存快照后的数据
  • 由于 RDB 是间隔性保存的快照，如果 Redis 崩溃，可能会丢失上次保存快照后的数据。

RDB 持久化命令：

  • save：主线程生成RDB，期间阻塞主进程进行，直到RDB结束
  • bgsave： Fork 子进程执行 RDB 生成，不会阻塞主进程（默认）

bgsave 流程（重点）：

  1. 检查AOF/RDB子进程是否正则执行，有返回错误
  2. 触发持久化：rdbSaveBackground，Fork子进程进行RDB操作，同时主进程响应其他操作
  3. 持久化完成，替换旧的RDB，子进程退出

注意事项（重点）：

  1. Fork操作本身会产生短暂阻塞（微妙级别），所以RDB严格来说不是完全非阻塞
  2. RDB快照备份，在Redis重启后速度得到保证，在大数据量下比AOF（写）快很多
  3. Fork是基于写时复制，参考：CopyOnWriteArrayList。

写时复制：

创建子进程时，会与主进程共享内存。

后续任一线程需要对内存修改时，才复制一块内存给修改线程单独使用

  • 649. Redis 在生成 RDB 文件时如何处理请求？

AOF 持久化详解

AOF将Redis写命令通过追加方式写入磁盘的AOF文件，记录完整的操作过程，重启后重新执行写命令来恢复。

优势：

  • 完整地在磁盘中记录了写命令，更可靠

缺点：

  • 生成文件更大，尤其是数据集大时
  • 数据恢复时间更耗时，因为要重新执行所有写指令

混合持久化

Redis 4.0 以后引入了混合持久化（ aof‐use‐rdb‐preamble 配置开启）

实现细节：AOF重写时（注意触发时机是AOF重写）：

  • 先生成RDB快照，写入新AOF文件头部
  • 生成过程中主线程的操作命令记录在重写缓冲区，其增量追加到新AOF文件，替换旧的AOF文件

当 Redis 通过 AOF 文件恢复数据时，先加载 RDB，然后再重新执行指令恢复后半部分的增量数据，这样就可以大幅度提高数据恢复的速度！

AOF 写回策略

写入磁盘的时机策略有三种：

  • always：每次写后立即调用fsync，保证最高数据安全性，但最大牺牲性能
  • everysec：每秒调用一次 fsync，将数据同步到磁盘。性能和数据安全性之间做了折中，最多会丢失1s的数据
  • no：操作系统决定何时写入磁盘，一般是经过一定时间 / 缓冲区满才写入磁盘，保证最高性能，数据安全较低，即崩溃时会丢失最多的数据

注：Always写回策略不能保证数据不丢失。 因为执行命令 + 写入AOF不是原子操作，执行命令后宕机无法通过AOF恢复对应的命令！

AOF 重写机制

AOF磁盘占用大，同时恢复速度慢，优化方案：AOF重写（压缩），用尽可能少命令生成等效的AOF。

注：AOF重写不是在原来的文件上写，是将每个键的最终值转换成写命令，写入新AOF文件

注： AOF 重写是后台操作，需要 fork 子线程，也是写时复制的策略。

拿 key A 举个例子，AOF 记录了每次写命令如 set A 1、set A 2、set A 3。实际上前面的 set A 1、set A 2 是历史值，我们仅关心最新的值，因此 AOF 重写就是仅记录数据的最终值即可，即set A 3，这样 AOF 文件就“瘦身”了。

AOF 重写流程如下：

  1. 创建子进程：BGREWRITEAOF命令创建一个子进程，负责 AOF 重写。

  2. 生成新的 AOF 文件：子进程根据Redis当前状态，将每个键的最新值转换为对应的写命令，并写入新的 AOF 文件。

    1. 如一个列表键，子进程会使用一条 RPUSH 命令将所有元素写入，而不是记录之前的多次操作。

  3. 处理新写入的命令：重写时，主进程正常进行写操作，为了避免数据不一致，主进程将新的写命令追加到现有AOF文件和缓冲区aof_rewrite_buf

  4. 合并新命令：子进程完成新AOF写入后，主进程将缓冲区新命令追加到新AOF文件，保证重写后的AOF包含最新操作

  5. 替换旧的 AOF 文件：将新 AOF 文件替换旧的文件，完成 AOF 文件的重写。

AOF 重写可以通过手动触发或自动触发：

1）手动触发：使用 BGREWRITEAOF 命令可以手动触发 AOF 重写。

2）自动触发：通过配置文件中的参数控制自动触发条件，参数如下：

  • auto-aof-rewrite-min-size：AOF 文件达到该大小时允许重写（默认 64 MB）。
  • auto-aof-rewrite-percentage：当前 AOF 文件大小相对于上次重写后的增长百分比达到该值时触发重写。

Redis 7.0 MP-AOF（Multi-Part Append Only File）

7.0 之前的 AOF 重写有三大问题：

  1. 内存开销：主进程追加的aof_buf 和缓存 aof_rewrite_buf 内容重复

  2. CPU 开销：主进程写入缓存aof_rewrite_buf，和向子进程发送缓存数据；子进程将缓存写入新AOF都是额外的CPU开销

  3. 磁盘开销：aof_buf写入到当前AOF，缓存数据写入新AOF，一份数据产生了两次写磁盘的开销

为了解决上述问题，Redis 7.0引入 MP-AOF（Multi-Part Append Only File）。

核心思路：将一个 AOF 文件拆分成多个文件（manifest），分治 + 增量的思想：

  • 基础文件（base file）： 数据的初始快照
  • 增量文件（incremental files），自基础文件创建以来的所有写操作（可以有多个）
  • 清单文件（manifest file）：统一跟踪和管理基础文件和增量文件，base+增量文件存放在一个单独的目录中

大致流程如下：

  • 重写过程中主进程的数据直接写入aof_buf缓存后，直接进入增量AOF文件，避免多个buf写入

  • 子进程也只需重写基础AOF，与主进程无交互，优化CPU开销

  • 重写完毕后，仅需更新manifest，加入新的增量AOF和基础AOF文件，通过异步删除旧的base+增量。

  • 更新manifest后，AOF重写结束。

AOF 文件修复

如果 AOF 文件因系统崩溃等原因损坏，可以使用 redis-check-aof 工具修复。该工具会截断文件中的不完整命令，使其恢复到一致状态。

10.事务
redis支持事务，但是事务的概念与mysql的ACID不同。Redis的事务指的是保证操作原子性，所有命令在原子操作中进行，不被打断。（注：redis的事务不支持回滚） 拓展 开始事务：MULTI开始，之后的命令进入队列 添加命令：添加命令，不会被立即执行，而是存储在队列 执行事务：EXEC原子性执行上述命令 取消事务：DISCARD清空命令，放弃事务 监视：WATCH监视一个/多个键，如果EXEC前被键修改了，EXEC不会执行，保证数据一致性。 Redis不支持回滚：所有命令都会被应用，即使某个步骤失败，依旧会执行，所以在设计时要充分考虑错误。 Redis2.6.5后，服务器在累计命令的过程中检测到错误，会拒绝执行事务，返回错误并丢弃该事务。 但是在执行过程中，仍会继续执行剩下的命令，不会进行回滚（这是和MySQL最不同的点，也不支持隔离级别设置，因为是单线程，只能是串行隔离级别），所以我们理解为是一个残血版的事务。
