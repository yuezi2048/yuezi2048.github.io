26.发生死锁如何解决

发表于2025-10-17|更新于2025-10-17|2.Areas🌐02.面经记录MySQL
|总字数:1.3k|阅读时长:5分钟|浏览量:

自动检测 / 回滚机制

  • MySQL锁自带死锁检测innodb_deadlock_detect，检测到死锁后，会自动回滚事务（通常解除回滚事务最少的那个）
  • 锁等待超时参数innodb_lock_wait_timeout，获取到锁的资源超过阈值后，释放锁进行回滚

手动kill

  • 通过命令找出被阻塞的事务，和对应的线程ID，手动kill掉及时释放资源

​

扩展

死锁日志

遇到死锁问题，SHOW ENGINE INNODB STATUS 命令获取到死锁的日志信息，定位死锁发生原因。

  • 发现事务回滚：WE ROLL BACK TRANSACTION (1)
  • 追踪问题：
    • 定位冲突位置：Record lock, heap no 3 PHYSICAL RECORD: n_fields 2; compact format; info bits 32（hex 00000002）对应 a=2一行
    • 分析锁：
      • 事务1请求X锁：delete from test where a = 2，等待释放索引a的X锁（index a ... trx id 2A8BD lock_mode X waiting）
      • 事务2持有X锁：insert into test (id,a) values (10,2)
      • 事务2请求S锁：由于 a 建立了唯一索引，因此需要 S 锁来进行重复 key 判断，因此日志显示等待释放X锁：(index a ... trx id 2A8BC lock mode S waiting)
      • 结论：事务 1 排着队等事务 2 释放 X 锁，而事务 2 又需要 S 锁，申请 S 锁的时候发现等锁的队伍前还有个 X 锁，这样一来就形成了相互依赖。
  • 解决问题

常见避免死锁或降低死锁的手段

  • 避免大事务：大事务拆分未多个小事务快速释放锁，降低死锁概率
  • 调整申请锁的顺序：更新数据要保证获得足够的锁
    • 可以先获取影响范围大的锁（如修改操作先得到排他锁，再获取共享锁，或者固定顺序访问数据）
  • 更改隔离级别：rr比rc多了间隙锁和临键锁，用rc可以减少死锁情况
  • 索引合理建立，减少加锁范围：命中索引会锁对应的行，否则会全表加锁，这样冲突大了，死锁概率提高
  • 死锁检测：适当调整锁等待时长

手动 kill 语句步骤

1）查找当前的事务和锁信息：

  • 查看当前执行事务和锁的信息，可以在输出中找到被阻塞的事务和线程ID

也可以利用INFORMATION_SCHEMA 中的 INNODB_LOCKS 和 INNODB_LOCK_WAITS 表，查看当前锁和锁等待情况，得到事务ID。

然后通过 INFORMATION_SCHEMA 的 innodb_trx 可以找到事务ID和线程 ID 的对应关系。

2）手动终止该事务：

确定线程ID后，运行kill命令即可杀掉线程

INNODB_LOCKS 、 INNODB_LOCK_WAITS 和 INNODB_TRX 示例

INNODB_LOCKS 查询当前锁的信息

示例结果

LOCK_ID LOCK_MODE LOCK_TYPE LOCK_TABLE LOCK_INDEX LOCK_OWNER LOCK_DATA
12345678:1 X RECORD my_database/my_table PRIMARY 123456 1
12345678:2 S RECORD my_database/my_table PRIMARY 123457 2

字段解释：

  • LOCK_ID: 唯一标识每个锁。
  • LOCK_MODE: 锁的类型（X：排它锁，S：共享锁）。
  • LOCK_TYPE: 锁的作用对象类型（如 RECORD 表示行锁）。
  • LOCK_TABLE: 被锁定的表。
  • LOCK_INDEX: 相关索引。
  • LOCK_OWNER: 拥有该锁的事务 ID。
  • LOCK_DATA: 被锁定的数据行的主键值。

INNODB_LOCK_WAITS 查询锁等待情况

示例结果

REQUESTING_TRX_ID HOLDING_TRX_ID LOCK_ID LOCK_TYPE
123456 123457 12345678:1 RECORD
123458 123459 12345678:2 RECORD

字段解释

  • REQUESTING_TRX_ID: 请求锁的事务 ID。
  • HOLDING_TRX_ID: 当前持有锁的事务 ID。
  • LOCK_ID: 与 INNODB_LOCKS 表中的锁相对应。
  • LOCK_TYPE: 请求的锁类型。

INNODB_TRX 查询线程 ID 示例

示例结果

trx_state trx_started trx_mysql_thread_id trx_query trx_id
LOCK_WAIT 2024-09-25 10:00:00 1234 SELECT * FROM my_table WHERE id = 1 123456

字段解释：

  • trx_state: 事务的当前状态。在此示例中，LOCK_WAIT 表示该事务正在等待锁。
  • trx_started: 事务开始的时间。在此示例中，事务于 2024-09-25 10:00:00 开始。
  • trx_mysql_thread_id: 与该事务关联的 MySQL 线程 ID，可以用来查找该事务的更多信息。
  • trx_query: 当前事务正在执行的查询。在此示例中，它正在执行一个简单的 SELECT 查询。
  • trx_id: 事务的唯一标识符。在此示例中，事务 ID 为 123456。
