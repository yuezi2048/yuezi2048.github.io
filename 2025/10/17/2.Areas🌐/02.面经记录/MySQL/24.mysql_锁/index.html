24.mysql_锁

发表于2025-10-17|更新于2025-10-17|2.Areas🌐02.面经记录MySQL
|总字数:2.5k|阅读时长:8分钟|浏览量:

范围分类：

  • 行级锁（Row Lock）*：特定行加锁，允许其他事务访问不同行，高并发场景
  • 表级锁（Table Lock）*：整个表加锁，其他无法对该表读写，适合保证完整性的小型表
  • 意向锁（Intention Lock）：表锁的优化，表示事务对某行加锁的意图，分为意向共享锁IS / 意向排他锁IX，行级锁和表级锁结合
  • 元数据锁（Metadata Lock, MDL）：保护数据库对象（表/索引）的元数据，防止DDL时其他事务进行修改

属性分类

  • 共享锁（Shared Lock）*：允许多个事务持有共享锁S，但不允许修改。只有释放共享锁后，其他事务才能获得排他锁
  • 排它锁（Exclusive Lock）*：只允许单个事务持有排他锁X，对资源读写，其他事务在获得排他锁前无法获得资源

按照行锁分类：

  • 记录锁（Record Locks）：锁对应的索引记录
  • 间隙锁（Gap Lock）*：索引两个记录之间加锁，防止其他事务在这个间隙中插入新记录，避免幻读（不同于记录锁，不锁定具体行，锁定的是行与行的空隙）
  • 临键锁（Next-Key Lock）*：行级锁和间隙锁结合，锁定具体行和前面的间隙，保证一个范围不出现幻读，常用于支持可重复读的隔离级别
  • 插入意向锁（Insert Intention Lock）：等待间隙的锁，指示事务打算在某个间隙插入，允许其他事务进行共享锁，但是插入时会阻止其他的排他锁

特殊的表集锁：

  • 自增锁（Auto Increment Lock）：插入自增列时，加锁保证自增值的唯一性，防止并发导致的冲突，通常在插入使用保证自增ID唯一。

共享锁和排他锁

在 MySQL 中锁可以分为两大类，分别是 shared (S) locks 和 exclusive (X) locks。

  • S锁（共享锁），在读取记录的时候获取S锁，多个事务可同时获取S锁，互相不冲突
    • SELECT ... LOCK IN SHARE MODE; 这种读取需要对记录上 S 锁。
  • X锁（排他锁），修改记录的时候获取X锁，只能一个事务获取X锁，其他事务需要阻塞等待
    • SELECT ... FOR UPDATE; 需要对记录上 X 锁。
冲突 S X
S 不冲突 冲突
X 冲突 冲突

MyISAM 引擎仅支持表锁，而 Innodb 即支持表锁也支持行锁。

  • LOCK TABLES yes READ 是对 yes 这个表上 S 锁。
  • LOCK TABLES yes WRITE 是对 yes 这个表上 X 锁。

注：一般情况，我们不会用表锁，性能太低。可能在用DDL语句（ALTER TABLE）的时候会考虑表锁，防止查询和修改

而在此基础上，引出 MDL 的锁，即 Metadata Locks（元数据锁）。

元数据锁

我们可以分成读锁S（MDL_SHARED）和写锁X（MDL_EXCLUSIVE），具体的作用：

1）解决并发时DDL和DML操作冲突（DDL + DML并行冲突）

  • 一个事务要进行表的结构性更改DDL（ALTER TABLE），元数据锁X会阻止其他事务对表进行操作，直到DDL完成
  • 一个事务要进行数据操作（SELECT INSERT UPDATE DELETE），元数据锁S阻止其他事务对表进行结构性的更改

2）保护元数据一致性（DML + DML冲突）

  • 确保在执行 DDL 操作（如 CREATE TABLE、DROP TABLE、ALTER TABLE）时，元数据不会被其他事务同时修改。

业务如果用到了表锁，我们还要考虑表锁和行锁的冲突。且InnoDB加表锁后，需要判断表里面是否有行锁，要通过遍历吗？不需要，引出意向锁标记行锁

意向锁

引出意向锁（Intention Locks）

  • IS（Intention Shared Lock），共享意向锁，表中某条记录上S锁后，会在表里上IS锁，表示该表有S锁
  • IX（Intention Exclusive Lock），独占意向锁。表中某条记录上X锁后，会在表里上IX锁，表示该表有X锁

具体就可以通过IS和IX来通过行锁来判断表锁是否冲突了（不需要遍历记录即可判断）：

  • 如果事务要上表级别的S锁，如果没有IX锁（有IS锁/没锁），那就可以上S锁
  • 如果事务要上标记别的X锁，如果没有IX且没有IS（没锁），说明表中记录没有加锁，可以上X锁

S 不冲突 冲突 不冲突 冲突
X 冲突 冲突 冲突 冲突
IS 不冲突 冲突 不冲突 不冲突

接下来再来看看 MySQL 中的行锁有哪几种。

主要有三种：

  • 记录锁
  • 间隙锁（Gap Locks）
  • 临键锁（Next-Key Locks）

记录锁

锁当前的记录，作用于索引（InnoDB就算没有主键也有隐藏索引），因此记录锁锁定索引记录。

举例：事务A执行： SELECT * FROM yes WHERE name = 'xx' FOR UPDATE;，此时name=xx这条记录就被锁住了，其他事务无法增删改name=xx的记录。即在事务A未提交前，另一个事务 B 执行 insert into yes (name) values ('xx')，此会被阻塞，这个很好理解。

追问：如果另一个事务 C 执行了 insert into yes (name) values ('aa')，这个语句会被阻塞吗？

在可重复读级别下：

  • name如果没有索引，只能全表找聚簇索引，此时会把整张表锁住。因此事务C将被阻塞
  • name如果有索引，是正常锁了那一条记录，则不会被阻塞

教训：没索引的列不要轻易加行锁！

间隙锁和临键锁

进一步优化，想要预防幻读问题，那么就需要对未存在的记录加锁。

解决方案：

  • 间隙锁（Gap Locks）：如锁定(3,5) 这个区间

  • 临键锁（Next-Key Locks）： 记录锁+间隙锁： Next-Key Locks 是一个前开后闭的区间(3,5]，这样能防止查询 id=5 的这个幻读。

比如此时有 1、3、5、10 这四条记录，为了完整划分间隙，数据页中还有两条虚拟的记录，分别是 Infimum 和 Supremum。

可以看到两个间隙中间的就被锁上了，比如我把 3 和 5 之间的间隙锁了，此时要插入 id = 4 的记录，就会被这个间隙锁给阻塞了，这样就实现了锁定未插入的记录的需求！

注：

  • 间隙锁之间不会冲突，因为间隙锁的目的是防止其他数据插入到间隙中，两个间隙中锁相同的间隙没有问题
  • 间隙锁可以显式禁用，该锁在RR隔离级别生效，在RC模式下就禁用了，即对于搜索和索引扫描是禁用的，只用于外键约束和重复键的检查

插入意向锁

插入意向锁（Insert Intention Locks），它也是一类和间隙相关锁，这个锁是用于等待某个间隙的而非锁定间隙。

比如上面举例的 id = 4 的那个事务 C ，由于被间隙锁给阻塞了，所以事务 C 会生成一个插入意向锁，表明等待这个间隙锁的释放。

注：插入意向锁之间不会阻塞，因为他也是等待某个间隙是否被释放，而不是锁定资源防止别人访问

回到最本质的实现，锁其实在内存里是单独的一个结构，每个事务来对某个记录/间隙上锁来争抢资源。而对于没有抢到的事务，就是等待状态，分配一个等待状态的锁对象，等相关事务释放掉锁之后，就会查找等待该资源的锁结构，然后选一个让它获得资源并唤醒对应的事务使之得以执行。

基于上述逻辑，对于等待间隙锁的插入事务，建立的锁结构，也就是插入意向锁，使得间隙锁的事务在释放间隙锁的时候，才能得以找到那些等待插入的事务，然后进行唤醒，而由锁的类型也可以得知是插入意向锁，之间不需要阻塞，所以可以一起执行插入。

Auto-Inc Lock

Auto-Inc Lock是特殊的表级锁，用于自增列插入数据，每次插入数据的时候，加入Auto-Inc Lock，然后自增列分配递增值，执行结束后，释放Auto-Inc Lock。

MySQL 5.1.22 版本之后，引入了互斥量来进行自增减的累加

  • 性能高于Auto-Inc Lock（因为在获得递增值之后就可以释放锁，Auto-Inc Lock是插入完成后才能释放锁）
  • 但并发数据一致性不如Auto-Inc Lock，并发插入时，基于 statement -based binlog 复制的时候，自增的值顺序无法把控，可能会导致主从数据不一致。

因此 MySQL 有个 innodb_autoinc_lock_mode 配置，有三个值：

  • 0，只用 Auto-Inc Lock。
  • 1，默认值，对于插入前已知行数的插入，用互斥量，如果不知道具体插入数，则用 Auto-Inc Lock，从而保证statement -based binlog 复制也是安全的
  • 2，只用互斥量。

扩展学习

Predicate Locks，了解用于空间索引隔离级别即可

InnoDB 是支持空间数据的，所以有空间索引，为了处理涉及空间索引的操作的锁定，next-key locking 不好使，因为多维数据中没有绝对排序的概念，因此不清楚 “下一个” key 在哪。

所以为了支持具有空间索引的表的隔离级别，InnoDB 使用谓词锁。

空间索引包含最小边界矩形（MBR）值，因此 InnodB 通过在用于查询的 MBR 值上设置谓词锁定，使得 InnoDB 在索引上执行一致性读， 其他事务无法插入或修改与查询条件匹配的行。
