46.内存碎片化+优化

发表于2025-10-27|更新于2025-10-27|2.Areas🌐02.面经记录Redis
|总字数:457|阅读时长:1分钟|浏览量:

内存碎片化：内存使用中出现小块空间被闲置，无法被有效利用。

  • Redis默认使用jemalloc作为内存分配器，是按照固定大小分配的
    • 如分配了12kb，但实际只需要8kb，这就造成了4kb的内存碎片
  • 在频繁创建 / 删除大量数据的时候，会导致内存块大小/位置不连续，会堆积更多

排查：通过 INFO memory 命令查看内存碎片率(mem_fragmentation_ratio)：

  • used_memory：Redis 实际使用的内存，单位是字节。
  • used_memory_rss：从操作系统的角度来看，Redis 占用的总内存量（含内存碎片），单位是字节。
  • mem_fragmentation_ratio = used_memory_rss / used_memory，所以大于 1 就代表有内存碎片

如果ratio值较大，就需要考虑内存碎片的清理了。如果小于 1 问题也很大，说明 redis 已经使用了 swap 用上磁盘空间了，性能会变得很差。

解决方案是：重启 / 优化数据结构类型；配置自动整理 activedefrag + 手动清理 MEMORY PURGE

扩展知识

如何解决内存碎片

1）最简单的解决方法是定期重启 Redis 服务，这样可以消除内存碎片并优化内存的布局，但是会导致服务不可用。

2）Redis 4.0 及以上版本引入了内存碎片整理功能。通过配置 activedefrag 选项，Redis 可以在运行时尝试整理内存碎片，将小的内存块合并为更大的块。

3）通过优化数据存储结构和类型。如：使用 ListPack 替代 ziplist。

4）利用 MEMORY PURGE 命令手动清理碎片，但是这个命令会阻塞主线程。

10.事务
redis支持事务，但是事务的概念与mysql的ACID不同。Redis的事务指的是保证操作原子性，所有命令在原子操作中进行，不被打断。（注：redis的事务不支持回滚） 拓展 开始事务：MULTI开始，之后的命令进入队列 添加命令：添加命令，不会被立即执行，而是存储在队列 执行事务：EXEC原子性执行上述命令 取消事务：DISCARD清空命令，放弃事务 监视：WATCH监视一个/多个键，如果EXEC前被键修改了，EXEC不会执行，保证数据一致性。 Redis不支持回滚：所有命令都会被应用，即使某个步骤失败，依旧会执行，所以在设计时要充分考虑错误。 Redis2.6.5后，服务器在累计命令的过程中检测到错误，会拒绝执行事务，返回错误并丢弃该事务。 但是在执行过程中，仍会继续执行剩下的命令，不会进行回滚（这是和MySQL最不同的点，也不支持隔离级别设置，因为是单线程，只能是串行隔离级别），所以我们理解为是一个残血版的事务。
