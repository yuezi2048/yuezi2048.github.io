12.索引一定有效吗-如何排查

发表于2025-10-13|更新于2025-10-13|2.Areas🌐02.面经记录MySQL
|总字数:1.2k|阅读时长:3分钟|浏览量:

不一定有效，是否使用索引要根据MySQL CPU I/O的计算成本（如回表开销），来确定是辅助索引还是全表扫描。

  • 场景：查询不包含索引列，低基数列索引效果不佳，查询条件复杂且不匹配索引顺序，这些场景下，索引并没有贡献性能
  • 场景2：在一些小表中，MySQL可能选择全表扫描，不用维护额外的索引，开销可能更小

排查方案：EXPLAIN，在查询语句前加上EXPLAIN查看MYSQL的执行计划，可以知道：是否使用索引、使用了哪个索引、估算行数等信息：https://blog.csdn.net/dennis211/article/details/78170079

  • type（访问类型）：查询使用的访问方法（ALL index ref ref_eq range const），使用索引时一般是index（全表扫描二级索引）、ref（常量等值匹配）、range const(主键条件查询)，否则就是ALL全表扫描
  • key（使用的索引）：显示索引名称，如果是NULL表示没有使用索引
  • rows（扫描行数）：查询扫描的行数，需要评估扫描量

拓展

索引失效的场景汇总：

  • 联合索引不符合最左匹配前缀：如select * from user where age=10 and id=1;不符合name_age_id联合索引的最左原则

    • 但在特定的场景下（覆盖索引）还是会用到索引的（type=index）

  • 索引使用了运算：select * from user where id + 3 = 8;，运算导致全表扫描

  • 索引使用了函数：select * from user where lower(name) like 'zhang%' 函数导致全表扫描

    • 字段类型错误使用（导致的隐式转换）*：select * from user where name=1，此时相当于 select * from user where CAST(name as signed int)=1，这一条比较隐蔽

  • like的错误使用：select * from user where lower(name) like '%zhang%' ，最左无法匹配，无法定位索引

  • or的错误使用：select * from user where name=‘zhang’ or age=18，age没有索引，也会失效

  • 不同的参数导致索引的失效：因为需要命中索引，这就导致在一些复杂查询（联表 子查询 回表场景）可能选择全表查询更好

    • 如商品从订单表查询，传入的商品id是热门商品且占据了80%的销量，那么这个查询很有可能是全表查询，相反，如果是冷门商品，可能就是走索引查询，同上一节定时任务的场景。

  • 不同字段比较：select * from user where id > age

  • order by错误：order by后面不是主键，或者不是覆盖索引，就会导致失效

用了索引还有可能变慢！

场景：

执行SELECT * FROM yes.t1 where f3=11，此时不符合最左匹配原则，但由于索引下推机制，此时还会走索引（省流：在idx_f2_f3的二级索引树中叶子节点遍历了一遍然后筛选返回给server），通过EXPLAIN可以验证我们的想法（type=index）。

思考：到底什么叫用了索引 什么叫全表扫描

InnoDB是依赖聚簇索引构建数据的，如果建立了二级索引就可以通过其来找数据。

平日里所说的全表扫描底层是通过主键索引找到最左的叶子节点然后往右遍历扫描。

而用上索引（type >= index级别）的意思是在索引树上查找数据所在的页，即使是对其全表扫描也算用上了二级索引，上面的案例就是覆盖索引的例子，只需要全扫描一遍二级索引就可以，type是index级别的

此时如果改成f2的查询语句，符合了最左匹配原则，则是变成了ref的常量等值匹配

总结，用了索引的情况：

  • 通过主键索引查找
  • 通过非主键索引查找
  • 全扫描二级索引（上述核心案例）

追问：为什么不直接扫描主键索引还要全扫描二级索引

回到一开始的需求，用索引是要考虑收益大于性能开销的

主键索引数据行包括事务ID、回滚指针等，二级索引只存储了主键值和索引列，所以相对二级索引开销小。

因此在能覆盖返回值的情况下，用二级索引会更好。

但是在一些覆盖不了的场景下（f4），那就才会用主键索引，此时有个回表的过程（type=ALL），效率就低了

15.为什么MySQL选用B+树作为索引
优势： 查找性能：B+树是自平衡树，根节点到叶子节点长度路径相同，在插入和删除时有分裂/合并操作保持树的平衡，同时还会有冗余节点，使得删除时树的变化小，更加高效 查找、插入、删除等操作时间复杂度O(logn)，在大数据量情况下也有很好的响应时间 高度增长速度稳定，减少磁盘I/O次数：不同于红黑树（数据越多高度增长越快），B+树是多叉树，非叶子节点仅保存主键/索引值和页面指针，使得每一页可以容纳更多的数据，在内存中可以放更多索引，容易命中缓存，减少I/O次数 范围查询能力：B+树由于叶子节点通过链表连接，从根节点到叶子节点确定范围起点后，只需要扫描链表即可遍历后续的数据，非常高效 扩展： B树 vs B+树 B树每个节点存储完整数据，B+树非叶子节点只存储key和指针，使得B+树可以在内存中放更多的索引页，减少磁盘I/O B+树叶子节点组成链表，更利于区间查找，B树只能遍历 B+树查询更稳定、平均，每次都是从根节点扫描到叶子节点，B树可能在非叶子节点找到数据就返回了
