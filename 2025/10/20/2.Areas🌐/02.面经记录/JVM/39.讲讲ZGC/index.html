39.讲讲ZGC

发表于2025-10-20|更新于2025-10-20|2.Areas🌐02.面经记录JVM
|总字数:1.6k|阅读时长:5分钟|浏览量:

ZGC（Z Garbage Collector）是Java11引入的低延迟垃圾回收器，支持大内存（4TB 理论16TB）的应用程序，并保持低GC延迟。

特点：

  • 并发收集：ZGC收集过程大部分和应用程序同时进行
  • 指针压缩：减少内存占用
  • 毫秒级停顿：停顿主要是在GC roots上，受堆大小影响小，且有一系列优化机制（如果重新标记超过1ms会再次进入并发标记阶段）

注：

  • ZGC超过G1很重要的一点在于并发转移
  • 到那时要注意内存分配过快导致的线程阻塞（设置 ParallelGCThreads 并行线程和 ConcGCThreads和并发线程加快回收速度，不能设置过多）
  • ZGC的STW和GC Roots有关，在多线程、动态加载ClassLoader频繁的场景下也会增加ZGC停顿时间

机制：

  • 分代收集：分代年轻代和老年代（JDK21后支持）
  • 记录和重标记：记忆集跟踪对象引用关系，在GC过程中快速识别活动对象。

染色指针和读屏障是 ZGC 能实现并发转移的关键所在

拓展

ZGC概览

ZGC的核心目标是低延迟，不管堆有多大 / 存活对象有多少

关键字：并发、基于Region、整理内存、支持NUMA、用了染色指针、用了读屏障，对了 ZGC 用的是 STAB。

  • Concurrent：和应用程序并发执行，有10个阶段，只有3个阶段是短暂STW
  • Region-based：支持细粒度的内存分配（2MB 32MB X*MB[内存系统]），回收的策略是优先收集小区，中、大区尽量不回收

  • 非一致内存访问 （Non-Uniform Memory Access，NUMA）：G1原先不支持现在也支持，小分区优先交给本地内存，中大对象交由操作系统管理，减少内存不平衡的情况（远程内存效率低下）
    • 前端总线-北桥-内存————SMP （Symmetric Multi-Processor）一致内存访问（Uniform Memory Access， UMA ）——非一致内存访问

  • 染色指针：Marked0、Marked1、Remapped、Finalizable四位标识对象状态
    • 42-45位标记位，基于多重映射实现多个标志位都能指向同一对象
    • 虽然X86_64指令集是64位，但总线只有48条，所以理论上是16TB，这里只用了4TB，给到了42+4位

  • 使用读屏障：保证读取时引用的正确性（判断对象是否被转移），使其可以并发转移对象
    • 读取应用对象时上读屏障，基于指针标志判断对象是否被转移，转移了就修正对象引用（“自愈”，不仅赋值的引用a是最新的，自身引用也修正了obj.foo对象本身，本质上是修正forwardingTable上的引用。）

ZGC过程分析

在G1基础上加入了基于重定位的并发转移

  • 标记：从根开始标记所有存活对象
  • 转移：选择部分活跃对象转移到新的内存空间上
  • 重定位：因为对象地址变了，所以之前指向老对象的指针都要换到新对象地址上。

注：第一次GC标记后，转移对象，不会重定位，只记录转移到哪里，后续发现对象被转移了才重定位。

  • 初始标记：STW，标记根节点可达对象，压倒标记栈中（重置TLAB 判断清除软引用…）
  • 并发标记：并发遍历对象图并统计region存活对象
    • 标记栈只有一个，但并发标记线程有多个，这样减少之间的竞争，每个线程分到不同的标记带（类比HashMap1.7的segment）
    • 每个线程速度不一样，标记快的线程会窃取别人的任务执行实现负载均衡（ForkJoinPool工作窃取机制）
  • **再标记：STW，**修改对象引用标记漏标的情况
    • ZGC目标是低延迟，如果该阶段再标记延迟时间过长，会重新执行并发标记
    • 实际上，这个阶段还会做非强根并行标记：系统字典 JVMTI JFR 字符串表（注：有些强根不能并发）
  • 非强引用并行/并发处理：软引用、弱引用、虚引用并发处理
  • 初始转移：
    • 重置转移集：在新一轮GC中，对转移分区的映射集复位（标记阶段重定位——即读屏障用到的forwardingtable映射集）
    • 回收无效分区：回收物理内存释放后的无效虚拟内存（分区需要在新一轮标记后才能释放）
    • 选择待回收分区：同G1，筛选垃圾多的分区，作为本次垃圾回收的集合
    • 初始化待转移集合转移表：初始化forwardingTable
    • 初始转移（STW）：从根集合触发，如果在分区集合则在新分区分配空间，否则标记为Remapped，只转移根直接可达对象
  • 并发转移：和并发标记类似，优化了G1转移对象需要STW的问题，延迟少了很多

分析染色指针的标记位：M0、M1、Remappe

  1. Remapped 是对象的 “正常状态视图”，GC 没运行时所有对象都处于这个视图，应用线程平时访问的就是它；在标记时，如果是活跃状态，则会置为状态试图的值（M0/M1，否则就是不活跃）
  2. M0 和 M1 是 “交替使用的活跃标记位”，每轮 GC 只用其中一个（比如本轮用 M0 标活跃，下轮就用 M1），避免上轮未转移的旧标记和本轮新标记混淆；
  3. 标记阶段用当前轮的标记位（M0/M1）标活跃对象，转移阶段把这些对象切回 Remapped，下轮 GC 换另一个标记位重复这个过程。

总结：引入地址视图（此时地址指针的标记位，标记位为 M0，就是 M0 视图。）的概念，交替判断本轮垃圾回收的对象（M1）和上一轮回收被标记但没有被转移的对象（M0）和不活跃对象Remapped

总结：

  • 多阶段并发+短暂STW实现低延迟——核心技术是指针染色 + 读屏障
  • 现代垃圾回收期在往并发上靠，减少停顿时间，要注意内存分配速率，因为并发导致一次垃圾回收总的时间变长了。
  • 同时内存分配不宜过快，需要预留内存空间或者说要更大的内存空间来应对快速的分配速率
