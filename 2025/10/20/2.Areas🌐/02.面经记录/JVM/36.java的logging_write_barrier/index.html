36.java的logging_write_barrier

发表于2025-10-20|更新于2025-10-20|2.Areas🌐02.面经记录JVM
|总字数:653|阅读时长:2分钟|浏览量:

**logging write barrier（日志写屏障）**用于追踪记录对象引用的变化（尤其是并发和增量式的垃圾回收中），更好地维护卡表等记忆集结构，避免全堆扫描，具体：

  • 记录引用记录变更（写屏障）：引用关系变化时，logging barrier会记录变更，便于垃圾收集器在增量回收时处理对象的引用
  • 减少对应用程序影响（日志）。写屏障会消耗应用程序性能，因为写屏障是在引用赋值时执行的逻辑；而logging是将写屏障要执行的逻辑搬到后台执行，降低对后台影响
  • 优化维护记忆集合（总结）：从而更好地维护卡表 / 记忆集（G1），优化跨代引用 / 跨区域引用的追踪

拓展

logging write barrier

write barrier的变体，可以理解为在AOP的基础上引入了一个日志系统+异步的优化

  • Pre-Write Barrier：在对象引用改变前触发，用于记录引用变更前的状态。
  • Post-Write Barrier：在对象引用改变后触发，用于记录引用变更后的状态。
  • Logging Write Barrier：属于 post-write barrier 的一种变体，用于记录和追踪引用变更的日志。该日志可以在 GC 阶段异步处理这些引用变更。

具体来讲，每次写屏障时，只需记录log到队列中，后台进程从队列中取出信息完成后续操作（异步）

G1的SATB write barrier为例，每个java线程有一个独立、定长的SATBMarkQueue，写屏障只需要把旧引用压入到队列中，满了之后会加入到全局SATBMarkQueueSet，后续后台进程则进行扫描，超过阈值后则tracing

该方法在不同GC上的体现

  • G1：logging write barrier更新Remembered Set，会记录每次对象引用的变更，帮助垃圾收集器在Mixed GC阶段快速定位跨代引用。
  • Shenandoah 和 ZGC：这些垃圾收集器也是基于并发，write barrier 用于保持对象引用的正确性，确保垃圾收集和应用线程能够同时运行。write barrier 是这两个垃圾回收器并发回收阶段不可或缺的机制。

与读屏障的区别

  • 写屏障（Write Barrier）：用于记录和追踪对象引用的写操作，主要用于处理对象之间的引用变更。
  • 读屏障（Read Barrier）：用于在读取对象引用时，确保对象引用的正确性，常见于并发垃圾回收器，用于处理对象的并发标记或压缩。

40.垃圾回收调优的目标
目标：最短暂停时间和高吞吐量 最短暂停时间：首要目标是减少程序停顿时间，保证GC过程中保证应用响应能力，尤其是实时/高并发应用 高吞吐量：单位时间内完成更多的业务处理，通过合理的GC配置，减少GC频率和时间，提升整体性能 方案1：GC停顿100ms，每秒停顿5次（时延低） 方案2：GC停顿200ms，每秒停顿2次（吞吐量高） 吞吐量：用户代码时间 /（用户代码执行时间 + 垃圾回收时间） 两者往往不可兼得，要明确实际应用的目标。
