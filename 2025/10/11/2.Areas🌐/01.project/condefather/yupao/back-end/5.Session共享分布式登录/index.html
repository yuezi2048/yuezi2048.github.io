5.Session共享分布式登录

发表于2025-10-11|更新于2025-10-11|2.Areas🌐01.projectcondefatheryupaoback-end
|总字数:612|阅读时长:2分钟|浏览量:

改造用户中心

单机登录改为分布式session登录

spring security 可以有更细粒度的权限

注：用jwt其实也是有缺陷的：https://zhuanlan.zhihu.com/p/108999941（适合短期验证的场景）

我们使用技术要考虑需求和成本

为了演示分布式，打包项目，然后换一个端口运行后端

发现由于是同一个域名下，在另外一个项目运行后，sessionID会发生变化。

中session的时候要注意范围，cookie.domain（aaa.yupi.com和bbb.yupi.com）

特别注意：如果读不到内容，可能是由于域名的限制！！

如果要共享cookie，我们可以种一个更高层的公共域名，比如yupi.com

这里是设置只有localhost域名下可携带cookie

为什么服务器A登录后，请求发到服务器B，却不认识用户了？

用户在A登录，session登录信息在A上，第二次请求B时，B没有用户信息，所以不认识。

解决：共享存储，将登录信息放到A和B都信任的服务器上即可

  • Redis（基于内存的键值对），此处选择Redis（用户信息 / 登录逻辑频繁，Redis读写性能高，单机qps 5w~10w）
  • MySQL
  • 文件服务器ceph

Redis改造

  • 引入redis

redis管理工具 quick redis

  • spring-session-data-redis依赖：整合spring-session和redis，实现自动将session存储到redis中
  • 修改spring-session配置，默认是none，表示存储在服务器，可以使用redis

但是3.x.x版本，没有store-type配置，编写配置类

运行后发现报错：Cannot serialize

Description The server encountered an unexpected condition that prevented it from fulfilling the request.

发现是序列化问题，我们对User进行序列化修复即可

可以发现redis对应上了我们的session域的登录态键，以及生成、存活、最近访问的时间

5.编译执行和解释执行
编译执行：程序执行之前，编译器将源代码直接编译为机器码，然后在CPU上运行 优势：编译后速度快，机器代码针对目标平台直接生成，不需要运行时再编译 缺点：对每个平台必须重新编译，同时机器代码较难调试和逆向工程 解释执行：在运行时通过解释器逐行翻译并执行，Python Ruby等也是解释语言 优势：跨平台好，每个平台通过对应平台的解释器运行，开发周期更短 缺点：运行速度慢，每次执行时需要动态翻译和解释 JVM使用的是编译执行 + 解释执行 混合的方式 解释执行：逐行结束时执行字节码，一般是初次运行时，有助于跨平台性。 即时编译（Just-In-Time Compiler, JIT）：程序运行时将热代码编译为本地代码，避免反复解释，提升性能 通俗理解：编译执行时是直接翻译为中文，运行的时候直接看中文，解释执行时看一行理解一行的实时翻译模式。 因此编译执行更快，但是解释执行的翻译兼容性更好。 JVM的逻辑是：正常情况解释执行，在热点代码通过JIT即时编译成机器码，进行编译执行。 扩展 JIT（Just-In-Time） AOT（Ahead-Of Time）
