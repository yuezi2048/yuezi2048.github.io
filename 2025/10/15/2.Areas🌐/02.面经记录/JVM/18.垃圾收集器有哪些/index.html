18.垃圾收集器有哪些

发表于2025-10-15|更新于2025-10-15|2.Areas🌐02.面经记录JVM
|总字数:2k|阅读时长:6分钟|浏览量:

细分为新生代收集器 + 老年代收集器

新生代垃圾收集器：

  • Serial收集器：单线程收集器
    • 触发**Stop-The-World（STW）**操作，即GC时所有线程暂停
    • 场景：单线程小型应用 / 客户端模式
  • ParView收集器：Serial多线程版本，并行处理垃圾收集
    • 与CMS配合使用时，选择ParNew收集器作为新生代收集器
    • 场景：多处理器环境，配合CMS使用
  • Parallel Scavenge收集器：吞吐量收集器，追求最大化CPU时间利用率（基于标记-复制算法）
    • 并行处理新生代垃圾，适合大规模的后台处理，关注吞吐量而不是延迟
    • 场景：大规模密集型计算后台任务等需要较高吞吐量场景

老生代垃垃圾收集器：

  • Serial Old收集器：Serial收集器老年代版本，通过标记-整理（Mark-Compact）算法垃圾处理

    • 场景：单线程 / 低内存场景，配合Serial收集器使用

  • Parallel Old收集器：Parallel Scavenge收集器老年代版本，多线程并行标记-整理算法

    • 场景：大规模并行计算场景，适合高吞吐量要求

  • CMS（Concurrent Mark-Sweep）收集器：并发标记-清除收集器，追求低延迟，减少GC停顿时间

    • 缺点：内存碎片问题，并发时产生Concurrent Mode Failure，导致Full GC
    • 场景：响应要求高场景，如Web服务 / 电商平台

  • G1（Garbage First）收集器：取代CMS的低延迟垃圾收集器，提供可预测的停顿时间

    • 分区管理内存，通过垃圾收集器收集最有价值回收的区域，同时标记-整理算法解决了CMS内存碎片问题
    • 场景：大内存、多CPU的服务器应用，在延迟和响应时间敏感场景下表现出色

  • ZGC（Z Garbage Collector）收集器：低停顿+高吞吐量垃圾收集器（停顿时间小于10ms）

    • 场景：需要管理大堆内存，和低延迟要求极高的应用

  • 在这个关系图中，可以发现除了我们拓展中的组合之外，收集器都可以配合Serial Old使用

拓展

垃圾收集算法：

  • 标记-清除 算法：主要用于CMS，标记存活对象后，清除不可达对象，缺点是产生内存碎片
  • 标记-整理 算法：主要用于G1、Parallel Old，标记存活对象后整理，避免内存碎片

垃圾回收调优思路：

  • 吞吐量调优：降低垃圾回收总时间，通过Parallel Scavenge和Parallel Old提高CPU使用效率
  • 延迟调优：关注最大停顿时间，通过CMS、G1、ZGC等垃圾收集器降低STW停顿时间
  • 堆大小调优：通过合理的堆内存和分代比例调优，避免频繁Minor GC和Full GC

扩展

Serial / Serial Old*（STW, To Client）

缺陷：Serial 收集器是单线程收集器（也是历史最悠久的），垃圾回收时，必须暂停其他所有的工作线程，直到收集结束

优势：单线程避免了多线程复杂的上下文切换，提高了收集效率非常高

由于这个优点，迄今为止，仍是 HotSpot 虚拟机在客户端模式下默认的新生代收集器（注：也可以配合CMS使用）

在老年代中，仍是单线程处理，采用标记-整理算法，用于客户端模式下的HotSpot

ParNew / CMS*

ParNew（多线程）

ParNew是Serial的多线程版本，可以看到新生代通过多线程回收

CMS（标记-清除）

CMS（Concurrent Mark Sweep）收集器则是以获取最短回收时间为目标的收集器，分为四个阶段：

  • 初始标记(inital mark) ：标记GC Roots直接关联对象
    • 耗时短，需要暂停用户线程
  • 并发标记 (concurrent mark) ：GC Roots直接关联的对象遍历整个对象图
    • 耗时长，但不用暂停用户线程
  • 重新标记 (remark) ：通过增量更新方法，对上一过程中由于用户线程产生变动的对象重新标记
    • 耗时长，需要暂停用户线程
  • 并发清除 (inital sweep) ：并发清除 死亡的对象
    • 耗时长，但不需要暂停用户线程。

优势：并发标记和并发清除过程不需要暂停用户线程，停顿时间较短

缺点：

  • 并发操作，对处理器资源敏感
  • 基于标记-清除算法实现，空间碎片
  • 无法处理浮动垃圾（Floating Garbage）：由于并发清除时用户线程还是在继续，所以此时仍然会产生垃圾，这些垃圾就被称为浮动垃圾，只能等到下一次垃圾收集时再进行清理。

Parallel Scavenge(标记-复制) / parallel Old（标记-整理）

是新生代收集器，基于标记-复制算法实现。目标是达到可控的吞吐量

Parallel Scavenge 收集器提供两个参数用于精确控制吞吐量：

  • -XX:MaxGCPauseMillis：控制最大垃圾收集时间，假设需要回收的垃圾总量不变，那么降低垃圾收集的时间就会导致收集频率变高，所以需要将其设置为合适的值，不能一味减小。
  • -XX:MaxGCTimeRatio：直接用于设置吞吐量大小，它是一个大于 0 小于 100 的整数。假设把它设置为 19，表示此时允许的最大垃圾收集时间占总时间的 5%（即 1/(1+19) ）；默认值为 99 ，即允许最大 1%（ 1/(1+99) ）的垃圾收集时间。

其老年代版本为Parallel Old收集器，通过标记-整理算法实现

G1*（标记-整理， To Server）

Garbage-First是面向服务器，用于多核CPU和大内存服务器环境

不同于按固定大小/数量划分分代区域，G1是直接把堆划分多个内存相等的独立区域（Region)，每一个Region可以是Eden、Survivior空间 或者 老年代空间，根据角色不同，采用不同收集策略

注：H是Humongous，表示Region存储大对象（Humongous Object, H-Obj），是大于等于一半Region空间的对象

G1收集器运行过程（标记对象基于bitmap）

  • 初始标记 (Inital Marking) ：标记GC Roots直接关联对象，在此基础上，修改TAMS（Top at Mark Start）指针值，让下一阶段用户线程并发运行时，能够正确在Region中分配新对象
    • G1为每个Region分配两个TAMS指针，新分配对象必须在这两个指针指向位置上面，默认被标记为存活，不会纳入回收范围，避免误杀这个过程中产生的对象
  • 并发标记 (Concurrent Marking) ：GC Roots关联的对象开始遍历整个对象图，在此基础上，处理SATB（snapshot-at-the-beginning，开始阶段快照）变动的对象
    • SATB可以解决并发标记过程中，由于用户线程导致的对象变动，效率比CMS重新标记阶段的增量更新效率高
  • 最终标记(Final Marking) ：此时对用户线程短暂暂停，处理上一阶段遗留的少量STAB记录（处理SATB过程中线程产生的少量变动，需要加一步最终标记处理）
  • 筛选回收 (Live Data Counting and Evacuation) ：对各个Region回收价值和成本进行排序，在用户期望的停顿时间下，指定回收计划（可以选择多个Region构成回收集）
    • 标记-整理算法：最终将回收集的存活对象复制到空的Region中，通过多个线程并行清理掉旧的Region（此时因为要迁移存活对象，要暂停用户线程）
