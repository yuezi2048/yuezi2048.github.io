37.组长让你分库分表你的实施流程

发表于2025-10-28|更新于2025-10-28|2.Areas🌐02.面经记录MySQL
|总字数:754|阅读时长:2分钟|浏览量:
  1. 首先要分析业务要求：分析数据量和增长的趋势，评估分库分表的必要性（可以有适度预判，但不要过度设计）
  2. 技术选型：设计分库分表方案
    1. 分片键：确定业务意义明确，查询高频，数据分布均匀的字段（user_id、order_id）
    2. 分库分表方式：range、hash、一致性hash、或结合上述方案，确定分片数量（要考虑未来拓展性）
    3. 确定路由逻辑：设置在Client端、Proxy层（ShardingSphere）还是自研中间件
  3. 数据迁移考虑：分批导入到新的分库分表结构中，可通过ETL工具或脚本完成
  4. 部分SQL语句和业务逻辑是否需要调整：如业务的链表查询，或者本地事务修改为分布式事务，要列举出来一一修改

常见分库分表中间件

ShardingSphere

主要有两种模式，分别是 ShardingSphere-JDBC 和 ShardingSphere-Proxy

ShardingSphere-JDBC (嵌入式)： 以 JAR 包形式集成到应用中，在 JDBC 层进行 SQL 解析和路由。应用直连数据库，改动的是应用的 JDBC 配置。

它性能最优（无网络中转），适合高性能要求、资源有限、易于横向扩展应用节点。部署相对轻量。

但是侵入应用逻辑（需适配 SQL、升级影响应用），配置需在应用服务中维护（多应用需各自配置），增加应用堆内存消耗。

ShardingSphere-Proxy (独立代理)： 伪装成数据库，应用将其视为单一数据源连接。核心工作在代理层完成。

它对应用透明（无侵入），配置集中管理（独立部署），语言无关（支持任何兼容数据库协议的客户端）。

但是增加一层网络跳转（性能损耗），需要独立部署、运维和高可用方案，本身是潜在瓶颈点。

MyCat

经典的独立数据库中间件代理，架构清晰。早期（2014-2018年）非常流行，文档和社区讨论多。后续发展相较 ShardingSphere 显得缓慢，但其成熟度和稳定性在许多存量系统中仍是优势。

它的特点：

  • 纯 Proxy 模式运行。
  • 应用端无需修改代码或框架，透明接入是其核心竞争力。

TiDB / PolarDB-X (分布式数据库，非中间件)

注意，这是分布式数据库，非分库分表中间件，但是分库分表的另一种选择，所以也放这里一起讨论。

换个分布式关系型数据库，天然的满足分库分表。

它：

  • 在数据库层面原生实现了数据分片（Sharding）、高可用、强一致性（TiDB）、水平扩展等能力。
  • 对应用完全透明，像使用单机数据库一样（SQL兼容性有差异）。

13.索引越多越好吗
不是，索引在空间和时间有成本。 时间：每次更新的操作下，索引也会动态更新，增加了写操作开销。还会产生B+树页分裂、合并等操作，产生更大的时间开销 注：MySQL内部有查询优化器机制（分析当前查询选择最优计划），就会考虑选择哪个索引成本低。索引过多的情况下，在优化器上的时间开销也会增加，甚至可能会由于数据不准确导致选择次优索引 空间：建立的每一个二级索引都会对应一颗B+树，每个数据页有16KB，数据量大且索引多时，占用空间不小
