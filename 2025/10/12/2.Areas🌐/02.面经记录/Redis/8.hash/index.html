8.hash

发表于2025-10-12|更新于2025-10-12|2.Areas🌐02.面经记录Redis
|总字数:1k|阅读时长:3分钟|浏览量:

hash是键值对，将多个字段和值存储在一个键中，便于管理关联数据

适合：

  • 存储小数据，使用哈希表实现，在内存中高效存储和操作
  • 支持快速字段crud操作，非常适合对象树形

场景：

  • 在鱼泡项目中，我使用到了session共享分布式登录，通过spring-session-data-redis依赖，只需通过少量配置和session的原生操作就能自动将session的登录态键对应信息推到了redis的存储桶上，实现分布式系统共享session的需求

扩展

  • HSET：HSET user:1001 name "mianshiya"
  • HGET：HGET user:1001 name
  • HMSET：HMSET user:1001 name "mianshiya" age "18" city "shanghai"
  • HMGET：HMGET user:1001 name age
  • HDEL：HDEL user:1001 age 删除一个或多个字段
  • HINCRBY：HINCRBY user:1001 age 1 自增整数值

Hash底层

hash支持redis存储2^32-1个键值对（40亿），底层结构有两种：

  • Redis6.0前，zipList + hashTable
  • Redis6.0后，ListPack + hashTable
  • zipList和listpack查找key效率类似，O(n)，区别在于listpack解决了级联更新的问题（注：redis7.0考虑到兼容性，并没有删除zipList）
  • 使用ziplist / hashTable条件：hash-max-listpack-entries（ziplist），hash-max-listpack-value（ziplist）对应键的个数（512）和字段名 / 字段值长度（64），小于这两个值，则使用listpack/ziplist，否则使用hashtable
  • 转换成hashtable后，没有退化机制
  • 默认的entries和value长度限制参数可修改

聊聊redis的HashTable

底层结构为：

  • table：存数据的载体，是由哈希节点（dictEntry）组成的数组（理解为java外面包装了mask和used参数）
  • size
  • sizemask：哈希表大小掩码，恒为size-1，可求得索引位置：hash值 & sizemask
  • used：使用的节点数量

来到dictEntry内部：可以看到value是一个union联合体，可以指向指针、64位无符号整数、整数、double，设计目的就是为了节省内存空间，整数/浮点数可以直接将值存在哈希节点中，不用指针指向，可以节省空间（jvm栈 方法区类比）

渐进式扩容

如果面试官聊到了redis的hashtable，一定绕不开渐进式扩容：扩容不是一次性的，而是逐步扩容的过程

操作细节：设计两个dictht哈希表实现渐进式逻辑

  • 正常插入数据写入ht[0]，ht[1]此时是空表
  • 触发扩容条件后，进行渐进式扩容（指出java hashmap扩容思想 + JVM GC标记-复制算法类似思想，引导面试官追问相关细节 ）：
    • ht[1]分配2倍used空间（和jdk1.8以后hashmap实现类似，要求2次方幂大小，追问：为什么？见集合章节），如used=1024，扩容后大小为2048，字典的rehashidx索引值从-1转变为0，开始数据迁移
    • 数据一致性：数据迁移时，每次的crud操作，会将rehashidx对应的数据从ht[0]迁移到ht[1]上，然后rehashidx+1（插入数据直接插入到2表即可），所以迁移是多次、渐进式的完成
    • 最终ht[0]数据全部迁移到ht[1]上，指针对象互换，ht[0]指向为空，rehashidx设置为-1

扩容条件：根据负载因子来判断（负载因子= 保存节点数量 / 哈希表大小）

  • ≥1，空间比较紧张，如果没有RDB快照/AOF重写的持久化机制，就会rehash
  • ≥5，hash冲突非常严重，无论是否有持久化机制，强制rehash

缩容条件：负载因子 < 0.1，为used最近的2次方幂，如used=1000，则缩容大小1024（没有RDB/AOF持久化机制条件下才会缩容）

10.事务
redis支持事务，但是事务的概念与mysql的ACID不同。Redis的事务指的是保证操作原子性，所有命令在原子操作中进行，不被打断。（注：redis的事务不支持回滚） 拓展 开始事务：MULTI开始，之后的命令进入队列 添加命令：添加命令，不会被立即执行，而是存储在队列 执行事务：EXEC原子性执行上述命令 取消事务：DISCARD清空命令，放弃事务 监视：WATCH监视一个/多个键，如果EXEC前被键修改了，EXEC不会执行，保证数据一致性。 Redis不支持回滚：所有命令都会被应用，即使某个步骤失败，依旧会执行，所以在设计时要充分考虑错误。 Redis2.6.5后，服务器在累计命令的过程中检测到错误，会拒绝执行事务，返回错误并丢弃该事务。 但是在执行过程中，仍会继续执行剩下的命令，不会进行回滚（这是和MySQL最不同的点，也不支持隔离级别设置，因为是单线程，只能是串行隔离级别），所以我们理解为是一个残血版的事务。
